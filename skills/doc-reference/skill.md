# Doc Reference Skill

## 适用场景

当用户请求以下任务时启用本 Skill：
- 在 `docs/` 中定位应读文档。
- 解释 QUIC 实现规划、测试用例、互操作流程。
- 进行 RFC 语义核对或 MegaLinter 配置核对。
- 需要“先读最少资料，再给结论”的检索流程。

## 强制流程（必须遵守）

1. 先读 `docs/menu.md`。
2. 从“任务阅读包”中选择最小集合。
3. 首轮只读 `P0` 文档，默认不超过 2 个文件。
4. 若信息不足：
   - 先明确写出信息缺口；
   - 先输出“缺失资料/工具请求”；
   - 再追加 1 个最小必要文档（`P1` 或 `P2`）。
5. 如果出现同主题多文档，按 `docs/menu.md` 的“去重与冲突规则”执行。
6. 禁止在缺口未声明时继续扩读全库。

流程表达式：
`先读 menu -> 选最小阅读包 -> 先读 P0 -> 不足先上报缺口 -> 再升级到 P1/P2`

## 阅读预算约束

- 默认预算：首轮最多 2 个文件。
- 升级预算：每轮只新增 1 个文件。
- 终止条件：信息足够支撑可执行结论时停止追加阅读。
- 禁止项：为“保险”一次性读取同主题全部文档。

## 文档优先级执行规则

- 工程实现与排障优先：`notes` / `plans` / `quic-interop-runner` / `mega-linter` 的 `P0` 条目。
- 权威核对再升级：对应 `ietf/txt` 的 RFC 原文。
- `9000` 系列默认策略：
  - 先读 `docs/ietf/notes/9000.md`。
  - 仅在状态机或帧细节不足时再读 `docs/ietf/notes/9000 copy.md`。

## 缺失资料与工具上报协议

出现以下任一条件时，必须触发缺口上报：
- 缺少必要规范文档。
- 文档中存在占位链接或未落地文件（例如仅指向 `quic.md` / `webtransport.md`，但仓库内无对应文件）。
- 执行所需工具未定义、未安装或当前环境不可用。

触发后必须输出：
- 缺失项：具体缺失的文档、脚本或工具名。
- 用途：缺失项用于解决什么问题。
- 建议补充位置或安装方式：优先给出仓库路径或可执行安装命令。
- 输出顺序要求：触发缺口时，将“缺失资料/工具请求”段放在“本次结论”之前。

## 输出模板（每次检索结果都用）

```markdown
已读文档：
- <path1>
- <path2>

本次结论：
- <可执行结论 1>
- <可执行结论 2>

若需继续应读文档：
- <path or 无>
- 缺口原因：<为什么需要继续读>

缺失资料/工具请求：
- 缺失项：<doc/tool>
- 用途：<为什么必须补齐>
- 建议补充位置或安装方式：<path/command>
```

> 当触发缺口上报时，将“缺失资料/工具请求”段提前到“本次结论”之前。

## 维护要求

- 若发现 `docs/` 有新增/重命名文件而 `docs/menu.md` 未同步：
  1. 先指出索引缺口。
  2. 优先补齐 `docs/menu.md`。
  3. 再继续文档检索任务。
