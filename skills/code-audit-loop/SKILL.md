# Code Audit Loop

## 概述

先完成本轮本地实现，再判断是否进入独立审计闭环。若触发审计，则先向用户说明原因并在需要时请求授权，再把受限上下文交给只读审计 agent；审计报告返回后，必须继续完成至少一轮代码修改或测试补强，不能只转述报告。

审计 agent 的交接提示词和报告格式见 [references/audit-agent-contract.md](references/audit-agent-contract.md)。

## 审计触发规则

满足以下任一条件时，提醒用户应考虑追加审计：

- 修改涉及认证、授权、加密、密钥、输入校验、反序列化、命令执行、路径处理。
- 修改涉及协议解析、状态机、重试、超时、并发、锁、内存、文件句柄、连接生命周期。
- 修改影响公共 API、持久化格式、网络边界、错误恢复、回滚、迁移逻辑。
- 修改跨越多个模块，或引入行为变化但缺少覆盖测试。
- 变更量较大，或你对正确性、边界条件、回归风险没有足够把握。
- 用户明确要求 `review`、`audit`、`审计`、`第二意见`、`代码检查`，或要求与另一个 agent 协作。

以下场景通常不必主动提醒审计：

- 纯注释、文案、格式化修改。
- 机械性重命名且无行为变化。
- 只补充与现有行为一致的低风险测试。

## 授权规则

- 若当前请求已经明确要求使用另一个 agent、sub-agent、delegation 或 parallel agent work，直接进入审计闭环。
- 若触发了审计条件，但用户尚未明确授权另一个 agent，先提醒用户，再等待确认，不要直接调用其他 agent。

使用以下提醒格式：

```markdown
建议追加审计：
- 触发原因：<命中的风险点>
- 审计范围：<文件、模块或 diff 范围>
- 审计目标：<最担心的正确性/回归/安全问题>
- 授权请求：如同意，我将调用独立审计 agent，并在报告返回后至少完成一轮修复。
```

## 审计交接流程

1. 先完成首轮本地实现，收敛出可审计的最小上下文。
2. 仅向审计 agent 发送任务目标、相关 diff、触及文件、必要代码片段、已运行测试与已知缺口。
3. 不要把你怀疑的 bug、预设结论或预期修复方案泄露给审计 agent，除非用户要求定向审计。
4. 明确要求审计 agent 只做审计，不做代码修改。
5. 使用 [references/audit-agent-contract.md](references/audit-agent-contract.md) 中的提示词和报告模板。

## 报告接收与筛选

收到审计报告后，先检查报告质量。出现以下任一问题时，要求补充而不是直接执行：

- 没有指向具体文件、函数、行号或可定位证据。
- 只有泛泛建议，没有可执行的最小修改方向。
- 把风格建议当成缺陷，或缺少影响说明。
- 与当前代码或 diff 明显不符。

对可用报告按以下顺序排序：

1. `severity` 更高。
2. `confidence` 更高。
3. 修改范围更小且更安全。
4. 能被现有测试或快速补测验证。

## 报告后的二次修改

拿到合格报告后，不要停在“转述报告”。默认执行以下闭环：

1. 选定一个最高优先级且可安全落地的问题。
2. 直接修改代码；若行为受影响，同步补充或更新测试。
3. 运行与该问题最相关的验证。
4. 向用户说明本轮处理了哪一项发现、做了什么改动、如何验证、还剩什么风险。

只有在以下情况才允许不做后续修改：

- 报告缺少可执行证据。
- 缺失关键上下文，继续修改会引入更大风险。
- 用户明确要求只收集报告，不继续改动。

若无法继续修改，必须明确写出阻塞原因和下一步所需信息。

## 输出要求

进入审计闭环后，面向用户的结果至少包含以下信息：

```markdown
审计摘要：
- overall_risk: <low|medium|high>
- 选中处理项：<finding id 或 none>

本轮修改：
- <实际完成的代码或测试改动>

验证：
- <已执行的命令、测试或人工检查>

剩余风险：
- <仍未处理的问题；若无则写 none>
```

若用户尚未授权另一个 agent，则先输出“建议追加审计”段，不要伪造审计结果。
