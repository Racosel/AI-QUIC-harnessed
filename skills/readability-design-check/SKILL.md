# Readability Design Check

## 概述

把“代码健康检查”提前到开发过程中，而不是只放在最终 review。每次检查都应尽量小、快、可落地：先发现会持续放大理解成本的问题，再优先做低风险改进；若发现重大设计偏差，应立即提醒用户，而不是继续在错误抽象上堆代码。

官方原则摘要见 [references/official-principles.md](references/official-principles.md)。

## 周期性触发规则

满足以下任一条件时，执行一次检查：

- 完成一轮可运行实现后，且准备继续叠加功能。
- 同一逻辑区域累计发生 2 到 5 次有意义修改。
- 即将新增公共 API、头文件导出、配置项、类型或新抽象层。
- 函数、类型或文件明显变长，出现多层分支、重复片段或难以命名的中间变量。
- 修改涉及状态机、并发、资源生命周期、错误处理分支或回滚路径。
- 你开始依赖大量注释来解释代码，但即使加了注释仍然难讲清楚。
- 准备给用户最终答复、提交补丁或发起正式 review 前。

以下情况可以只做极轻量检查：

- 纯注释、文案、格式化调整。
- 明显机械性的重命名。
- 已有模式下的小型测试补充。

## 检查顺序

1. 先看本地规范：优先遵循仓库已有命名、文件布局、错误处理、日志、测试和头文件组织方式。
2. 再做可读性检查：确认名称、作用域、函数边界、控制流、注释、常量、数据流和文件职责是否便于新读者理解。
3. 再做设计检查：确认抽象层次、模块边界、依赖方向、状态与资源所有权、公开接口、错误语义和测试切面是否合理。
4. 最后决定动作：立即修复、拆成后续事项，或立即向用户上报重大设计问题。

## 联网与参考规则

- 默认先读本仓库规范和相邻代码，不要拿外部风格硬压本地约定。
- 当本地规范不足、语言特定规则不清楚、或需要权威依据支撑判断时，优先查询官方文档。
- 仅使用第一方或官方来源，例如语言官方风格指南、项目官方编码规范、维护者发布的工程实践文档。
- 本 skill 内置了一份基于官方资料整理的基线，见 [references/official-principles.md](references/official-principles.md)。

## 可读性检查清单

- 名称是否让第一次读到这段代码的人理解“它是什么”和“它为什么存在”。
- 函数是否只做一件主要事情；若职责混杂，优先提取命名良好的小步骤。
- 变量作用域是否足够小；是否把声明和首次使用放得足够近。
- 分支、循环和条件表达式是否能快速扫读；若条件过长，优先拆出语义化布尔量或辅助函数。
- 是否存在魔法常量、隐式协议、隐藏副作用、跨层写入或难以追踪的状态修改。
- 注释是否解释“为什么”与边界条件，而不是逐行翻译“代码正在做什么”。
- 文件和模块名、头文件暴露面、目录位置是否与真实职责相符。
- 测试名、测试结构和断言是否表达清楚行为意图，而不是堆砌实现细节。

## 设计检查清单

- 这段改动是否真的属于当前模块，而不是更底层的库或更靠近调用方的适配层。
- 是否引入了过早抽象、过度泛化或为了“将来可能需要”而增加的复杂度。
- 公共接口是否足够小、明确、可解释；调用方是否容易误用。
- 数据所有权、生命周期和状态转换是否显式，而不是藏在副作用或时序假设里。
- 错误处理、返回值、日志和回滚行为是否一致，是否便于定位问题。
- 依赖方向是否清晰；是否把上层知识泄漏到下层，或让模块互相缠绕。
- 新增结构是否提升了测试能力；若更难测试，通常说明边界设计有问题。

## 立即修复与上报规则

满足以下条件时，直接修改代码，不必等到最后统一处理：

- 修复范围局部、风险低、不会改变外部行为。
- 只是重命名、缩小作用域、提取小函数、拆分长条件、补充必要注释、删除重复代码或补一个直接相关的测试。
- 能通过现有测试或快速新增的最小测试验证。

满足以下条件时，先向用户上报，再决定是否继续改动：

- 设计问题已经大到可能推翻当前实现方向。
- 修复需要跨越多个模块、公共 API 或数据格式。
- 本地代码模式与外部最佳实践冲突，且你无法确认是否应以本地模式为准。
- 你已经发现继续堆代码只会进一步恶化可读性或设计质量。

上报时优先给出结论，不要先埋头修边角：

```markdown
代码健康预警：
- trigger: <为什么现在必须停下来检查>
- major_issue: <主要设计或可读性问题>
- impact: <继续下去会造成什么理解/维护/回归成本>
- recommended_direction: <建议的最小重构方向>
```

```markdown

- overall_health: <improving|stable|degrading>

- [high|medium|low] <问题> -> <fix-now|follow-up>

```

## 硬约束

- 不要把大规模格式化和行为修改混在同一个补丁里。
- 不要用注释掩盖糟糕命名或糟糕结构。
- 不要为了抽象而抽象；优先解决当前已知问题。
- 不要把“个人风格偏好”冒充成必须修复的问题。
- 只要能以小步改善代码健康，就不要把问题无限延期到“最后再看”。
