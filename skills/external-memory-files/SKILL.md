# 外部记忆记录器

## 何时使用

当任务涉及以下任一内容时，使用本 skill：

- 需要在仓库里进行任何改动，并保留可追溯的时间线
- 需要记录 bug、根因、修复与验证结论
- 任务可能被用户打断、被权限或环境阻塞，或需要跨轮继续
- 需要维护仓库根目录 `log`、`debug`、`break` 三个文件

## 快速开始

1. 在仓库根目录确认 `log`、`debug`、`break` 存在；如果不存在，先创建。
2. 开始写入前先读现有内容，避免覆盖仍然有效的 bug 状态或断点。
3. 明确三者分工：
   - `log` 记执行步骤和时间线
   - `debug` 记 bug 生命周期
   - `break` 记当前断点和下一步入口

## 文件分工

### `log`

- 只记录执行步骤、命令、修改动作、结果和下一步。
- 每完成一个有意义的步骤后追加一条；连续小步骤可以合并，但不能完全省略。
- 不要把完整 bug 分析、修复复盘或长段推理写成 `log`；这些内容属于 `debug`。

### `debug`

- 只记录 bug、缺陷群、根因、修复、验证和当前状态。
- 发现 bug、确认根因、实施修复、完成验证、收到新的失败输出时更新。
- 同一个 bug 的后续分析与最终修复继续写回原条目，不默认新开替代 bug。
- 已确认解决的条目要把状态更新为 `已解决`。
- 当一个问题跨越多轮修复并演化为“缺陷群”时，要主动整理成高信号摘要，优先保留“现象、影响、主根因、修复、验证、后续风险”。

### `break`

- 只保留当前可恢复的断点，不维护完整历史时间线。
- 准备结束本轮、等待用户执行高权限步骤、被环境阻塞、或即将切到另一个大任务前更新。
- `break` 以覆盖式刷新为主，让下一次接手的人能快速恢复现场。
- 不要把已经失效、互相矛盾的旧状态长期堆在一起。

## 格式要求

### `log` 格式

- 每追加一条前，都重新获取当前时间，不能复用任务开始时或上一条日志的时间戳。
- 统一使用 `date '+%Y-%m-%d %H:%M:%S %z'` 获取时间。
- 每条写成 `[时间] 内容`。
- 内容至少包含动作、涉及的文件或命令，以及结果、结论或下一步。

示例：

```text
[2026-04-07 12:00:00 +0800] 开始拆分 skill 中的 log/debug/break 规则，检查现有 skill 与根目录记录文件。
[2026-04-07 12:08:00 +0800] 已创建 ai-quic-memory-files skill，并开始回收现有 skill 中的重复规则。
```

### `debug` 格式

- 使用 Markdown。
- 优先按 `Bug NNN` 组织条目，保证后续可持续更新。
- 每条记录至少包含以下字段：
  - `时间`
  - `阶段`
  - `标题`
  - `现象`
  - `影响`
  - `根因`
  - `修复`
  - `验证`
  - `相关文件`
  - `状态`
- 每次收到新的运行输出后，先回看相关条目，再决定是更新状态还是补充原条目。

### `break` 格式

- 使用 Markdown。
- 保持单份“当前断点”视图，而不是流水账。
- 推荐结构：

```markdown
阶段 X / 主题 Y 断点（更新于 YYYY-MM-DD HH:MM:SS +0800）

当前状态
- ...

已完成
- ...

未完成
- ...

下一步
1. ...

关键文件
- ...

待确认 / 风险
- ...

如需用户执行
- ...

最近验证
- ...
```

- 如果旧内容已经失效，直接重写对应小节，不要为了保留历史而降低可恢复性。

## 决策规则

- `log`、`debug`、`break` 不能互相替代。
- `log` 以追加为主，`debug` 以维护原条目为主，`break` 以覆盖式刷新为主。
- 任务没有 bug 时，不要为了“补全格式”强行写 `debug`。
- 任务不会跨轮延续且没有阻塞时，可以不更新 `break`。
- 记录应追求高信号，不要把三个文件写成低价值流水账。

## 与其他 skill 的配合

- 执行其他skill时，如果任务需要维护根目录记录文件，同时遵循本 skill。
- 项目阶段边界、协议约束、测试与回归要求仍以对应 skill、`README.md` 和代码事实为准；本 skill 只负责外部记忆文件的维护方式。
