### 第1章 引言与qlog规范基础

#### 双方均应遵守的设计

* **标准目的与格式**：双方均需理解 qlog 是一种用于网络协议的可扩展结构化日志格式。它的目标是提供通用的调试和分析数据共享方式，解决由于自定义非标准日志格式导致的互操作性分析困难问题。
* **核心特性支持**：日志体系应包含（或能够包含）版本控制 (versioning)、元数据包含 (metadata inclusion)、日志聚合 (log aggregation)、事件分组 (event grouping) 以及日志文件大小缩减技术。
* **序列化标准**：所有的 qlog 模式 (schemas) 均可通过多种方式序列化，但默认且基准的序列化必须采用 JSON、其子集 I-JSON 或其流式衍生版本 JSON-Text-Sequences。除非另有说明，序列化示例均使用 JSON。
* **术语规范**：必须严格遵循 BCP 14 [RFC2119] [RFC8174] 中的关键字要求（如 MUST, REQUIRED, SHOULD, MAY 等）。
* **数据定义语言 (CDDL) 使用**：双方均须使用 CDDL (Concise Data Definition Language) 来定义和理解事件与数据结构。支持基础的 CDDL 语法及特定控制操作符（如 `.default`, `.size`, `.regexp` 等）。
* **自定义数据类型**：必须支持以下为了清晰而定义的自定义 CDDL 类型：
    * `uint8`, `uint16`, `uint32`, `uint64`：明确大小的无符号整数。
    * `hexstring`：必须是一串偶数长度的、仅包含小写字母的十六进制编码字节字符串（例如 `82dc`）。为了兼容 JSON，禁止使用 CDDL 默认的二进制字符串 (`bytes/bstr`)。
* **时间戳精度**：所有在 qlog 中的时间戳和与时间相关的值（例如偏移量）都必须以毫秒为分辨率的 `float64` 类型进行记录和解析。

#### 发送端应遵守的设计

* **事件重要性**：在生成和记录日志事件时，发送端（日志生成器）必须按照规范为每个事件定义一个“重要性级别” (importance level)。
* **数据输出格式**：发送端在序列化日志文件时，必须输出符合上述 CDDL 数据结构定义的 JSON 格式，尤其要注意二进制数据必须转换为 `hexstring` 格式以兼容 JSON。
* **其他 qlog 文档的扩展**：发送端可根据具体的协议（如 QUIC 或 HTTP/3）提供特定于该协议的事件模式（如定义协议支持的每种数据包类型的 CDDL struct）。

#### 接收端应遵守的设计

* **字段顺序无关性**：在解析 JSON 和 JSON Text Sequences 格式的 qlog 序列化数据时，成员字段的顺序是无关紧要的。接收端（日志分析工具）**绝不能 (MUST NOT)** 假设字段存在任何特定的顺序。
* **通用工具兼容性**：接收端工具的设计应具有通用性和可复用性，能够处理来自各种不同网络协议和用例的日志文件。
* **CDDL 语法解析能力**：接收端需要能够理解文档中常见的 CDDL 语法约定，以便正确读取数据结构模型：
    * `? obj`：表示该对象是可选的。
    * `TypeName1 / TypeName2`：表示类型的联合（该对象可以是类型 1 或类型 2）。
    * `obj: [* TypeName]`：表示最小长度为 0 的类型数组。
    * `obj: [+ TypeName]`：表示最小长度为 1 的类型数组。
    * `* text => any`：表示 0 个或多个键为字符串、值为任意类型的字段（用于表示通用的 JSON 对象）。

### 第2章 设计概述

#### 双方均应遵守的设计

* **核心设计原则**：
  * **流式与基于事件**：日志必须是可流式传输的 (Streamable) 且基于事件的 (event-based)。
  * **可扩展与实用**：日志格式设计必须具备可扩展性 (Extensible) 和实用性 (pragmatic)。
  * **聚合与转换友好**：对于非流式格式，顶层元素必须是各个独立跟踪 (traces) 的容器；可以使用 `group_ids` 将事件标记关联到特定的上下文中。
  * **元数据捆绑**：元数据 (Metadata) 必须与具体的事件数据 (event data) 存储在一起。
* **逻辑日志层次结构**：所有的 qlog 数据必须遵循以下严格的逻辑层级：
  1. 日志文件 (Log file)
     * 包含一个或多个 跟踪 (Trace(s))
       * 包含一个或多个 事件 (Event(s))
* **基本数据模型**：
  * **LogFile（日志文件）**：存在一个抽象的 `LogFile` 类，所有具体的日志文件格式都必须通过日志文件模式 (schemas) 从中派生（例如本规范中定义的 `QLogFile` 和 `QLogFileSeq`）。
  * **Trace（跟踪）**：概念上是流动的，但常规的约定用例是将单一观察点上与单一数据流（例如，单个逻辑 QUIC 连接）相关的事件进行分组。
  * **Event（事件）**：存在一个包含公共字段的抽象 `Event` 类，所有具体事件都从中派生。具体事件由事件模式 (event schemas) 定义，这些模式声明或扩展一个命名空间 (namespace)，其中包含一个或多个相关的事件类型或其扩展（例如通用的 `loglevel` 和 `simulation` 命名空间）。

#### 发送端应遵守的设计

* **性能权衡**：发送端（日志生产者）在生成日志时，可以利用 qlog 灵活的格式特性来降低自身的性能开销 (producer overhead)。
* **事件记录时机**：发送端必须在特定的时间点 (time instant) 记录事件，并准确传达该日志用例的具体细节（例如，网络数据包的发送或接收）。
* **模式遵循**：在生成具体的日志文件和事件时，发送端必须严格按照当前所选的日志文件模式（如 `QLogFile` 或 `QLogFileSeq`）和事件模式进行构建和派生。

#### 接收端应遵守的设计

* **处理复杂性的能力**：接收端（日志消费者，如分析工具）必须在设计上能够处理由于发送端降低开销而转移过来的、格式上增加的复杂性 (increased complexity)。
* **层级解析**：工具必须能够正确解析和映射 qlog 的“日志文件 -> 跟踪 -> 事件”三层逻辑嵌套结构。
* **命名空间与上下文识别**：接收端需要能够正确识别和解析通过 `group_ids` 分组的上下文事件，并能处理各种具体事件模式所声明的命名空间。


### 第3章 抽象LogFile类

#### 双方均应遵守的设计

* **LogFile 的目的与定义**：日志文件 (Log file) 旨在包含一组以某种方式相关的事件。必须基于一个抽象的 `LogFile` 类进行设计和解析，该类包含所有日志文件共有的字段（使用 CDDL 定义如下）：
  ```cddl
  LogFile = {
      file_schema: text
      serialization_format: text
      ? title: text
      ? description: text
  }
  ```
* **具体模式的派生**：每个具体的日志文件模式都必须使用 CDDL 的解包操作符 (`~`) 从此抽象类派生，并可以通过定义语义和自定义字段来扩展它。
* **URI 标识**：具体的日志文件模式必须 (MUST) 使用 URI [RFC3986] 来标识自身。
* **URI 命名规范 (RFC定义)**：由 RFC 定义的日志文件模式必须 (MUST) 在“qlog 日志文件模式 URIs”注册表中注册一个 URI，并且应该 (SHOULD) 使用 `urn:ietf:params:qlog:file:<schema-identifier>` 格式的 URN。其中 `<schema-identifier>` 是一个全局唯一的文本名称，且仅使用 URI 未保留范围内的字符。
* **URI 命名规范 (私有/非标准)**：私有或非标准的日志文件模式可以 (MAY) 注册 URI，但绝不能 (MUST NOT) 使用上述的 `urn:ietf...` 格式。包含域名的 URI 应该 (SHOULD) 包含 `mmyyyy` 格式的月-日期（例如 `https://example.org/072024/globallyuniquelogfileschema`）。该域名所有者必须在该日期或该日期前后授权该定义和分配，以避免域名所有权变更带来的问题。该 URI 不需要是可解引用的 (dereferencable)。
* **注册表审核机制**：“qlog 日志文件模式 URIs”注册表根据 [RFC8126] 的专家审查政策运行，非标准请求若使用保留 URN 格式必须 (MUST) 被拒绝。

#### 发送端应遵守的设计

* **必填字段：`file_schema`**：发送端必须 (MUST) 在日志文件中提供该字段，用于标识具体的日志文件模式。该值必须 (MUST) 是一个绝对 URI，并遵循上述的 URI 标识规则。
* **必填字段：`serialization_format`**：发送端必须 (MUST) 提供该字段，使用媒体类型 (media type [RFC2046]) 指示序列化格式。该值不区分大小写。
* **可选字段：`title` 与 `description`**：发送端可以提供这两个字段，用于包含有关该文件的附加自由文本信息。
* **字段位置优化**：为了使解析和识别 qlog 文件及其序列化格式更加容易，发送端应该 (SHOULD) 将 `file_schema` 和 `serialization_format` 字段及其对应的值放置在生成的日志文件的前 256 个字符/字节内。

#### 接收端应遵守的设计

* **基础字段解析**：接收端必须能够正确解析和提取日志文件顶层的 `file_schema` 和 `serialization_format`，并支持可选的 `title` 和 `description` 字段。
* **格式识别机制**：接收端在读取日志文件时，应优先检查文件的前 256 个字符/字节以快速定位 `file_schema` 和 `serialization_format`，以便应用正确的反序列化器和模式校验器。
* **URI 验证与兼容**：接收端必须支持将 `file_schema` 的值解析为绝对 URI，且需要兼容不可解引用的私有或非标准 URI。
* **大小写不敏感处理**：接收端在处理 `serialization_format` 字段时，必须忽略大小写差异。


### 第4章 QlogFile 模式

#### 双方均应遵守的设计

* **QlogFile 模式用途**：使用 `QlogFile` 模式的 qlog 文件可以包含多个独立的跟踪 (traces) 以及来自多个相关观察点 (vantage points) 的日志。
* **QlogFile 数据结构**：顶层元素定义了少量的“头部”字段和一个组件跟踪的数组。它通过 CDDL 的解包操作符 (`~`) 继承自抽象的 `LogFile` 类，并添加了可选的 `traces` 字段。
  ```cddl
  QlogFile = {
      ~LogFile
      ? traces: [+ Trace / TraceError]
  }
  ```
* **URI 标识**：`QlogFile` 模式的 URI 必须为 `urn:ietf:params:qlog:file:contained`。
* **Trace (跟踪) 数据结构**：`Trace` 是一个概念上灵活的实体（常规用例是在单个位置收集的单一数据流的日志）。它包含元数据和 qlog 事件数组：
  ```cddl
  Trace = {
      ? title: text
      ? description: text
      ? common_fields: CommonFields
      ? vantage_point: VantagePoint
      event_schemas: [+text]
      events: [* Event]
  }
  ```
* **TraceError (跟踪错误) 数据结构**：如果在尝试查找/转换文件以包含到聚合的 qlog 中时发生错误，必须使用 `TraceError` 显式记录，而不是静默丢弃：
  ```cddl
  TraceError = {
      error_description: text
      ? uri: text
      ? vantage_point: VantagePoint
  }
  ```

#### 发送端应遵守的设计

* **非流式特性注意事项**：`QlogFile` 的默认序列化格式是 JSON。发送端必须注意，标准的 JSON 格式具有**非流式**的固有缺陷。在文件结束前，必须附加 `]}]}`（或相应的闭合标签）来“关闭”文件，否则大多数解析器将无法完整解析它。如果需要流式传输，应该考虑使用替代的 `QlogFileSeq` 模式。
* **单个观察点的生成**：当发送端在单个观察点捕获 qlog 时，预期 `traces` 字段中只包含一个条目。
* **事件模式声明**：发送端在生成 `Trace` 对象时，必须在必填的 `event_schemas` 字段中声明具体的事件模式 URI 数组，以标识 `events` 字段中记录的事件命名空间和相关类型。

#### 接收端应遵守的设计

* **聚合与合并**：接收端工具（例如作为后处理操作的一部分）可以聚合多个文件。这可以通过将组件文件中的跟踪复制到新的、聚合的 qlog 文件的组合 `traces` 数组中来实现。
* **解析 Trace 元数据**：接收端工具主要可以通过解析 `Trace` 对象中的 `common_fields` 列表和 `vantage_point` 字段来推断该跟踪的语义和上下文。
* **错误处理可见性**：接收端在解析聚合文件时，必须能够识别并呈现 `traces` 数组中的 `TraceError` 对象，以告知用户哪些日志文件在聚合过程中出现了丢失或转换错误。
* **事件关联替代方案**：接收端需要理解，除了将不同跟踪放入 `traces` 数组外，另一种在单个 qlog 文件中组合不同跟踪事件的方法是通过使用 `group_id` 字段（详见后续章节）。


### 第5章 QlogFileSeq 模式

#### 双方均应遵守的设计

* **QlogFileSeq 模式用途**：使用 `QlogFileSeq` 模式的 qlog 文件旨在序列化为一种可流式传输的 JSON 格式，即 JSON 文本序列 (JSON Text Sequences, JSON-SEQ, [RFC7464])。
* **QlogFileSeq 数据结构**：顶层元素定义了少量的“头部”字段和一个单独的跟踪。它通过 CDDL 的解包操作符 (`~`) 继承自抽象的 `LogFile` 类，并包含一个必需的 `trace` 字段：
  ```cddl
  QlogFileSeq = {
      ~LogFile
      trace: TraceSeq
  }
  ```
* **URI 标识**：`QlogFileSeq` 模式的 URI 必须为 `urn:ietf:params:qlog:file:sequential`。
* **TraceSeq (流式跟踪) 数据结构**：`TraceSeq` 专门用于 `QlogFileSeq`。它在概念上与普通的 `Trace` 相似，但**不包含** qlog 事件数组 (`events` 字段被移除)：
  ```cddl
  TraceSeq = {
      ? title: text
      ? description: text
      ? common_fields: CommonFields
      ? vantage_point: VantagePoint
      event_schemas: [+text]
  }
  ```

#### 发送端应遵守的设计

* **流式序列化格式**：发送端在序列化 `QlogFileSeq` 时，必须严格遵守 JSON-SEQ ([RFC7464]) 格式。每个独立的 JSON 记录（首先是包含元数据的顶层对象，随后是各个事件对象）都必须以记录分隔符 (Record Separator, `<RS>`) 开头，并以换行符结尾。
* **流式事件的追加**：发送端绝不能将事件放置在 `TraceSeq` 内部。所有的 qlog 事件都必须作为独立的 JSON 对象，在顶层的 `QlogFileSeq` 头部对象之后依次流式追加。
* **单一跟踪限制**：与 `QlogFile` 可以包含多个 `traces` 不同，发送端在使用 `QlogFileSeq` 模式时，顶层对象只能包含一个单独的 `trace` 字段。
* **模式声明**：发送端必须在 `TraceSeq` 对象中提供 `event_schemas` 字段，声明后续追加的流式事件所对应的事件模式。

#### 接收端应遵守的设计

* **流式解析能力**：接收端工具必须支持解析 JSON 文本序列 (JSON-SEQ)，能够通过识别记录分隔符 (`<RS>`) 和换行符来逐个提取和解析流式的 JSON 对象。
* **事件上下文关联**：由于事件不再嵌套于 `Trace` 对象内部，接收端必须在解析过程中维护状态，将读取到的所有后续 qlog 事件都关联到文件开头解析出的那个单一的 `TraceSeq` 元数据上下文上。
* **处理头部与事件流**：接收端应期望流的第一个完整对象是符合 `QlogFileSeq` 结构的头部对象（包含 `file_schema` 等字段），随后接收到的对象均应解析为具体的日志事件。


### 第6章 观察点 (VantagePoint)

#### 双方均应遵守的设计

* **VantagePoint 定义**：`VantagePoint` 用于描述跟踪（Trace）起源的观察位置。其 CDDL 结构定义如下：
    ```cddl
    VantagePoint = {
        ? name: text
        type: VantagePointType
        ? flow: VantagePointType
    }

    ; client = 发起连接的端点
    ; server = 接受连接的端点
    ; network = 位于客户端和服务器之间的观测者
    VantagePointType = "client" / "server" / "network" / "unknown"
    ```
* **观察点类型含义**：
    * `client`：指代发起连接的一方。
    * `server`：指代接受连接的一方。
    * `network`：指代中间观察者（如数据包捕获工具）。
    * `unknown`：表示观察点类型未知。

#### 发送端应遵守的设计

* **必填字段 `type`**：发送端必须明确指定 `type` 字段。
* **字段 `flow` 的使用约束**：
    * **网络观察者场景**：当 `type` 为 `network` 时（例如从数据包捕获生成的日志），**必须**提供 `flow` 字段。
    * **消除歧义**：由于日志中可能因隐私等原因缺失 IP 地址等可推导方向的数据，`flow` 字段用于明确“数据包发送”和“数据包接收”等事件的参考系。
* **`flow` 语义设定**：
    * 若 `flow` 为 `client`：表示该观察点遵循客户端数据流语义（即“发送”事件的方向是朝向服务器）。
    * 若 `flow` 为 `server`：表示该观察点遵循服务器数据流语义（即“发送”事件的方向是朝向客户端）。
    * 若 `flow` 为 `unknown`：表示无法确定流向。

#### 接收端应遵守的设计

* **处理 `unknown` 值**：当接收端（工具）遇到 `type` 或 `flow` 为 `unknown` 的情况时，应采取以下策略之一：
    * **启发式推断**：利用协议层面的领域知识进行推断（例如：在 QUIC 协议中，第一个数据包始终由客户端发送）。
    * **人工干预**：提供用户界面选项，允许用户手动切换“客户端视图”或“服务器视图”。
* **语义转换**：接收端必须根据 `type` 和 `flow` 的组合正确解释事件方向，确保在分析不同来源的日志时，发送/接收行为的一致性。


### 第7章 抽象事件类与时间戳

#### 双方均应遵守的设计

- **抽象 Event 类定义**：所有 qlog 事件必须基于以下 CDDL 结构，包含三个强制字段 `time`、`name` 和 `data`：
  ```cddl
  Event = {
      time: float64
      name: text
      data: $ProtocolEventData
      ? tuple: TupleID
      ? time_format: TimeFormat
      ? group_id: GroupID
      ? system_info: SystemInformation
      * text => any ; 允许自定义扩展字段
  }
  ```
- **时间戳规范**：`time` 字段必须使用 `float64` 类型，通常代表以毫秒为单位的持续时间，支持分数部分以达到微秒或纳秒分辨率。
- **参考时间 (ReferenceTime)**：定义时间测量的起点：
    - `clock_type`：可选值为 "system"（系统时钟，可能跳变）或 "monotonic"（单调时钟，不回退）。
    - `epoch`：在使用 "monotonic" 时必须为 "unknown"；使用 "system" 时推荐 [RFC3339] 格式。
- **时间编码格式 (TimeFormat)**：
    - `relative_to_epoch`：（默认值）相对于参考时间 `epoch` 的持续时间。
    - `relative_to_previous_event`：增量编码，相对于前一个事件的时间偏移。
- **网络路径标识 (Tuple)**：`TupleID` 默认为空字符串 ""，用于关联特定的网络四元组。
- **逻辑分组 (Grouping)**：使用 `group_id` 字符串将事件关联到特定的逻辑实体（如 QUIC 连接 ID 或 QoS 级别）。
- **系统信息 (SystemInformation)**：可选记录 `processor_id`、`process_id` 和 `thread_id`（均为 `uint32`）。
- **公共字段字典 (CommonFields)**：支持定义 `common_fields` 字典以存储 Trace 中所有事件共享的键值对。

#### 发送端应遵守的设计

- **强制记录**：每个生成的事件**必须 (MUST)** 包含 `time`、`name` 和 `data`。
- **事件排序**：在单个 Trace 中，事件**应该 (SHOULD)** 按照时间戳严格升序记录（增量格式下指逻辑时间升序）。
- **优化冗余**：若 Trace 中多个事件的字段值相同（如 `group_id`、`time_format`、`reference_time`），**应该 (SHOULD)** 将其提取到 `common_fields` 中。
- **提取限制**：若 Trace 中至少有一个事件的某个字段值不同，该字段**绝不能 (MUST NOT)** 放入 `common_fields`，必须在每个事件中单独定义。
- **隐私保护**：对 `reference_time` 可以 (MAY) 进行最小化或匿名化处理。

#### 接收端应遵守的设计

- **字段解析能力**：工具**必须 (MUST)** 能够处理字段定义在单个事件实例中或组合在 `common_fields` 中的情况。
- **未知字段处理**：**必须 (MUST)** 忽略 `common_fields` 字典中未知的条目，不应视为错误。
- **排序鲁棒性**：工具可以 (MAY) 在处理前对事件按时间戳排序，但不应强制要求发送端完成此操作，以避免给多线程/流式记录器带来过大开销。
- **时间转换限制**：工具**不应该 (SHOULD NOT)** 假设能从 qlog 导出绝对日历时间，也不应假设不同 Trace 间的时间戳具有一致性。


### 第8章 具体事件类型与事件模式 (Event Schemas)

#### 双方均应遵守的设计

* **命名空间与标识符**：
    * 具体事件类型必须属于唯一的**事件命名空间** (Event Namespace)。
    * 命名空间必须是全局唯一的文本标识符，且仅使用 URI 未保留字符。
    * **事件全名格式**：必须为 `命名空间标识符:事件类型标识符`（例如 `quic:packet_sent`）。这种级联格式必须保证全局唯一，以防止不同模式间的命名冲突。
* **模式 (Schema) 标识与 URI**：
    * 每个事件模式必须拥有一个唯一的**绝对 URI**。
    * 模式本身是**不可变**的，不得直接扩展。
    * 扩展现有命名空间的模式，其 URI 必须包含非空的全局唯一“扩展 (extension)”标识符（使用 URI Fragment 形式，即 `#` 后的内容）。
* **CDDL 扩展机制**：
    * **组插座 (Group Socket)**：用于扩展现有数据字段定义。命名格式为 `* $$NAMESPACE-EVENTTYPE-extension`。使用 `//=` 运算符分配新字段。
    * **类型插座 (Type Socket)**：用于定义全新的事件数据结构。通过 `$/= ProtocolEventData` 运算符将新类型合并到全局事件枚举中。
* **事件重要性分级**：定义了三级重要性，用于指导记录与分析优先级：
    * **Core (核心)**：关联协议基础解析和核心指标。
    * **Base (基础)**：增加调试选项（如状态机变更），通常可从 Core 数据中推断，但显式记录更有利于分析。
    * **Extra (额外)**：用于实现层级的底层调试。

#### 发送端应遵守的设计

* **模式声明**：应该在 `Trace` 或 `TraceSeq` 的 `event_schemas` 字段中列出所有在用的事件模式 URI。这作为工具解析的“提示”。
* **扩展规范**：
    * 定义新事件时，**必须 (MUST)** 正确扩展 `$ProtocolEventData`。
    * **应该 (SHOULD)** 在新定义的事件数据中添加扩展插座（`* $$NAMESPACE-EVENTTYPE-extension`），以便其他模式后续扩展。
* **触发器 (Triggers)**：在需要解释事件原因（如丢包原因）时，**推荐 (RECOMMENDED)** 在 `data` 字段中包含一个名为 `trigger` 的可选字符串字段。
* **性能与隐私权衡**：
    * 如果出于性能或隐私不记录 Core 级别事件，**应该 (SHOULD)** 记录相关的 Base 级别事件（例如：不记录 `packet_received` 时，改记 `packets_acked`）。
    * 具体事件应定义其重要性级别。
* **设计原则**：
    * 避免记录重复数据：对于在多个包中保持不变的值（如连接 ID），应拆分为独立事件（如 `connection_id_updated`）。
    * 避免记录可推断事件：如果状态变化可以通过线缆数据（Wire image）直接推断（如流控限制变化），通常应避免记录额外事件，除非该事件具有独立识别价值。

#### 接收端应遵守的设计

* **解析容错性**：
    * **绝不能 (MUST NOT)** 因为跟踪中包含了未在 `event_schemas` 列出的事件，或缺失了已列出模式中的某些事件而报错。
    * **绝不能 (MUST NOT)** 遇到不认识的字段名或值而产生破坏性错误（Breaking errors）。
    * 对于未知的事件字段，应尝试可视化；若无法处理，则必须忽略。
* **重要性处理预期**：
    * **应该 (SHOULD)** 期望并支持所有 Core 级别事件。
    * 对于 Base 级别事件，至少应支持显示其原始内容。
    * 可根据需求选择性支持 Extra 级别事件。
* **触发器处理**：应利用 `trigger` 字段在高度并行或优化的实现中重建准确的事件上下文。


### 第9章 通用事件模式 (Generic Event Schemas)

#### 双方均应遵守的设计

* **通用性设计**：本章定义的两个模式旨在跨协议、应用和用例通用。它们通过扩展全局 `$ProtocolEventData` 来注入具体事件类型。
* **命名空间与 URI**：
    * **Loglevel（日志级别）**：用于复制传统的文本日志分类。命名空间为 `loglevel`，URI 为 `urn:ietf:params:qlog:events:loglevel`。
    * **Simulation（仿真）**：用于记录互操作性或基准测试中的环境变化。命名空间为 `simulation`，URI 为 `urn:ietf:params:qlog:events:simulation`。
* **CDDL 扩展预留**：所有定义的事件结构均包含 `* $$NAMESPACE-EVENTTYPE-extension` 插座，允许后续扩展。

#### 发送端应遵守的设计

* **Loglevel 事件应用**：
    * **error**：记录可能不会在网络线缆上反映的内部错误。**重要性级别：Core**。
    * **warning**：记录内部警告。**重要性级别：Base**。
    * **info / debug / verbose**：主要用于将 qlog 作为唯一日志格式时记录非结构化字符串。**重要性级别：Extra**。
* **Simulation 事件应用**：
    * **scenario**：用于指定当前正在测试的具体场景。支持将多个仿真任务聚合到单个跟踪中。**重要性级别：Extra**。
    * **marker**：用于标记特定仿真条件（如丢包、NAT 重绑定）被触发的时刻。**重要性级别：Extra**。
* **数据结构填充**：
    * `error` 和 `warning` 事件中，发送端可选择填充 `code` (uint64) 和 `message` (text)。
    * `scenario` 事件中，可使用 `details` 字段存放任意键值对来描述测试详情。

#### 接收端应遵守的设计

* **文本日志替代方案处理**：接收端应能识别 `loglevel` 命名空间下的事件名（如 `loglevel:error`），并将其视为传统日志级别的结构化等价物进行展示。
* **时间线关联**：对于 `simulation:marker` 事件，接收端工具应将其作为关键时间点（Milestone）进行可视化，以帮助用户分析网络环境变化与其他协议事件（如拥塞控制、丢包）之间的相关性。
* **重要性解析**：接收端应意识到 `simulation` 命名空间下的所有事件重要性均为 `Extra`，在资源受限的情况下可以选择不处理，但在性能测试分析工具中应优先予以支持。


### 第10章 原始数据包与帧信息

#### 双方均应遵守的设计

* **RawInfo 结构定义**：用于记录原始线缆镜像信息（如字节长度和原始字节值）。其 CDDL 定义如下：
  ```cddl
  RawInfo = {
      ? length: uint64          ; 实体的完整字节长度（含头部、负载、尾部）
      ? payload_length: uint64  ; 实体负载的字节长度（不含头部和尾部）
      ? data: hexstring         ; 实体的完整内容（可能是截断的），包含头部和尾部
  }
  ```
* **字段可选性与独立性**：`RawInfo` 中的所有字段都是可选的。可以只记录其中任何一个字段而省略其他字段。
* **长度一致性原则**：如果记录了长度相关的字段（`length` 或 `payload_length`），即使 `data` 字段被省略或截断，长度字段也必须指示该实体的**完整**长度。
* **头部/尾部长度计算逻辑**：
  * 规范未定义显式的 `header_length` 或 `trailer_length` 字段。
  * 在没有“尾部 (trailers)”的协议中，`header_length = length - payload_length`。
  * 在有尾部（例如 QUIC 的 AEAD 标签）的协议中，具体的事件定义文档应说明如何支持头部长度的计算。

#### 发送端应遵守的设计

* **数据分组**：必须将原始字节流相关的字段归类在名为 `raw` 的可选字段中。
* **隐私与安全保护**：
  * 强烈建议通过仅记录长度相关字段（`length`, `payload_length`）并省略 `data` 字段来执行协议调试，以降低泄露敏感数据的风险。
  * `data` 字段不要求包含实体的全部内容，可以根据需要进行截断。
* **字段复用建议**：对于在协议本身定义中就包含“在线长度 (on-the-wire length)”字段的实体（如数据包头或 QUIC 的 STREAM 帧），强烈建议直接复用 `raw.length` 字段，而不是定义重复的自定义长度字段，以保持一致性。

#### 接收端应遵守的设计

* **容错处理**：必须能够处理 `RawInfo` 中字段缺失的情况。
* **截断识别**：在解析 `data` 字段时，必须考虑到数据可能已被截断，不能假设 `data` 的实际字节数等于 `length` 字段的值。
* **协议特定解析**：接收端工具应查阅特定协议的事件定义文档，以了解如何根据 `RawInfo` 提供的字段正确计算头部或尾部长度。


### 第11章 qlog 序列化

#### 双方均应遵守的设计

* **序列化无关性**：qlog 模式定义在本质上应与特定的序列化格式无关。实施者在定义特定协议的事件时，也应保持这种无关性，除非明确说明支持的格式及用法。
* **格式权衡与选择**：选择格式时需考虑可用性、灵活性、互操作性和效率。
    * **JSON**：作为推荐的默认格式，具备高灵活性和互操作性，适合直接操作日志的用户。
    * **二进制格式**（如 CBOR、Protobuf）：比文本格式更高效，适合大规模操作。
* **基本编码要求**：
    * 对于 JSON 相关格式，必须使用 **UTF-8** 编码。
    * 序列化为 JSON 或 JSON-SEQ 时，所有字段名称必须为**小写**。
* **映射标准**：将 CDDL 定义映射到 JSON 时，**应该 (SHOULD)** 采用 [CDDL] 附录 E 定义的官方映射方案。
* **格式标识**：必须通过 `serialization_format` 字段（见第3章）明确指示所选的序列化格式，以便解析器正确匹配。

#### 发送端应遵守的设计

* **普通 JSON 序列化 (QlogFile)**：
    * **媒体类型**：`application/qlog+json`。
    * **文件后缀**：**应该 (SHOULD)** 使用 `.qlog`。
    * **局限性**：属于非流式格式，必须在文件末尾添加闭合标签（如 `]}]}`）才能保证通用解析器正常工作。
* **JSON 文本序列化 (JSON-SEQ/QlogFileSeq)**：
    * **媒体类型**：`application/qlog+json-seq`。
    * **文件后缀**：**应该 (SHOULD)** 使用 `.sqlog`。
    * **记录格式**：每个对象（记录）必须以 ASCII 记录分隔符（`<RS>`, 0x1E）开头，并以换行符（`\n`, 0x0A）结尾。
* **I-JSON 互操作性**：由于 JavaScript 无法处理完整的 64 位整数，发送端**可以 (MAY)** 将 `uint64` 值编码为 JSON 字符串。
* **截断值处理**：
    * 为了减小体积或保护隐私，可以对 `hexstring` 类型的原始数据进行截断。
    * **应该 (SHOULD)** 为所有 `hexstring` 字段提供一个独立的长度指示字段（如 `content_length`），以确保即使数据被截断或完全省略，原始长度信息仍可追溯。
* **性能优化**：对于大规模部署，建议对 JSON 或 JSON-SEQ 数据使用 GZIP 或 Brotli 算法进行压缩。

#### 接收端应遵守的设计

* **JSON-SEQ 解析逻辑**：
    * 接收端工具应能够解析 [RFC7464] 格式。可以通过将文件按记录分隔符（`<RS>`）拆分，并逐个将记录喂给普通的 JSON 解析器来完成。
    * 每个记录本身必须是一个有效的 JSON 对象。
* **数值解析鲁棒性**：
    * 为了兼容 I-JSON，解析器 **应该 (SHOULD)** 支持同时从 JSON 字符串和 JSON 数字中解析 `uint64` 值。
* **截断数据的还原与推断**：
    * 工具必须能够处理以下四种情况：
        1.  **长度与值均存在**：以长度字段为准。
        2.  **仅存在值**：通过 `字符串长度 / 2` 推断字节长度。
        3.  **仅存在长度**：表示原始值已被完全省略。
        4.  **长度与值不匹配**：表示值已被截断，此时必须保留长度字段所指示的原始大小信息。
* **新格式支持预留**：接收端应允许存在未知的字段，并尝试可视化未知数据，否则必须将其忽略。

### 第12章 访问与生成方法

#### 双方均应遵守的设计

* **环境变量控制**：为了统一用户对 qlog 文件生成位置和方式的控制，定义了两个标准环境变量：
    * **`QLOGFILE`**：指向单个 qlog 文件的完整路径。该路径**必须 (MUST)** 包含完整的文件扩展名。
    * **`QLOGDIR`**：指向放置 qlog 文件的通用目录路径。该路径**必须 (MUST)** 在末尾包含目录分隔符（如 `/` 或 `\`）。
* **变量选择逻辑**：
    * 如果端点倾向于生成多个 qlog 文件（例如为每个连接创建独立文件的服务器），则**应该 (SHOULD)** 优先使用 `QLOGDIR`。
    * 如果所有连接的事件都记录在单个文件中，则使用 `QLOGFILE`，并依赖 `group_id` 字段（见第 7.3 节）进行事后事件分离。

#### 发送端应遵守的设计

* **支持要求**：
    * 发送端实现**应该 (SHOULD)** 支持 `QLOGDIR` 环境变量。
    * 发送端实现**可以 (MAY)** 支持 `QLOGFILE` 环境变量。
* **文件命名规范 (使用 `QLOGDIR` 时)**：
    * 发送端可自行选择合适的命名方案，通常取决于上下文或协议。
    * **针对 QUIC 协议的推荐方案**：建议使用“原始目标连接 ID (ODCID)”后接“观察点类型 (Vantage Point Type)”。例如：`abcde_server.qlog`。
* **单文件输出逻辑 (使用 `QLOGFILE` 时)**：
    * 当发送端（如 QUIC 服务器）将多个连接的日志写入同一个 `QLOGFILE` 指定的文件时，**必须**确保每条事件实例都标记了正确的 `group_id`，以便后续处理。
* **执行示例**：
    * `QLOGFILE=/path/file.qlog binary`：生成单一指定文件。
    * `QLOGDIR=/path/ binary`：在目录下根据协议标识（如 ODCID）生成多个文件。

#### 接收端应遵守的设计

* **日志发现与关联**：
    * 接收端（工具）在处理由 `QLOGDIR` 生成的文件时，应能够识别发送端采用的命名模式（如 ODCID 命名），以便将日志文件与其代表的逻辑实体自动关联。
* **多流分离处理**：
    * 当接收端处理由 `QLOGFILE` 生成的包含多个逻辑实体（如多个连接）的单一日志文件时，**必须**具备根据 `group_id` 字段提取和隔离特定组事件的能力。


### 第13章 工具要求

#### 双方均应遵守的设计

* **默认格式支持**：工具应该 (SHOULD) 至少支持默认 JSON 格式（见第 11.1 节）的 `.qlog` 文件。
* **日志不完整性预期**：双方必须意识到日志事件在某些情况下是不完整的，原因可能包括：
    * 使用了固定大小的循环日志缓冲区（早期事件如连接设置可能被新事件覆盖）。
    * 出于隐私保护或减小文件体积的考虑而有意省略某些事件。

#### 发送端应遵守的设计

* **提供逻辑执行基础**：发送端生成的日志应当尽量包含工具执行其逻辑所需的核心字段。虽然允许截断或省略，但应意识到这可能限制工具的分析能力。

#### 接收端应遵守的设计

* **支持能力声明**：工具在接入 qlog 时 **必须 (MUST)** 明确标示其支持的以下范围：
    * qlog 版本与序列化格式。
    * 支持的 qlog 文件模式 (File Schemas) 与事件模式 (Event Schemas)。
    * 支持的压缩方法。
    * 其他潜在的支持格式（例如 `.pcap`）。
* **逻辑依赖说明**：工具 **应该 (SHOULD)** 明确说明其逻辑执行所依赖的具体 `name`（命名空间:事件类型）值及其 `data` 字段中的特定属性。
* **模式校验**：工具 **应该 (SHOULD)** 对输入的 qlog 文件进行高级别检查，验证其是否符合预期的文件和事件模式。
* **错误与鲁棒性处理**：
    * **信息不足提示**：若工具判定 qlog 文件中缺少足够信息来正确执行逻辑，**应该 (SHOULD)** 生成清晰的错误消息。
    * **非破坏性处理**：对于不认识的字段名或值，工具 **绝不能 (MUST NOT)** 产生破坏性错误（如崩溃）。
    * **未知事件可视化**：工具 **应该 (SHOULD)** 在其上下文中指示出未知的事件（例如在时间线上标记出未知事件，以便用户手动解读）。
    * **不完整日志处理**：工具作者被鼓励增强工具的鲁棒性，使其在面对不完整日志时仍能提供适度的分析输出。


### 第14章 安全与隐私考量

#### 双方均应遵守的设计

* **安全保护意识**：双方必须认识到，尽管 TLS [RFC8446] 和 QUIC [RFC9000] 保护了线缆镜像（Wire Image），但 qlog 日志记录可能会泄露原本受保护的信息。设计时必须权衡可观测性（Observability）与安全性/隐私之间的冲突。
* **关联风险防范**：必须意识到日志数据可能跨数据源被关联。在处理包含 IP 地址、连接 ID 或高精度时间戳的数据时，应防范通过关联分析识别出个人用户或会话的风险。
* **加密要求**：**强烈建议** qlog 文件在传输过程中和存储静态状态时均进行加密处理。
* **合规性参考**：所有参与方应遵循 [RFC6973] 中关于隐私风险评估的指南，意识到数据一旦泄露将威胁所有参与者的隐私。

#### 发送端应遵守的设计

* **显著性开启机制（Conspicuous Enabling）**：日志捕获的开启必须是显式的且易于察觉的（例如需要明确的操作）。必须能够抵抗社交工程、自动化攻击或路过式（drive-by）攻击。
* **隔离设计**：建议将日志捕获逻辑与同一进程或组件中的其他活动进行隔离或沙箱化处理，以防止跨组件的隐私泄露。
* **细粒度控制**：实现方案应提供细粒度的控制，允许根据具体用例或每个连接（per-connection）来决定是否记录特定字段。
* **敏感数据最小化**：
    * **标识符处理**：谨慎记录 IP 地址、传输端口、QUIC 连接 ID、进程/线程 ID 等。
    * **状态与凭据处理**：原则上应避免或通过最小化手段记录地址验证令牌、Retry 令牌、TLS 会话票据、HTTP Cookie、TLS 解密密钥、密码及 API 授权令牌。
    * **原始负载（Raw Payload）**：对于 `RawInfo` 类型的字段（见第10章），除非在极少数严格用例下，否则应限制记录。
* **数据脱敏技术**：建议支持字段删除、匿名化（Anonymization）、伪名化（Pseudonymization）、置换、截断、重加密或聚合等脱敏手段。特别是在记录 `hexstring` 类型的原始数据时，应采取截断措施。

#### 接收端应遵守的设计

* **访问控制与审计**：对 qlog 的捕获、读取和存储访问必须实施严格的访问控制和审计。
* **基于权限的采集分级**：应根据角色和权限支持不同级别的采集精度（例如：采集高敏感数据必须要求更高的权限等级）。
* **数据保留政策（Retention Policies）**：必须为 qlog 文件的存储定义明确的数据保留期限和自动删除政策。
* **风险等效性认知**：操作人员必须理解，访问 Internet 协议的 qlog 日志在很大程度上等同于拥有读取明文通信的能力，应适用所有针对明文访问的安全考虑。
* **尊重用户偏好**：工具和操作环境应尽量将 qlog 的捕获、存储和删除操作与现有的用户隐私控制偏好对齐，支持激进的删除或聚合策略。

### 第15章 IANA 考量

#### 双方均应遵守的设计
* **注册参数标识符**：`qlog` 已在“IETF URN Sub-namespace for Registered Protocol Parameter Identifiers”注册表中注册。双方在处理 URN 相关的参数时，应将其视为标准化的协议参数标识符。
* **IANA 注册表基准**：所有 qlog 日志文件模式及事件模式的官方定义和引用应参考 IANA 维护的地址：`https://www.iana.org/assignments/qlog`。
* **URI 唯一性**：双方必须认可在 IANA 注册的 URI 作为模式（Schema）的唯一身份标识。

#### 发送端应遵守的设计
* **日志文件模式声明**：发送端在生成文件时，必须根据文件结构使用以下已注册的 URI 进行声明：
  | 日志文件模式 URI | 描述 | 参考章节 |
  | :--- | :--- | :--- |
  | `urn:ietf:params:qlog:file:contained` | 可包含来自多个观察点的多个跟踪（traces）的具体日志文件模式。 | 第4章 |
  | `urn:ietf:params:qlog:file:sequential` | 包含单个跟踪、针对顺序读写访问优化的具体日志文件模式。 | 第5章 |
* **事件模式及命名空间应用**：发送端在使用以下通用事件集时，必须确保 URI、命名空间与事件类型的对应关系：
  * **LogLevel 模式**：
    * **URI**: `urn:ietf:params:qlog:events:loglevel`
    * **命名空间**: `loglevel`
    * **可用事件类型**: `error`, `warning`, `info`, `debug`, `verbose`
    * **用途**: 用于自由文本的常用日志级别。
  * **Simulation 模式**：
    * **URI**: `urn:ietf:params:qlog:events:simulation`
    * **命名空间**: `simulation`
    * **可用事件类型**: `scenario`, `marker`
    * **用途**: 用于仿真测试相关的事件。

#### 接收端应遵守的设计
* **注册表支持**：接收端工具应具备识别并解析 IANA “qlog log file schema URIs” 和 “qlog event schema URIs” 注册表中条目的能力。
* **事件类型解析**：接收端必须支持识别上述两个预定义模式中的具体事件类型列表：
  * 应能正确处理 `loglevel` 命名空间下的错误、警告、信息、调试及冗长日志。
  * 应能识别 `simulation` 命名空间下的场景切换（scenario）和标记（marker）事件。
* **模式信息对齐**：当接收端需要获取未硬编码的模式详细信息时，应参照 IANA 注册表模板中的“命名空间（Namespace）”、“事件类型（Event Types）”及“描述（Description）”字段进行逻辑处理。
