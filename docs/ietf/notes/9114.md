### 1. 简介 (Introduction)

#### 双方均应遵守的设计
* **核心协议映射**：HTTP/3 必须使用 QUIC 作为底层的传输协议，并在其之上承载并维持标准的 HTTP 语义。
* **安全性与可靠性**：必须依赖 QUIC 协议来提供数据的机密性（confidentiality）与完整性（integrity）保护、对等端身份验证（peer authentication），以及可靠的、按序的、基于单流（per-stream）的数据交付。
* **内置加密**：需支持 QUIC 在传输层集成的 TLS 1.3，以实现与 TCP 上的 TLS 相当的安全级别，并获得类似 TCP Fast Open (TFO) 的低延迟连接建立优势。
* **生命周期与流量控制**：必须将流的生命周期管理（stream lifetime）和流量控制（flow-control）委托给 QUIC 协议处理。
* **分帧机制**：在每个独立的 QUIC 流上，必须使用类似于 HTTP/2 的二进制分帧（binary framing）机制。
* **多路复用与拥塞控制**：利用 QUIC 提供的流多路复用和连接级别的全局拥塞控制，以避免 HTTP/2 中因 TCP 丢包或乱序导致的全局事务停滞（即队头阻塞问题）。

#### 发送端应遵守的设计
* **数据封装与映射**：发送端必须将 HTTP 消息语义正确映射到 QUIC 协议上，避免使用 HTTP/1.1 容易导致解析复杂性的纯文本空白字符分隔机制，而是采用二进制分帧封装数据。
* **连接复用优化**：发送端应充分利用 QUIC 的单连接多路复用特性来并行处理请求，避免像 HTTP/1.1 那样建立多个平行的 TCP 连接从而降低网络效率。

#### 接收端应遵守的设计
* **独立流处理**：接收端必须能够独立处理 QUIC 多路复用层传来的各个流。当发生网络丢包或数据包乱序时，接收端不应使所有活跃事务停滞，而应仅等待受直接影响的流进行恢复。
* **分帧解析**：接收端必须能够从 QUIC 提供的可靠按序数据流中，正确提取和解析 HTTP 二进制分帧，并还原为标准的 HTTP 语义。


### 2. HTTP/3 协议概述 (HTTP/3 Protocol Overview)

#### 双方均应遵守的设计
* **底层传输与分帧**：必须使用 QUIC 作为传输协议来承载 HTTP 语义，并在内部使用类似于 HTTP/2 的分帧层（framing layer）。
* **连接与流管理**：必须依赖 QUIC 提供的协议协商（protocol negotiation）、基于流的多路复用（stream-based multiplexing）以及流量控制（flow control）功能。
* **通信基本单位**：在每个流（stream）内，HTTP/3 通信的最基本单位是“帧（frame）”。每种类型的帧具有不同的用途（例如，HEADERS 和 DATA 帧构成了 HTTP 请求和响应的基础）。
* **控制流分离**：作用于整个 HTTP/3 连接的帧，必须通过专用的控制流（dedicated control stream）进行传输，不得与常规数据流混淆。
* **独立的流多路复用**：每个“请求-响应”对必须独占一个单一的 QUIC 流。各个流之间必须保持完全独立，任何一个流的阻塞或数据包丢失，都不得阻碍其他流的进度（解决队头阻塞问题）。
* **首部压缩机制 (QPACK)**：由于 QUIC 不提供跨流的按序传输保证，双方必须使用 QPACK（而非 HTTP/2 中的 HPACK）来压缩请求和响应字段。
* **状态表管理**：在使用 QPACK 时，必须使用独立的单向流（unidirectional streams）来修改和跟踪字段表的状态；而实际的编码字段部分在传输时仅引用该表的状态，不得在数据流中直接修改它。
* **错误定义与区分**：双方必须明确区分两种级别的错误：“流错误（stream error）”指发生在单个流上的应用层错误；“连接错误（connection error）”指影响整个 HTTP/3 连接的全局错误。在遇到严重错误时，可能需要引发中止（abort）操作，即突然终止连接或流。
* **帧术语的区分**：双方在解析协议栈时，必须明确区分 QUIC 层的帧（例如 `QUIC CONNECTION_CLOSE` 帧）与 HTTP/3 应用层的帧。

#### 发送端应遵守的设计
* **流数据封装**：发送端必须将传输内容（无论是请求还是响应）正确拆分成由“头部（header）”和“基于帧类型结构化的可变长度字节序列”组成的 HTTP/3 帧。
* **服务器推送 (Server Push) 管理**：当服务器（作为发送端）预期客户端将发起某项请求，并希望提前推送“请求-响应”交互以换取延迟收益时，必须使用特定的 HTTP/3 帧来管理推送流程，例如 `PUSH_PROMISE`、`MAX_PUSH_ID` 和 `CANCEL_PUSH`。
* **整数编码**：发送端在编码数据时，必须使用来自 QUIC 传输协议定义的变长整数编码（variable-length integer encoding）格式。

#### 接收端应遵守的设计
* **端点发现与连接发起**：客户端（作为请求数据的接收期望方）在获知特定端点存在 HTTP/3 服务器后，必须主动发起并建立 QUIC 连接（即 HTTP/3 连接）。
* **异步帧处理**：接收端必须能够并发且独立地处理来自底层 QUIC 传来的多个流中的数据帧。当某个流发生丢包时，接收端需继续处理其他不受影响的流。
* **帧类型解析**：接收端必须根据接收到的 HTTP/3 帧的头部信息，准确识别帧的类型，并按照该类型规定的结构对后续的变长字节序列进行解析。


### 3. 连接建立与管理 (Connection Setup and Management)

#### 双方均应遵守的设计
* **底层协议基础**：必须依赖 QUIC 版本 1（或未来规范定义的版本）作为底层传输协议，且其握手协议必须为 TLS 1.3 或更高版本。
* **协议协商 (ALPN)**：在建立连接及 TLS 握手期间，必须使用 ALPN 扩展令牌 `"h3"` 来指示对 HTTP/3 的支持。
* **初始化控制流**：在 QUIC 连接建立后，每个端点必须（MUST）发送 `SETTINGS` 帧，且该帧必须是各自 HTTP 控制流（control stream）的首个数据帧。
* **持久连接与优雅关闭**：HTTP/3 连接在多个请求之间是持久的。当任一端点选择关闭连接时，终止端点应该（SHOULD）首先发送 `GOAWAY` 帧，以确保双方能可靠地确认先前发送的帧是否已处理，并优雅地完成或终止所有剩余任务。

#### 发送端应遵守的设计
* **端口选择**：服务端（作为响应发送端）可以在任何 UDP 端口上提供 HTTP/3 服务。
* **替代服务通告 (Alt-Svc)**：服务端可以通过在 HTTP 响应中包含 `Alt-Svc` 头部字段，或发送带有 `"h3"` 令牌的 `ALTSVC` 帧，来向客户端通告等效 HTTP/3 端点的可用性及对应端口。
* **空闲连接管理**：鼓励服务端尽可能长时间地维持打开的 HTTP/3 连接，但也允许其在必要时主动终止空闲连接。
* **控制连接复用**：如果服务端不希望客户端针对某个特定的源（origin）复用当前的 HTTP/3 连接，它可以通过返回 `421 (Misdirected Request)` 状态码来响应请求，以此声明自身对该请求不具权威性。

#### 接收端应遵守的设计
* **目标主机指示 (SNI)**：客户端（作为请求发起与接收端）必须（MUST）支持在 TLS 握手期间向服务器指示目标主机。如果服务器由域名标识，客户端必须发送 SNI (Server Name Indication) 扩展，除非使用了其他的目标主机指示机制。
* **证书权威性验证**：在 TLS 握手接收到服务器证书后，客户端必须（MUST）验证该证书是否与目标 URI 的源服务器匹配。如果无法验证，客户端绝对不能（MUST NOT）将该服务器视为对该源具有权威性。
* **连接回退机制**：如果因网络连通性问题（如 UDP 端口被封锁）导致建立 QUIC 连接失败，客户端应该（SHOULD）尝试回退使用基于 TCP 的历史 HTTP 版本。
* **替代服务连接尝试**：在接收到指示支持 HTTP/3 的 `Alt-Svc` 记录后，客户端可以（MAY）尝试向指定的受信任主机和端口建立 QUIC 连接。
* **非 HTTPS 方案处理**：HTTP/3 不能直接用于访问 `"http"` 方案标识的权威服务器（因为没有 TCP）。在为非 `"https"` 方案的源发起请求前，客户端必须（MUST）确保服务器愿意为该方案提供服务。
* **跨源连接复用限制**：
  * 客户端可以将现有连接复用于具有不同 URI 权威组件的请求，但前提是必须（MUST）验证服务器为新源提供的证书的合法性。客户端需要保留证书及其验证所需的信息以支持此操作。
  * 如果新源的证书不可接受，客户端绝对不能（MUST NOT）复用该连接，而应该（SHOULD）为新源建立新连接。
  * 如果某证书验证失败（如过期或被吊销），客户端应该（SHOULD）重新验证依赖该证书建立权威性的所有其他源。
* **并发连接限制**：客户端不应该（SHOULD NOT）使用相同的传输或 TLS 配置，向同一个 IP 地址和 UDP 端口打开多个 HTTP/3 连接。可以在配置不同的情况下建立多个连接，但应尽量避免配置重复。


### 4. 在 HTTP/3 中表达 HTTP 语义 (Expressing HTTP Semantics in HTTP/3)

#### 双方均应遵守的设计
* **HTTP 消息分帧**：HTTP 消息（请求或响应）由以下部分组成：1. 包含控制数据的头部区块（单个 `HEADERS` 帧）；2. 可选的内容区块（一系列 `DATA` 帧）；3. 可选的尾部区块（单个 `HEADERS` 帧）。
* **非法的帧序列**：接收到无效的帧序列（如在 `HEADERS` 之前出现 `DATA` 帧，或在尾部 `HEADERS` 之后出现 `HEADERS` 或 `DATA` 帧）必须被视为类型为 `H3_FRAME_UNEXPECTED` 的连接错误。
* **状态依赖与处理顺序**：`HEADERS` 和 `PUSH_PROMISE` 帧可能会更新 QPACK 动态表。虽然这些更新不直接属于消息交换的一部分，但必须在消费消息之前接收并处理它们。
* **禁止的传输编码**：HTTP/3 中未定义传输编码（Transfer codings），绝对不能（MUST NOT）使用 `Transfer-Encoding` 头部字段。
* **流与消息的生命周期**：一次 HTTP 请求/响应交换会完全消耗一个双向 QUIC 流。流的关闭标志着最终 HTTP 消息的结束。
* **请求取消**：任一端点均可取消请求。取消操作应当通过重置流的发送端并中止流的接收端读取来实现。
* **畸形消息处理**：包含违禁字段、缺失必填伪头部、伪头部在普通字段之后、大写字段名、无效字符或 `Content-Length` 与实际收到数据长度不符的消息均为畸形（Malformed）。代理/中介绝对不能（MUST NOT）转发畸形消息，检测到畸形消息必须作为 `H3_MESSAGE_ERROR` 类型的流错误处理。
* **HTTP 字段与压缩**：字段名称必须在编码前转换为小写。必须使用 QPACK 压缩头部和尾部区块。绝对不能生成包含连接特定字段（Connection-specific fields）的 HTTP/3 字段区块（唯一的例外是 `TE` 字段，且其值只能为 "trailers"），中介在转换时必须移除这些字段。
* **控制数据（伪头部）**：伪头部字段（以 `:` 开头）不是常规 HTTP 字段。必须出现在常规头部字段之前。绝对不能在请求中包含响应的伪头部，反之亦然；绝对不能在尾部区块中出现伪头部。
* **不支持 HTTP Upgrade**：HTTP/3 不支持 HTTP Upgrade 机制及 101 (Switching Protocols) 状态码。

#### 发送端应遵守的设计
* **请求流的使用（客户端）**：客户端必须在一个给定的客户端发起的双向 QUIC 流上仅发送单一请求。发送完请求后，客户端必须关闭流的发送方向（使用 CONNECT 方法除外）。客户端不应依赖接收到响应来决定是否关闭流。
* **响应流的使用（服务端）**：服务端在请求流上发送 0 个或多个临时响应（1xx，不含内容或尾部），随后发送一个最终响应。发送完最终响应后，服务端必须关闭流的发送方向。
* **提前响应与终止请求（服务端）**：如果服务端在接收完整请求前已能提供完整响应，它可以中止读取请求流、发送完整响应并干净地关闭流的发送部分。建议使用 `H3_NO_ERROR` 错误码请求客户端停止发送请求。
* **取消与拒绝的错误码（服务端）**：如果服务端在未进行任何应用层处理时取消请求（即“拒绝”），应使用 `H3_REQUEST_REJECTED` 中止响应流。如果已进行了部分或全部处理后放弃响应，应使用 `H3_REQUEST_CANCELLED`。绝对不能对已处理的请求使用 `H3_REQUEST_REJECTED`。
* **取消请求的错误码（客户端）**：客户端应使用 `H3_REQUEST_CANCELLED` 来取消请求。除了响应服务端要求的关闭外，客户端绝对不能使用 `H3_REQUEST_REJECTED`。
* **请求伪头部（客户端）**：除 CONNECT 请求外，所有请求必须包含确切的一个 `:method`、`:scheme` 和 `:path`（http/https 不能空，至少为 `/`）。如果协议包含强制授权组件，还必须包含 `:authority` 或 `Host` 头部（推荐 `:authority`），若同时存在两者值必须相同。
* **响应伪头部（服务端）**：所有响应必须包含单一的 `:status` 伪头部，携带状态码。
* **CONNECT 方法封装（客户端）**：构造 CONNECT 请求时，`:method` 设为 "CONNECT"，省略 `:scheme` 和 `:path`，`:authority` 包含目标主机和端口。请求完成后，该流上只允许发送 `DATA` 帧。
* **服务器推送管理（服务端）**：
  * 必须在客户端允许的 `MAX_PUSH_ID` 范围内按顺序分配 Push ID。
  * 只能推送可缓存、安全、无请求体且无尾部的请求，且服务端必须对 `:authority` 具有权威性。
  * `PUSH_PROMISE` 必须包含在生成推送的请求流中发送。应在发送引用该承诺响应的 `HEADERS` 或 `DATA` 帧之前发送 `PUSH_PROMISE`。
* **首部大小限制**：发送端在收到对端通过 `SETTINGS_MAX_FIELD_SECTION_SIZE` 发送的头部大小限制参数后，不应发送超过该大小的 HTTP 消息头部。

#### 接收端应遵守的设计
* **多重消息处理**：在给定流上，接收到多个请求或在最终响应后接收到额外的 HTTP 响应，必须视为畸形。
* **不完整请求处理（客户端/服务端）**：如果客户端发起的流在未提供完整响应时终止，服务端应使用 `H3_REQUEST_INCOMPLETE` 中止响应流。由于消息可能很大，接收端应在接收到足够多的片段时就开始处理部分消息。
* **处理提前终止的响应（客户端）**：客户端绝对不能因为请求被服务端突然终止（提前响应）而丢弃已收到的完整响应。
* **安全重试（客户端）**：如果流在接收到部分响应后被取消，则不应使用该响应。客户端不应自动重试非幂等方法的请求，除非请求被明确视为“已拒绝”（`H3_REQUEST_REJECTED`）。
* **Cookie 解码（中介/接收端）**：为了提升压缩率，`Cookie` 头部可能被拆分为多行。如果在非 HTTP/2 或 HTTP/3 上下文（如传给 HTTP/1.1 或后端应用）中使用，解压后必须使用双字节分隔符 `; ` (ASCII 0x3b, 0x20) 将它们拼接成单一字符串。
* **CONNECT 隧道处理（代理服务端）**：
  * 支持 CONNECT 的代理需要与目标建立 TCP 连接。成功后向客户端发送包含 2xx 状态码的 `HEADERS` 帧。
  * 将收发到的 `DATA` 帧与 TCP 连接的数据相互转换转发。
  * 任何 TCP 连接错误（如收到 RST 报文段）都应作为类型为 `H3_CONNECT_ERROR` 的流错误处理。
  * 当客户端结束请求流或发生 QUIC 错误时，代理必须关闭或重置 TCP 连接的对应方向。
* **服务器推送处理（客户端）**：
  * 如果未发送 `MAX_PUSH_ID` 帧或收到的 Push ID 超出最大值，必须将其视为 `H3_ID_ERROR` 连接错误。
  * 客户端必须对推送的来源进行权威性验证。如果验证失败，绝对不能认为该服务端具有权威性。
  * 对于不可缓存、不安全、带有请求内容或不具权威性的 `PUSH_PROMISE`，客户端应发送 `CANCEL_PUSH` 帧予以拒绝，且绝对不能使用或缓存相关响应。
  * 若推送流数据先于 `PUSH_PROMISE` 到达，客户端可缓冲数据；若合理时间内未收到匹配的 Promise，应中止读取并丢弃数据。如果客户端已取消推送却仍收到数据，可以 `H3_REQUEST_CANCELLED` 中止读取。


### 5. 连接关闭 (Connection Closure)

#### 双方均应遵守的设计
* **空闲超时处理**：如果 QUIC 连接的空闲时间超过了握手期间协商的超时时间，对端将假定连接已关闭。此时必须为新请求建立新的 HTTP/3 连接；当连接接近空闲超时限制时，也应该（SHOULD）主动建立新连接。
* **优雅关闭与 GOAWAY 帧**：任一端点均可通过发送 `GOAWAY` 帧来发起连接的优雅关闭。该帧包含一个标识符：服务器发送客户端发起的双向流 ID；客户端发送 Push ID。发送端将拒绝处理标识符大于或等于该值的请求或推送（如果没有处理任何请求/推送，该值可以为 0）。
* **清理底层状态**：发送 `GOAWAY` 帧后，端点应该（SHOULD）显式取消标识符大于或等于所指示标识符的任何请求或推送，以清理受影响流的传输层状态。随着更多请求/推送到达，应继续执行此清理操作。
* **多次发送 GOAWAY 的限制**：端点可以发送多个指示不同标识符的 `GOAWAY` 帧，但每个帧中的标识符绝对不能（MUST NOT）大于任何先前帧中的标识符。收到包含比先前更大标识符的 `GOAWAY` 帧时，必须将其视为类型为 `H3_ID_ERROR` 的连接错误。
* **两阶段优雅关闭策略**：尝试优雅关闭的端点可以先发送带有最大可能值的 `GOAWAY` 帧（服务器为 2^62-4，客户端为 2^62-1），以确保对端停止创建新请求或推送。等待飞行中的请求或推送到达后，再发送第二个 `GOAWAY` 帧以指示最终的接受范围。
* **完成优雅关闭**：处理完所有接受的请求和推送后，端点可以允许连接进入空闲状态，也可以立即发起连接关闭。完成优雅关闭的端点在关闭连接时应该（SHOULD）使用 `H3_NO_ERROR` 错误码。
* **立即应用层关闭**：HTTP/3 实现可以随时通过发送 QUIC `CONNECTION_CLOSE` 帧立即关闭连接，并使用应用层错误码说明原因。未显式关闭的打开流将随着连接的关闭而隐式关闭。建议在发送 `CONNECTION_CLOSE` 帧的同一个数据包中包含一个 `GOAWAY` 帧，以增加客户端成功重试请求的机会。

#### 发送端应遵守的设计
* **空闲连接的维持策略**：当有未完成的请求或服务器推送响应时，期望客户端请求传输层保持连接打开。如果不期望收到响应，允许连接超时优先于维持无用的连接。网关可以预见性地维持连接，但服务器不应该（SHOULD NOT）主动保持连接打开。
* **提前发送 GOAWAY**：当服务器预先知道要关闭连接时（即使提前量很小），应该（SHOULD）发送 `GOAWAY` 帧，以便客户端能够明确判断其请求是否已被部分处理。
* **客户端推送控制**：客户端在发送 `GOAWAY` 帧时可以灵活设置 Push ID：设置为 2^62-1 表示服务器可以继续完成已承诺的推送；设置较小的值则表示客户端将拒绝大于或等于该值的推送。
* **流 ID 耗尽的特殊情况**：如果客户端已经耗尽了所有可用的双向流 ID，服务器不需要发送 `GOAWAY` 帧，因为客户端已经无法发起进一步的请求。

#### 接收端应遵守的设计
* **停止发起新交互**：在收到对端的 `GOAWAY` 帧后，绝对不能（MUST NOT）在该连接上发起新请求或承诺新推送。客户端可以（MAY）建立新连接来发送额外的请求。
* **处理未决请求（客户端）**：
  * 收到 `GOAWAY` 帧后，客户端流 ID 大于或等于帧中标识符的请求将不会被处理，客户端可以安全地在另一个连接上重试这些请求。如果无法重试，这些飞行中的请求将丢失。
  * 流 ID 小于帧中标识符的请求可能已被处理，其状态在收到响应、流被单独重置、收到包含更小流 ID 的后续 `GOAWAY` 帧或连接终止之前是未知的。
* **拒绝未处理的低 ID 请求（服务端）**：如果流 ID 小于 `GOAWAY` 标识符的请求实际上并未被处理，服务器可以（MAY）单独拒绝这些请求。
* **处理未决推送（服务端）**：如果服务器在承诺了推送之后收到客户端的 `GOAWAY` 帧，且其 Push ID 大于或等于帧中的标识符，这些推送将不会被客户端接受。
* **底层传输终止的回退假设**：如果连接因底层传输原因（如对端显式关闭、传输层错误或网络中断）终止，且客户端未收到 `GOAWAY` 帧，客户端必须（MUST）假设任何已发送的请求（无论是全部还是部分）都有可能已被服务器处理。


### 6. 流的映射与使用 (Stream Mapping and Usage)

#### 双方均应遵守的设计
* **多路复用与映射**：HTTP/3 必须依赖 QUIC 层来处理多路复用。在 QUIC 流上发送的数据必须直接映射到特定的 HTTP 事务或整个 HTTP/3 连接上下文中，HTTP 自身不需要进行额外的多路复用。
* **按序交付机制**：HTTP/3 依赖 QUIC 提供流内（within a stream）的可靠、按序字节交付，但不要求跨流（across streams）的交付顺序。HTTP/3 不使用 QUIC 允许的流内乱序交付特性。
* **单向流配额要求**：双方在连接建立时的传输参数（transport parameters）必须（MUST）允许对端创建至少 3 个单向流（用于 HTTP 控制流和 2 个 QPACK 流）。这些传输参数应该（SHOULD）为每个单向流提供至少 1024 字节的流量控制信用额度。
* **关键流的优先创建**：端点应该（SHOULD）首先创建 HTTP 控制流以及强制扩展（如 QPACK 编码器和解码器流）所需的单向流，然后再根据对端允许的额度创建额外的流。
* **控制流的唯一性与生命周期**：
  * 类型为 `0x00`。连接开始时，双方必须（MUST）各自发起且仅发起一个控制流。
  * 发送端绝对不能（MUST NOT）主动关闭控制流，接收端也绝对不能（MUST NOT）请求发送端关闭控制流。如果任何一个控制流在任何时候被关闭，必须（MUST）将其视为类型为 `H3_CLOSED_CRITICAL_STREAM` 的连接错误。
  * 接收到声称是控制流的第二个流，必须（MUST）视为类型为 `H3_STREAM_CREATION_ERROR` 的连接错误。
* **保留流类型的忽略机制**：格式为 `0x1f * N + 0x21`（N 为非负整数）的流类型为保留流，用于测试忽略未知流类型的机制。这些流没有语义，端点收到这些流时绝对不能（MUST NOT）赋予其任何含义。

#### 发送端应遵守的设计
* **单向流头部声明**：发送端在发起单向流时，必须在流的起始位置发送一个变长整数（variable-length integer）作为流类型（Stream Type），以指示该流的用途。
* **控制流初始化**：发送端必须（MUST）将其 `SETTINGS` 帧作为控制流上的第一个帧发送。
* **未知流类型的发送限制**：发送端可以在不知道对端是否支持的情况下发送某些流类型；但是，对于可能修改现有协议组件（包括 QPACK 或其他扩展）状态或语义的流类型，在确认对端支持之前绝对不能（MUST NOT）发送。
* **保留流的发送与终止**：发送端可以自行决定保留流的有效载荷和长度，并在发送时可以（MAY）干净地终止该流或重置它。如果重置，应该（SHOULD）使用 `H3_NO_ERROR` 错误码或保留错误码。
* **双向请求流容量配置（服务端）**：为了不必要地限制并行性，HTTP/3 服务端应该（SHOULD）为允许的双向流数量和初始流流量控制窗口配置非零的最小值，且每次应该（SHOULD）至少允许 100 个请求流。
* **推送流封装（服务端）**：只有服务端可以发起推送流（类型为 `0x01`）。发送端必须在流类型之后附加其履行的承诺的 Push ID（编码为变长整数），且每个 Push ID 在推送流头部中必须（MUST）只使用一次。

#### 接收端应遵守的设计
* **未授权双向流处理（客户端）**：HTTP/3 默认不使用服务端发起的双向流。除非通过扩展协商同意，否则客户端必须（MUST）将收到服务端发起的双向流视为类型为 `H3_STREAM_CREATION_ERROR` 的连接错误。
* **未知单向流处理**：
  * 如果收到不支持的单向流类型，接收端必须（MUST）中止读取该流或直接丢弃传入数据。如果中止读取，应该（SHOULD）使用 `H3_STREAM_CREATION_ERROR` 或保留的错误码。
  * 接收端绝对不能（MUST NOT）将未知流类型视为任何形式的连接错误。
  * 在读取到流类型之前，接收端不应该（SHOULD NOT）丢弃来自传入单向流的数据（因为某些流类型可能会影响连接状态）。
  * 接收端必须（MUST）容忍单向流在接收到流头部之前被发送端关闭或重置。
* **控制流的首帧验证**：如果接收端在控制流上收到的第一个帧不是 `SETTINGS` 帧，必须（MUST）将其视为类型为 `H3_MISSING_SETTINGS` 的连接错误。
* **控制流流量窗口管理**：接收端应该（SHOULD）为对端的控制流提供足够的流量控制信用额度，以防止其被阻塞。
* **推送流处理（客户端）**：
  * 客户端不应该（SHOULD NOT）在读取到推送流头部之前中止读取推送流，以避免双方对哪些 Push ID 已被消耗产生分歧。
  * 如果客户端检测到推送流头部包含的 Push ID 在其他推送流头部中已被使用过，必须（MUST）将其视为类型为 `H3_ID_ERROR` 的连接错误。
  * 如果服务端（或客户端）收到了“客户端发起的推送流”，必须（MUST）将其视为类型为 `H3_STREAM_CREATION_ERROR` 的连接错误。

### 8. 错误处理 (Error Handling)

#### 双方均应遵守的设计
* **错误分类与语义区分**：必须明确区分“流错误（stream error）”与“连接错误（connection error）”。流错误指中止单个流，表示发送端未能完整传输或消费请求/响应；连接错误指中止整个 QUIC 连接。这两者均不同于 HTTP 状态码（状态码表示成功接收请求后的应用层处理结果）。
* **非故障性错误操作**：关闭流或连接的所谓“错误”操作并不一定代表连接或实现存在问题（例如，当不再需要请求的资源时，端点可以重置流）。
* **错误升级机制**：在特定情况下，端点可以（MAY）选择将“流错误”视为“连接错误”，从而因为单个流上的状况而关闭整个连接。但在做出此决定前，实现需要评估其对其他未决（outstanding）请求的影响。
* **未知错误码处理**：对于在意外上下文中使用的错误码，或接收到的任何未知错误码，端点必须（MUST）将其等同于 `H3_NO_ERROR (0x0100)` 进行处理。
* **保留错误码的使用**：格式为 `0x1f * N + 0x21`（N 为非负整数）的错误码为保留值，用于测试未知错误码的忽略机制。实现应该（SHOULD）在原本计划发送 `H3_NO_ERROR` 时，以一定概率随机选择一个保留空间的错误码发送。
* **通用错误码 (`H3_NO_ERROR`)**：当连接或流需要被关闭，但没有实际的错误需要发出信号时，使用 `H3_NO_ERROR (0x0100)`。
* **常规协议错误 (`H3_GENERAL_PROTOCOL_ERROR`)**：当对端违反协议要求的行为不匹配任何更具体的错误码，或者端点拒绝使用更具体的错误码时，使用 `H3_GENERAL_PROTOCOL_ERROR (0x0101)`。

#### 发送端应遵守的设计
* **主动终止与错误传达**：当发送端决定异常终止（重置）流或立即关闭 HTTP/3 连接时，必须通过底层 QUIC 协议传达本节定义的 HTTP/3 错误码以说明原因。
* **内部错误报告**：当 HTTP 协议栈发生内部错误时，发送端应使用 `H3_INTERNAL_ERROR (0x0102)`。
* **过载保护**：当检测到对端表现出可能产生过载（excessive load）的行为时，发送端可使用 `H3_EXCESSIVE_LOAD (0x0107)` 终止交互。
* **拒绝请求（仅限服务端）**：如果服务端在未执行任何应用层处理的情况下拒绝请求，必须使用 `H3_REQUEST_REJECTED (0x010b)`。
* **取消请求**：当请求或其响应（包括推送的响应）被取消时，发送端应使用 `H3_REQUEST_CANCELLED (0x010c)`。
* **版本回退建议**：如果请求的操作无法通过 HTTP/3 提供服务，且对端应通过 HTTP/1.1 重试，发送端应使用 `H3_VERSION_FALLBACK (0x0110)`。

#### 接收端应遵守的设计
* **流创建异常拦截**：如果接收端检测到对端创建了其不接受的流，必须使用 `H3_STREAM_CREATION_ERROR (0x0103)` 进行处理。
* **关键流关闭检测**：如果 HTTP/3 连接所必需的流（如控制流）被意外关闭或重置，接收端必须触发 `H3_CLOSED_CRITICAL_STREAM (0x0104)`。
* **非法帧与布局异常拦截**：
  * 在当前状态或当前流上收到了不允许出现的帧时，触发 `H3_FRAME_UNEXPECTED (0x0105)`。
  * 收到的帧不符合布局要求或大小无效时，触发 `H3_FRAME_ERROR (0x0106)`。
* **ID 违规处理**：如果流 ID 或 Push ID 使用不当（例如超出限制、减小最大限制或重复使用），接收端必须触发 `H3_ID_ERROR (0x0108)`。
* **SETTINGS 帧异常检测**：
  * 如果在 `SETTINGS` 帧的有效载荷中检测到错误，触发 `H3_SETTINGS_ERROR (0x0109)`。
  * 如果在控制流的开头没有收到 `SETTINGS` 帧，必须触发 `H3_MISSING_SETTINGS (0x010a)`。
* **消息与完整性校验**：
  * 如果 HTTP 消息格式错误（malformed）且无法处理，必须触发 `H3_MESSAGE_ERROR (0x010e)`。
  * 如果客户端的流在没有包含格式完整的请求的情况下提前终止，服务端接收端应触发 `H3_REQUEST_INCOMPLETE (0x010d)`。
* **代理/隧道异常**：响应 `CONNECT` 请求而建立的 TCP 连接如果被重置或异常关闭，接收代理必须将此视为 `H3_CONNECT_ERROR (0x010f)`。


### 9. HTTP/3 的扩展 (Extensions to HTTP/3)

#### 双方均应遵守的设计
* **扩展的作用域与边界**：协议扩展仅在单个 HTTP/3 连接的范围内有效。本机制仅适用于本文档定义的协议元素，不影响现有的 HTTP 扩展方式（如定义新的请求方法、状态码或头部字段）。
* **允许的扩展点**：协议允许使用新的帧类型（frame types）、新设置参数（settings）、新错误码（error codes）或新的单向流类型（unidirectional stream types）。
* **未知元素的忽略机制**：对于所有可扩展的协议元素，实现必须（MUST）忽略未知或不支持的值。这使得扩展点可以安全地被使用，而无需事先安排或协商。
* **语义变更的强制协商**：如果扩展可能会改变现有协议组件的语义，则在使用该扩展之前必须（MUST）进行协商（例如改变 `HEADERS` 帧的布局）。
* **协商设置的默认值**：如果使用设置（setting）来进行扩展协商，该设置的默认值必须（MUST）被定义为：当省略该设置时，该扩展处于禁用状态。

#### 发送端应遵守的设计
* **向前兼容的扩展策略**：发送端可以安全地发送不改变现有语义的新扩展元素，而无需等待对端确认。但若要修改现有协议元素，为了降低同步复杂性，发送端在设计扩展时应优先考虑为现有协议元素的新定义分配新的标识符，而不是直接修改已有帧的布局。

#### 接收端应遵守的设计
* **未知单向流的安全处理**：接收端遇到具有未知或不支持类型的单向流时，必须（MUST）丢弃其数据或中止对该流的读取。
* **严格位置的帧类型校验**：当协议要求在特定位置必须出现已知的帧类型时（例如控制流的首帧必须是 `SETTINGS` 帧），如果接收端在此处收到未知的帧类型，该未知帧不能满足位置要求，接收端应该（SHOULD）将其作为错误进行处理。


### 10. 安全注意事项 (Security Considerations)

#### 双方均应遵守的设计
* **跨协议攻击防护**：依赖 TLS 和 QUIC 握手阶段的 ALPN 来在处理应用层字节之前确立目标协议，以提供强有力的对等端协议一致性保证。
* **非法字段与字符拦截（防中介封装攻击）**：请求或响应中如果包含无效的字段名称，或者包含不允许出现在字段值中的字符（例如回车 0x0d、换行 0x0a 和空字符 0x00），必须（MUST）将其视为畸形（malformed）消息，以防止中间件转换时被恶意利用。
* **严格的压缩隔离（防数据窃取）**：在安全通道上通信时，绝对不能（MUST NOT）将包含机密数据和攻击者可控数据的内容放在同一个压缩上下文中进行压缩。如果无法可靠确定数据来源，绝对不能使用压缩（防范如 BREACH 类的 Web 压缩攻击）。
* **帧解析安全校验**：实现必须（MUST）确保帧的声明长度与它实际包含的字段长度完全匹配，防止因解析嵌套的变长整数而导致的潜在安全风险。
* **0-RTT 防重放攻击**：当使用带有 0-RTT 的 HTTP/3 时，必须（MUST）应用专门的防重放缓解措施。
* **资源监控与 DoS 防御**：实现应该（SHOULD）跟踪服务器推送、字段压缩以及未知协议元素的使用情况，并对它们的使用设置严格的资源边界限制。当发现耗费过多处理时间的异常行为时，端点可以（MAY）将这种可疑活动视为类型为 `H3_EXCESSIVE_LOAD` 的连接错误。

#### 发送端应遵守的设计
* **多租户环境下的推送隔离（服务端）**：在多租户共享的服务器上，服务器必须（MUST）确保各个租户无法推送他们不具有权限（无权威性）的资源。未能强制执行此规则将导致缓存污染攻击。
* **填充与流量分析防御**：发送端可以使用底层传输层填充，或通过发送保留的帧和流类型（以模拟发送恒定数据流）来模糊帧内容的实际大小和流量突发的时间点。但应注意，分布可预测的或固定大小的随机填充方案提供的保护极其有限。
* **忽略字段大小限制**：`SETTINGS_MAX_FIELD_SECTION_SIZE` 仅是建议性的限制，发送端可以（MAY）选择发送超出此限制的字段区块，但需承担该请求或响应被接收端视为畸形而被拒绝的风险。

#### 接收端应遵守的设计
* **拒绝无权威性推送（客户端）**：客户端必须（MUST）拒绝接受由不具权威性的源服务器发出的推送响应。
* **控制推送承诺数量（客户端）**：接受服务器推送的客户端应该（SHOULD）限制其一次性发出的 Push ID 数量，以防止被服务端消耗过多状态资源。
* **超大头部处理**：如果服务端接收到的字段区块过大以至于不愿意处理，可以返回 HTTP 431 (Request Header Fields Too Large) 状态码；如果客户端遇到无法处理的超大响应头部，可以丢弃该响应。
* **CONNECT 并发保护（代理服务端）**：由于 QUIC 流的创建成本远低于 TCP 连接，支持 `CONNECT` 的代理应在接受并发请求数量上更为保守。为应对目标 TCP 连接进入 TIME_WAIT 状态带来的资源消耗，代理可以延迟增加 QUIC 流的并发限制。
* **地址迁移处理（服务端）**：QUIC 客户端的 IP 地址在连接期间可能会改变。依赖客户端 IP 地址进行日志记录或访问控制的服务端实现，必须主动获取当前有效地址，或在架构上接受原始地址可能发生变更的事实。


### 11. IANA 注意事项 (IANA Considerations)

#### 双方均应遵守的设计
* **ALPN 协议标识**：在 TLS 应用层协议协商 (ALPN) 中，必须使用标识序列 `0x68 0x33`（即字符串 `"h3"`）来明确标识 HTTP/3 协议。
* **保留值空间限制**：在帧类型、设置参数、错误码和流类型的注册表中，所有符合格式 `0x1f * N + 0x21`（其中 N 为非负整数，即 0x21, 0x40, ..., 直至 0x3ffffffffffffffe）的值均为保留空间。双方在实现协议或开发自定义扩展时，绝对不能试图赋予这些值任何实际语义，必须保留它们用于测试“未知类型忽略”机制。
* **注册表扩展隔离**：HTTP/3 拥有独立于 HTTP/2 的注册表。在进行协议扩展时，如果新的帧类型、设置参数或错误码与 HTTP/2 有重叠代码空间，应尽量保持含义平行，避免将相同的值分配给不相关的操作。

#### 发送端应遵守的设计
* **遗留帧类型的禁发**：对于在 HTTP/2 中使用但在 HTTP/3 中无对应机制且被 IANA 明确注册为“保留 (Reserved)”的初始帧类型（值为 `0x02`、`0x06`、`0x08`、`0x09`），发送端绝对不能发送。
* **遗留设置参数的禁发**：对于被 HTTP/3 注册表明确标记为“保留 (Reserved)”的初始设置参数（值为 `0x00`、`0x02`、`0x03`、`0x04`、`0x05`），发送端绝对不能在 `SETTINGS` 帧中发送。
* **流类型发起权限合规**：发送端在创建新的单向流时，必须遵守注册表规定的合法发起方（Sender）权限。例如，控制流（Control Stream, `0x00`）的发送方可以是“双端（Both）”，而推送流（Push Stream, `0x01`）的发送方只能是“服务端（Server）”。

#### 接收端应遵守的设计
* **注册表默认值继承**：对于 `SETTINGS` 帧中的参数，接收端应依据注册表的默认值进行初始设定（例如 `SETTINGS_MAX_FIELD_SECTION_SIZE` `0x06` 的默认值为“无限制 (Unlimited)”）。
* **对冲突注册的拒绝**：此章节主要规范 IANA 和专家审查员的行为。作为运行时的扩展验证机制补充，如果接收端收到声明为扩展但具有已知冲突语义的分配，应依赖前述章节定义的协议错误机制（如 `H3_SETTINGS_ERROR` 或 `H3_FRAME_UNEXPECTED`）进行拦截。


### 附录 A. 从 HTTP/2 迁移的注意事项 (Appendix A. Considerations for Transitioning from HTTP/2)

#### 双方均应遵守的设计
* **流生命周期与并发控制**：HTTP/3 的流并发由 QUIC 层管理。与 HTTP/2 在发送 `END_STREAM` 后即认为流关闭不同，QUIC 流只有在所有数据被接收且发送的数据被对端确认后才算完全关闭。因此，等效交互中的流可能会保持更长时间的“活跃”状态。
* **全局流量控制**：在 HTTP/2 中，仅请求和响应主体（`DATA` 帧载荷）受流量控制。而在 HTTP/3 中，所有帧均在 QUIC 流上发送，因此所有帧的头部和有效载荷均受流量控制机制约束。
* **帧布局与标志位简化**：由于底层 QUIC 已处理相关机制，HTTP/3 帧布局中移除了流编号（Stream ID）、`END_STREAM` 标志以及通用的标志（Flags）字段。同时，不再需要支持可变最大长度的包（variable-maximum-length packets）。
* **优先级机制移除**：HTTP/3 移除了 HTTP/2 中的优先级信号机制，不再定义 `PRIORITY` 帧和 `HEADERS` 帧中的优先级区域。
* **状态压缩替换 (QPACK)**：由于 QUIC 不保证跨流的严格顺序交付（HPACK 的核心假设），双方必须使用 QPACK 来压缩字段，且必须使用单向流来按序修改动态表状态，数据流上的帧仅引用该表的状态而不修改它。
* **帧类型的废弃与保留**：HTTP/2 中的 `RST_STREAM`、`PING`、`WINDOW_UPDATE` 和 `CONTINUATION` 帧在 HTTP/3 中被移除（因为 QUIC 提供了等效功能或 HTTP/3 允许更大的帧）。这些在 HTTP/2 中定义的帧 ID 在 HTTP/3 中被标记为保留。
* **SETTINGS 的一次性原则**：`SETTINGS` 帧必须作为控制流的首个帧发送，且仅发送一次，此后绝对不能更改。
* **HTTP/2 SETTINGS 参数的废弃**：在 HTTP/3 `SETTINGS` 帧中，绝对不能发送以下 HTTP/2 保留标识符：`SETTINGS_ENABLE_PUSH` (0x02)、`SETTINGS_MAX_CONCURRENT_STREAMS` (0x03)、`SETTINGS_INITIAL_WINDOW_SIZE` (0x04) 以及 `SETTINGS_MAX_FRAME_SIZE` (0x05)。收到它们将被视为错误。`SETTINGS_MAX_HEADER_LIST_SIZE` (0x06) 被重命名为 `SETTINGS_MAX_FIELD_SECTION_SIZE`。
* **变长整数编码**：流 ID、设置值等参数现在使用 QUIC 的变长整数编码（6、14、30 或 62 位长），而不是 HTTP/2 的固定长度 32 位字段。
* **错误码的不可移植性**：HTTP/2 和 HTTP/3 具有相同概念的流错误和连接错误，但错误码空间不可直接相互移植，必须依赖规范定义的映射表进行转换（如 `NO_ERROR` 映射为 `H3_NO_ERROR` 等）。

#### 发送端应遵守的设计
* **自定义帧标志位的分配**：由于通用的 HTTP/3 帧移除了 `Flags` 字段，如果在设计 HTTP/3 扩展帧时依赖标志位，发送端必须在帧的有效载荷（Frame Payload）部分自行分配空间来存储这些标志。
* **服务器推送的并发控制（服务端）**：由于存在其他单向流类型，服务端不能仅仅依靠单向流的并发数来控制推送。服务端必须严格遵循客户端通过 `MAX_PUSH_ID` 帧设置的上限来控制发出的并发推送。
* **跨版本错误传递（作为中介代理时）**：
  * 中介在遇到上游源服务器错误时，通常应向下游返回通用的 HTTP 状态码（如 502 Bad Gateway）。
  * 在特定情况下为了保留重试信号等，中介可以将 HTTP/2 的错误精准映射并发送给下游 HTTP/3 客户端。例如，将 HTTP/2 的 `REFUSED_STREAM` 映射并发送为 `H3_REQUEST_REJECTED`，或者将下游客户端的 `H3_REQUEST_CANCELLED` 映射为上游的 `CANCEL` 并发送。

#### 接收端应遵守的设计
* **流容量规划（服务端）**：由于 HTTP/3 中 QUIC 流的生命周期更长（等待确认），服务器端预期可能需要允许比 HTTP/2 数量更多的并发客户端双向流，以达到等同于 HTTP/2 的并发吞吐量。
* **异步处理机制**：由于 QUIC 流可能乱序到达，接收端不应该等待对端的 `SETTINGS` 帧到达后才开始响应或处理其他流上的数据。
* **跨版本错误的拦截与处理（作为中介代理时）**：代理中介允许将上游传入的流错误提升为下游的连接错误，但在执行此操作之前，代理必须意识到因为一个临时或间歇性错误而牺牲整个 HTTP/3 连接的代价。
