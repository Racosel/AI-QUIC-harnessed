# 1. Introduction

### 一、 收发两端通用设计原则与规范

* **版本协商的双路径架构支持**：状态机和网络层设计必须支持两种不同的版本协商路径：
    1.  **兼容性协商（0-RTT）**：对于具有相似性的“兼容（compatible）”版本，系统应该利用相似性机制，在不增加额外往返时间（RTT）的情况下完成版本协商。
    2.  **非兼容性协商（+1 RTT）**：对于完全不相交（disjoint）的两个 QUIC 版本，必须能够安全地利用 `Version Negotiation`（版本协商）报文进行协商，并在连接建立过程中允许增加一次额外的 RTT。
* **全局状态变量（Negotiated Version）**：连接的上下文（Context）中必须维护一个 `Negotiated_Version`（协商后版本）状态变量。该变量表示版本协商过程彻底完成后，当前连接最终确定的并正在使用的 QUIC 版本。在协商完成前，该变量处于未决（Pending）状态。
* **全局常量定义（MSL）**：网络层必须定义“最大段生命周期”（Maximum Segment Lifetime, MSL）配置项，代表 QUIC 报文在物理网络中可能存活的最长时间。
    * **代码约束**：建议（RECOMMENDED）将此配置的默认值设定为 60 秒（1 分钟）。
    * **架构约束**：该值应当作为系统的可配置参数暴露，供超时清理、报文去重或生命周期管理模块调用。
* **规范语义的严格映射**：在 Agent 生成代码时，必须将规范词汇映射为代码逻辑级别：
    * `MUST` / `MUST NOT` / `REQUIRED` / `SHALL` / `SHALL NOT`：必须映射为强校验逻辑。若违反，直接抛出异常、丢弃报文或中断连接。
    * `SHOULD` / `SHOULD NOT` / `RECOMMENDED`：必须实现标准逻辑，但允许提供配置开关或降级策略以供忽略该限制。
    * `MAY` / `OPTIONAL`：映射为特性的开关配置或可选执行的分支逻辑。

### 二、 接收端 (Receiver) 独有规范

* *(注：本章主要为概念定义，接收端行为在后续章节会详细展开，基于本章推导如下)*
* **版本不可接受时的响应（版本拦截）**：如果接收端（服务端）判定发送端发起的 QUIC 版本是不可接受的，接收端必须具备构造并回复 `Version Negotiation`（版本协商）报文的能力，以此向发送端明确指出当前版本不被支持，从而触发发送端的重连/协商机制。

### 三、 发送端 (Sender) 独有规范

* **初始飞行数据（First Flight）的状态隔离**：发送端（客户端）的代码逻辑中，必须能够准确标记并识别“First Flight”状态。即在发起连接后、**尚未收到服务端的任何响应之前**，所创建并发送的所有数据包集合，必须被统一归类为“首批飞行数据”，在收到首个有效 ACK 或响应前，涉及拥塞控制和重传逻辑时需按此状态进行特殊处理。
* **维护客户端选择版本（Client's Chosen Version）**：发送端连接实例中必须持久化记录一个 `Client_Chosen_Version` 变量。
    * **赋值规则**：该变量的值必须严格等于上述 `First Flight` 数据包所采用的 QUIC 版本号。
* **维护绝对初始版本（Original Version）**：发送端必须维护一个 `Original_Version` 变量，用于记录发起通信的最初意图。
    * **赋值规则**：该变量等于发送端向接收端发送的**绝对第一个**数据包的 QUIC 版本。
    * **跨连接状态保持（极重要）**：如果版本协商过程由于版本不兼容导致跨越了多个不同的底层 QUIC 连接实例（例如：断开后用新版本发起新连接），`Original_Version` 的值**不得被覆盖**。它必须始终保持等于**第一次**发起 QUIC 连接时的 `Client_Chosen_Version`。后续的安全校验机制将依赖此变量防止降级攻击。


# 2. Version Negotiation Mechanism

### 一、 收发两端通用设计原则与规范

* **双重版本协商模式抽象**：代码架构中需明确区分以下两种机制：
    1.  **兼容性版本协商（Compatible Version Negotiation）**：在一个单一的 QUIC 连接上下文中完成，通过版本兼容性直接转换数据包，节省 1-RTT。
    2.  **不兼容版本协商（Incompatible Version Negotiation）**：概念上跨越两个独立的 QUIC 连接，第一个连接因无法解析而中止并触发 `Version Negotiation` 报文，客户端随后用新版本发起第二个连接，会增加 1-RTT。
* **版本兼容性的非对称性与强约束（MUST NOT）**：
    * 两个 QUIC 版本（A 与 B）是否兼容是不对称的（A 兼容 B 不代表 B 兼容 A）。
    * 在任何显式标准文档未规定兼容性的情况下，系统**绝对不得（MUST NOT）**默认或假设任何两个版本之间存在兼容性。
* **全握手生命周期的规则覆盖（MUST）**：当发生兼容性版本协商并成功将“首批飞行数据（first flight）”转换为协商版本（Negotiated Version）后，该协商版本所要求的所有握手校验规则（例如：QUIC v1 要求的 5-tuple 稳定性），必须**严格适用**于整个握手阶段（包含被转换的这批首飞数据）。

### 二、 接收端 (Receiver) 独有规范

* **首飞数据（First Flight）解析拦截**：接收端在收到首批报文时，必须先验证是否能解析客户端选择的版本（Chosen Version）。如果不能解析，必须进入**不兼容版本协商**流程。
* **发起不兼容版本协商**：
    * 接收端**必须（SHALL）**构造一个 `Version Negotiation` 报文并发送。
    * 该报文的 `Supported Version` 字段**必须（SHALL）**包含接收端实际支持的所有可用版本（Offered Versions）集合。
    * 代码实现中**允许（MAY）**在此字段中加入一些保留（伪造）的版本号（防僵化机制）。
* **处理兼容版本协商**：
    * 如果能解析首飞数据，接收端需从中提取客户端的 `Version Information` 结构。
    * 接收端**必须（MUST）**从该结构声明的客户端兼容版本中，选择一个自己既支持、且确认与客户端 Chosen Version 兼容的版本，作为最终的 `Negotiated Version`。
    * 若找不到任何匹配的兼容版本，接收端回退执行**不兼容版本协商**。
    * 若找到了匹配版本，接收端尝试将收到的首飞数据按照 `Negotiated Version` 的格式进行转换。如果格式本身正确，但接收端代码侧转换逻辑失败，接收端**必须（MUST）**中止握手（abort the handshake）。
* **拒绝无意义的协商报文**：接收端**绝对不得**处理收到的 `Version Negotiation` 报文，如果收到必须直接丢弃（Ignore）。

### 三、 发送端 (Sender) 独有规范

* **不发送版本协商报文（MUST NOT）**：发送端（客户端）代码逻辑中，**绝对不得（MUST NOT）**主动构造和发送 `Version Negotiation` 报文。
* **原始版本（Original Version）的选择偏好**：在发起连接时，发送端**应该（SHOULD）**采用最大化成功率的策略来选择 `Original Version`，以避免不兼容协商带来的 RTT 惩罚。具体策略为：选择一个既能大概率被服务端解析、又与其期望使用的较新版本兼容的版本（例如系统支持的最老但兼容性最好的版本），同时在首飞数据中附带期望的新版本。
* **禁用兼容协商的开关**：发送端可以通过在发送的 `Version Information` 结构的 `Available Versions` 字段中**仅**放置其 `Chosen Version`，来单方面强制禁用兼容版本协商功能。
* **接收 `Version Negotiation` 报文的安全校验与响应**：
    * **丢弃校验**：如果收到的 `Version Negotiation` 报文满足以下任一条件，发送端必须静默忽略（Ignore）它：
        1. 报文的版本列表中包含了发送端最初尝试的 `Original Version`。
        2. 报文头部的 Connection ID (CID) 字段不正确（防止伪造）。
    * **版本匹配（SHALL）**：如果报文合法，发送端必须在服务端提供的版本列表中搜索自己支持的版本。
    * **匹配失败处理（SHALL）**：如果没有找到任何一个共同支持的版本，发送端必须立刻中止连接尝试。
    * **匹配成功处理（SHALL）**：如果找到了共同支持的版本，发送端必须选择其中一个作为新的 `Negotiated Version`，重置连接状态，并使用该版本重新生成并发送全新的“首批飞行数据（first flight）”。


# 3. Version Information

### 一、 收发两端通用设计原则与规范

* **版本信息（Version Information）的数据结构定义**：必须定义 `Version Information` 结构体用于在握手中交换，该结构体包含：
    1.  `Chosen Version` (32-bit)：发送方当前为该连接选择的版本号。
    2.  `Available Versions` (32-bit 数组)：发送方提供的可用版本列表。
* **认证交换机制约束（MUST）**：任何支持此版本协商机制的 QUIC 版本，都**必须（MUST）**在底层协议层面提供一种机制，确保在握手期间双向交换 `Version Information` 数据，并且这些数据必须经过安全认证（Authenticated），防止被中间人篡改。
    * *实现参考：对于 QUIC v1，该结构被封装在 `version_information` 传输参数（Transport Parameter）中进行交换。*
* **防御性编程与防僵化机制支持（MAY）**：代码逻辑在处理（无论是发送还是接收解析） `Available Versions` 列表时，**必须允许并正确处理**符合 `0x?a?a?a?a`（GREASE 模式）的保留版本号。
    * **发送端策略（MAY）**：客户端和服务端**可以（MAY）**在自己的可用版本列表中随机插入此类伪版本号，以测试链路对未知版本的透明度。
    * **处理约束（绝对禁止选择）**：任何一方在遍历或匹配可用版本时，遇到此类保留版本号，**绝对不得**将其选为协商使用的版本，只能安全忽略。

### 二、 接收端 (Receiver / Server) 独有规范

* **服务端 Available Versions 填充规则**：服务端在构造发送给客户端的 `Version Information` 时，`Available Versions` 数组的填充必须遵循以下规则：
    1.  **范围限制**：只列出当前服务端部署的“完全部署版本（Fully Deployed Versions）”。
    2.  **顺序无关性**：列表的顺序不代表服务端的偏好，代码实现无需保证特定顺序。
    3.  **不强制包含 Chosen Version**：由于服务端可能正在下线某个版本的支持，因此该列表**不强制**包含服务端当前正在使用的 `Chosen Version`。
    4.  **允许为空（MAY）**：同理，服务端发送的 `Available Versions` 列表**可以（MAY）**为空。
* **处理客户端偏好的独立性（MAY）**：当服务端接收并解析客户端的 `Available Versions` 列表时，它会获取客户端的版本偏好（按降序排列）。但是，服务端**可以（MAY）**忽略客户端的偏好顺序，完全基于自身的内部策略或偏好来选择最终的兼容版本。

### 三、 发送端 (Sender / Client) 独有规范

* **客户端 Chosen Version 赋值规则**：客户端的 `Chosen Version` 字段通常等于当前承载此数据包的长报头（Long Header）中的 `Version` 字段。*(注：除非未来协议扩展有特殊规定，默认实现应保证两者一致。)*
* **客户端 Available Versions 填充规则**：客户端在构造发送给服务端的 `Version Information` 时，`Available Versions` 数组的填充必须严格遵循以下规则：
    1.  **范围限制**：列出当前“首批飞行数据（first flight）”所兼容的所有版本。
    2.  **偏好排序（强约束）**：列表**必须**按照客户端的偏好程度**降序（Descending Preference）**排列（最期望使用的版本放最前）。
    3.  **必须包含 Chosen Version（MUST）**：为了明确传达当前选择版本的偏好权重，客户端填写的 `Chosen Version` 的值，**必须（MUST）**存在于 `Available Versions` 列表中。如果由于代码 bug 导致未包含，将被视为违反协议规范。


# 4. Version Downgrade Prevention

### 一、 收发两端通用设计原则与规范

* **防降级核心安全目标**：代码逻辑必须能防御降级攻击，即阻止恶意实体通过伪造或篡改报文，迫使两端协商出一个与无攻击环境下不同的（通常是更低级的）QUIC 版本。
* **Version Information 强制解析与结构强校验 (MUST)**：握手期间双方**必须（MUST）**解析对端的 `Version Information`。若触发以下任一条件，**必须（MUST）**判定为解析失败，并立即关闭连接（在 QUIC v1 中，需使用 `TRANSPORT_PARAMETER_ERROR` 类型的传输错误）：
    1. 数据流长度不足，或其字节长度不是 4 的倍数。
    2. 解析出的 `Chosen Version` 字段的值等于 0。
    3. 解析出的 `Available Versions` 列表中包含任何等于 0 的版本号。
* **统一版本协商错误码机制 (MUST)**：任何支持版本协商的 QUIC 版本实现中，底层都**必须（MUST）**定义并暴露出一种带有“版本协商错误”语义的连接关闭方法（例如 QUIC v1 中的 `VERSION_NEGOTIATION_ERROR`，见规范 10.2 节）。
* **连接版本的最终事实认定**：在连接的整个生命周期中，不论长报头（Long Header）的 Version 字段如何改变，当前连接所使用的绝对版本，以服务端在 `Version Information` 中发送的 `Chosen Version` 为唯一事实基准。

### 二、 接收端 (Receiver / Server) 独有规范

* **客户端 Available Versions 逻辑校验 (MUST)**：服务端在解析客户端的 `Version Information` 时，如果发现客户端的 `Chosen Version` 竟然不存在于客户端自己提供的 `Available Versions` 列表中，服务端**必须（MUST）**将其直接视为解析失败（触发连接关闭机制）。
* **真实使用版本与声明版本的防篡改校验 (MUST)**：服务端处理客户端的“首批飞行数据（first flight）”时，会通过检查包头（例如长报头中的 Version 字段 = `0x00000001`）来确定当前连接实际使用的版本。随后处理客户端的 `Version Information` 时，服务端**必须（MUST）**校验客户端声明的 `Chosen Version` 是否与当前实际使用的版本严格匹配。若两者不同，服务端**必须（MUST）**以版本协商错误（`VERSION_NEGOTIATION_ERROR`）关闭连接。
* **容忍 Version Information 缺失 (MAY)**：在某些场景下，如果客户端的 `Version Information` 缺失，服务端侧的代码逻辑**可以（MAY）**允许握手继续完成，而不是强制阻断。

### 三、 发送端 (Sender / Client) 独有规范

* **对 Version Negotiation (VN) 报文的二次拦截 (MUST)**：
    1. **拒绝 Original Version**：如果收到的 VN 报文包含客户端最初尝试的 `Original Version`，客户端**必须（MUST）**忽略该报文。
    2. **拒绝链式协商**：如果客户端**已经**是因为处理了一个 VN 报文而发起了新的连接尝试，那么在这个新连接的响应中，客户端**必须（MUST）**忽略任何再次收到的 VN 报文（防止无限重连死循环或链式降级）。
* **对服务端 Version Information 缺失的条件性阻断**：
    1. **重连状态下缺失 (MUST NOT)**：如果客户端当前是基于收到 VN 报文后发起的重连，此时如果服务端未发送 `Version Information`，客户端**绝对不得（MUST NOT）**完成握手，且**必须（MUST）**以版本协商错误关闭连接。
    2. **正常状态下缺失 (MAY)**：如果客户端不是在响应 VN 报文（即正常首次连接），服务端未发送该信息，客户端**可以（MAY）**继续完成握手。
* **服务端 Chosen Version 越界校验 (MUST)**：客户端在收到服务端的 `Version Information` 后，如果发现服务端的 `Chosen Version` **不在**客户端最初发出的 `Available Versions` 列表（即客户端声明兼容的版本集合）内，客户端**必须（MUST）**以版本协商错误关闭连接。
* **服务端 Chosen Version 假冒校验 (MUST)**：客户端**必须（MUST）**验证服务端发送的 `Chosen Version` 是否等于当前的 `Negotiated Version`（客户端通过兼容性协商从长报头中获知的版本）。若不匹配（说明长报头被攻击者篡改），客户端**必须（MUST）**以版本协商错误关闭连接。
* **基于伪造 VN 报文的降级推演校验 (MUST, 极重要)**：如果客户端是**响应 VN 报文后**发起的连接，必须对服务端的 `Available Versions` 字段执行安全推演校验，以反查之前的 VN 报文是否为伪造：
    1. **空列表防范**：如果服务端的 `Available Versions` 字段为空，客户端**必须（MUST）**以版本协商错误关闭连接。
    2. **重播推演逻辑**：客户端必须运行以下逻辑判断：*“如果我最初收到的 VN 报文里的版本列表，就是【服务端现在的 Available Versions + 当前的 Negotiated Version】，按照我的版本选择偏好算法，我会不会选出完全相同的版本？”*
    3. **推演失败阻断**：如果客户端发现基于服务端真实的 `Available Versions` 会选出另一个不同（通常是更高）的版本，这证明最初触发重连的 VN 报文过滤了高级版本（降级攻击），客户端**必须（MUST）**立即以版本协商错误关闭连接。


# 5. Server Deployments of QUIC

### 一、 收发两端通用设计原则与规范

* **分布式视角的架构假设**：代码实现与安全校验逻辑不能假设对端是单一状态的单点服务器，必须包容服务端为多实例集群（Fleet）且各个实例版本更新存在时间差（不同步）的物理现实。
* **瞬态安全边界认知**：在服务端进行版本上线或下线的过渡窗口期，针对那些处于“可接受”但尚未“完全部署”的版本，系统在机制上承认存在降级攻击的脆弱性（因为客户端无法区分这是合法的集群异步响应还是恶意的降级攻击）。在这个窗口期内，不应引入额外的启发式预测算法，严格依据最终握手时的确切版本集合进行判定即可。

### 二、 接收端 (Receiver / Server) 独有规范

* **服务端版本状态集解耦（核心架构约束）**：服务端的配置管理或内存模型中，不能仅用单一的“支持版本列表”来控制逻辑，**必须**将版本支持状态解耦为三个独立维护的集合变量：
    1.  `Acceptable_Versions`（可接受版本集）：当前服务器实例底层的实际解析能力。只要客户端首批数据使用的是该集合中的版本，服务端就会尝试建立连接。
    2.  `Offered_Versions`（主动提供版本集）：当服务端遇到未知版本并决定触发不兼容协商时，用于构造 `Version Negotiation` 报文内容的数据源。
    3.  `Fully_Deployed_Versions`（完全部署版本集）：代表整个服务端集群所有实例都已共同支持的版本基线。**服务端在构造 `Version Information` 数据结构中的 `Available Versions` 字段时，必须使用此集合的值**（呼应第 3 章规范）。
* **版本配置平滑变更状态机（SHOULD）**：服务端的配置下发与热更新模块，**应该（SHOULD）**实现基于最大段生命周期（MSL）延迟的三步状态机，以防止由于飞行中的旧报文导致客户端触发防降级误判：
    * **添加新版本（Add Version）逻辑链路**：
        1. 步骤一：更新并全量下发至所有实例的 `Acceptable_Versions` 集合。**挂起等待至少 1 MSL**。
        2. 步骤二：更新并全量下发至所有实例的 `Offered_Versions` 集合。**再次挂起等待至少 1 MSL**。
        3. 步骤三：更新并下发至所有实例的 `Fully_Deployed_Versions` 集合。
    * **移除旧版本（Remove Version）逻辑链路（逆向执行）**：
        1. 步骤一：从所有实例的 `Fully_Deployed_Versions` 集合中移除。**挂起等待至少 1 MSL**。
        2. 步骤二：从所有实例的 `Offered_Versions` 集合中移除。**再次挂起等待至少 1 MSL**。
        3. 步骤三：从所有实例的 `Acceptable_Versions` 集合中彻底移除底层的解析支持。

### 三、 发送端 (Sender / Client) 独有规范

* **状态盲区与严格执行**：发送端（客户端）无法且**不得**尝试感知服务端的集群更新状态或网络中残存的旧版本报文。
* **拒绝绕过安全机制**：如果因为服务端运维人员未遵守 MSL 延迟更新规范，导致客户端收到了状态不一致的报文（例如先收到包含新版本的 VN 报文，随后又连上了尚未更新的旧实例），这会触发第 4 章中的降级攻击阻断逻辑。在此场景下，客户端代码**必须**严格执行阻断并抛出版本协商错误，**绝对不得**因为判定其可能是“服务端集群异步导致的假阳性”而尝试静默忽略错误或重连。防降级规则的执行优先级高于连接可用性。


# 6. Application-Layer Protocol Considerations

### 一、 收发两端通用设计原则与规范

* **ALPN 与 QUIC 版本的跨版本复用强约束 (MUST NOT)**：协议栈底层**绝对不得（MUST NOT）**将为某个 QUIC 版本原始定义的 ALPN（应用层协议协商）标识符，直接混用于另一个不同的、新的 QUIC 版本上，除非代码逻辑确认**同时满足**以下全部三个先决条件：
    1.  **特性支持**：该新的 QUIC 版本必须能够提供该应用层协议所要求的所有底层传输特性。
    2.  **ALPN 机制支持**：该新的 QUIC 版本底层本身必须支持 ALPN 协商机制。
    3.  **版本兼容性支持**：该 ALPN 标识符最初定义时所绑定的那个 QUIC 版本，必须与当前的这个新 QUIC 版本属于“兼容（compatible）”关系。

### 二、 接收端 (Receiver) 独有规范

* **ALPN 与协商版本的交叉匹配责任**：当接收端（服务端）解析客户端的“首批飞行数据（first flight）”时，如果客户端同时提供了多个 ALPN 标识符和多个兼容的 QUIC 版本，可能会存在某些 ALPN 无法在某些给定的兼容版本上运行的情况。服务端侧的代码逻辑**必须全权负责**执行交叉校验：服务端最终选定的 ALPN 标识符，**必须**能够安全、合法地运行在其最终选定的那个兼容的 QUIC 版本之上。

### 三、 发送端 (Sender) 独有规范

* **基于应用层意图的兼容版本过滤**：发送端（客户端）建立连接的最终目的是使用应用层协议。因此，客户端在构造 `Version Information` 结构中用于声明兼容性的 `Available Versions` 列表时，必须进行应用层前置过滤：**只能**将那些支持客户端预期使用的应用层协议的 QUIC 版本列入兼容列表，不得盲目提供仅在传输层兼容但无法承载目标应用协议的版本。
* **不兼容协商后的 ALPN 状态彻底重置 (MUST)**：当触发“不兼容版本协商”机制，导致客户端基于 `Version Negotiation` 报文创建第二个全新的连接实例时，客户端**必须（MUST）**在新的连接上下文中彻底重启其应用层协议协商（ALPN）流程。在重新进行 ALPN 决策时，**绝对不得**考虑或继承最初尝试的那个 `Original Version` 所带来的任何上下文或限制。


# 7. Considerations for Future Versions

### 一、 收发两端通用设计原则与规范

* **向后兼容的架构拓展性 (SHOULD)**：在代码架构设计与未来版本的拓展中，**应该（SHOULD）**尽量保持新版本与当前广泛部署的版本（如 QUIC v1）之间的“兼容（compatible）”关系，以便能复用 0-RTT 的兼容性版本协商机制。
* **Retry 机制与版本协商的交互时序约束 (MUST)**：如果协议栈支持两个带有 Retry（地址验证）机制的兼容版本，底层状态机**必须（MUST）**明确处理握手时的时序问题。即：**必须**实现在新的协商版本（Negotiated Version）上下文中，去安全认证（Authenticate）之前使用原始版本（Original Version）发送的 Retry 报文的能力。
* **0-RTT 数据的跨版本转换/丢弃约束 (MUST)**：如果协议栈支持跨两个具备 0-RTT 能力的版本进行兼容性协商，代码逻辑中**必须（MUST）**提供一个明确的钩子（Hook）或策略，以处理首批飞行数据（first flight）中的 0-RTT 报文（例如：指定如何将 0-RTT 数据结构转换为新版本格式，或者直接在框架层定义在协商过程中强制丢弃 0-RTT 并在后续重传）。

### 二、 接收端 (Receiver / Server) 独有规范

* **无状态 Retry 的前置触发允许**：服务端的代码逻辑允许在尚未解析或验证客户端的 `Version Information` 之前，直接基于客户端首飞报文中的版本号回复 Retry 报文（用于防 DDoS 的 IP 验证）。
* **TLS 会话票据 (Session Ticket) 的版本校验 (SHOULD)**：在支持 TLS 1.3 会话恢复（0-RTT 建连）的场景下，服务端侧的认证逻辑**应该（SHOULD）**严格校验客户端提供的 Session Ticket 所绑定的 QUIC 版本。如果发现票据绑定的版本与当前尝试建立的 QUIC 版本不匹配（即防范跨协议攻击），服务端必须拒绝该会话恢复。
* **兼容协商时的 0-RTT 数据拒绝动作**：根据具体兼容版本的规范，服务端网络层需具备在触发“兼容版本协商”流程时，直接拦截并拒绝（Reject）客户端首批报文中的 0-RTT 数据的能力。

### 三、 发送端 (Sender / Client) 独有规范

* **TLS 会话票据 (Session Ticket) 的版本级缓存隔离 (SHOULD)**：客户端在实现 TLS 1.3 会话恢复的缓存管理模块时，**应该（SHOULD）**将会话票据的作用域（Scope）与当前的 QUIC 版本进行强绑定，作为缓存的复合 Key。
* **禁止跨版本滥用票据 (MUST NOT)**：客户端在发起新的 QUIC 连接尝试时，**绝对不得**将从某个特定 QUIC 版本上下文中获取的 Session Ticket，提取并重用于另外一个不同版本的 QUIC 连接尝试中。


# 8. Special Handling for QUIC Version 1

### 一、 收发两端通用设计原则与规范

* **向后兼容的历史包袱处理**：由于 QUIC version 1 是在本规范之前唯一发布的 IETF 标准轨道版本，因此协议栈在处理 v1 版本时存在特例。代码架构必须能够识别并安全处理这种由于历史遗留导致的安全参数缺失情况。
* **新版本协商机制的强制性（MUST）**：任何期望利用版本协商机制来协商**除 QUIC version 1 之外**的其他版本的协议栈实现，**必须（MUST）**完整实现本文档定义的所有版本协商与防降级机制，不能依赖对 v1 的特例处理。

### 二、 接收端 (Receiver) 独有规范

* **单版本 (v1-only) 服务的免责**：如果接收端（服务端）是一个仅支持 QUIC version 1 的遗留实现，它可能不会在其传输参数（transport parameters）中携带本规范要求的 `version_information` 参数。现代代码框架在与此类遗留服务端交互时，必须允许这种缺失情况的发生。

### 三、 发送端 (Sender) 独有规范

* **响应 VN 报文时的 QUIC v1 特例回退机制（SHALL）**：当发送端（客户端）是因为收到了一个 `Version Negotiation` (VN) 报文，从而作为响应去发起一个 **QUIC version 1** 的新连接尝试时，必须执行特殊的状态机逻辑。
* **虚拟状态注入（SHALL 强约束）**：在上述场景下，如果客户端在握手过程中发现服务端返回的传输参数中**缺失**了 `version_information` 参数，客户端代码**必须（SHALL）**拦截常规的“参数缺失导致解析失败/降级阻断”逻辑，并执行状态注入：
    * 客户端必须假装（proceed as if）服务端确实发送了 `version_information` 传输参数。
    * 客户端必须将虚拟的 `Chosen Version` 强行赋值为 `0x00000001` (即 QUIC v1)。
    * 客户端必须将虚拟的 `Available Versions` 列表强行初始化为仅包含一个元素：`[0x00000001]`。
    * 完成虚拟状态注入后，继续执行后续的防降级推演校验（参照第 4 章逻辑，由于只有 v1，校验将顺利通过）。此规则确保了客户端能够正常回退并连接到仅支持 v1 且尚未实现新版本协商机制的旧版服务端。


# 9. Security Considerations

### 一、 收发两端通用设计原则与规范

* **认证依赖性机制 (MUST)**：版本协商机制的安全根基完全依赖于握手期间双向交换的 `Version Information` 的真实性。底层网络栈的实现**必须（MUST）**确保承载该信息的底层通道或数据结构（例如 QUIC v1 中的传输参数 transport parameters）在密码学上是经过安全认证（Authenticated）的，以防止中间人篡改。
* **安全水位的“木桶效应”原则**：在执行“兼容性版本协商”时，当前连接的最终安全强度将被拉低至参与协商的兼容版本中**安全性最弱**的那个版本（the weakest common version）的水平。架构与策略评估时必须将此安全损耗计算在内。
* **防御跨协议攻击的底线 (MUST NOT)**：协议栈**绝对不得**在没有明确规范的情况下默认或隐式假设版本之间存在兼容性。严格遵循“默认不兼容”原则，是缓解和防御跨协议攻击（Cross-protocol attacks）的核心基石，任何代码绕过都将引入致命的安全漏洞。

### 二、 接收端 (Receiver) 独有规范

* **强制密码学校验**：接收端（服务端）在提取并解析客户端的 `Version Information` 以进行防降级比对之前，必须确认承载该信息的握手数据包已经通过了底层的完整性和真实性校验。如果底层握手认证失败，必须立即中止连接，绝不可信任或应用未经验证的版本信息。

### 三、 发送端 (Sender) 独有规范

* **兼容列表的安全降维感知**：发送端（客户端）在构造首飞数据（first flight）中的 `Available Versions` 列表时，如果为了提高建连成功率而主动列入了一些古老且安全机制较弱的兼容版本，客户端代码的决策逻辑应当“知晓”此举可能赋予攻击者将连接安全水平降维至该弱版本的能力。


# 10. IANA Considerations

### 一、 收发两端通用设计原则与规范

* **全局协议常量硬编码 (MUST)**：代码库底层的协议字典或枚举类中，**必须（MUST）**准确无误地注册并硬编码以下两个由 IANA 永久分配的 QUIC 核心常量，供收发两端的状态机统一调用：
    1.  **版本信息传输参数 (Transport Parameter)**：将参数名 `version_information` 的字面值映射为 `0x11`。
    2.  **版本协商错误码 (Transport Error Code)**：将错误码 `VERSION_NEGOTIATION_ERROR` 的字面值映射为 `0x11`（附带语义："Error negotiating version"）。
* **常量命名规范约束**：Agent 在生成协议栈基础代码时，应严格保持上述规范的命名约定（例如在 C/C++ 中使用 `#define QUIC_TP_VERSION_INFORMATION 0x11` 和 `#define QUIC_ERR_VERSION_NEGOTIATION 0x11`），确保后续涉及报文解析和错误处理的逻辑全部引用此宏定义，而非魔法数字（Magic Numbers）。

### 二、 接收端 (Receiver) 独有规范

* **传输参数解码路由约束**：接收端（服务端）的报文解析器在遍历对端的 Transport Parameters 拓展块时，一旦遇到标识符为 `0x11` 的参数类型（Type），**必须**立即将其 Payload 路由至“Version Information”数据结构的专用反序列化逻辑中。
* **致命错误帧封装**：当接收端的防降级策略（如第 4 章定义）被触发，需要强制断开连接时，底层框架**必须**将 `0x11` 填入 `CONNECTION_CLOSE` 帧的错误码字段中发送给对端。

### 三、 发送端 (Sender) 独有规范

* **首飞数据 (First Flight) 序列化写入**：发送端（客户端）在构造并发出首批飞行数据（特别是 QUIC v1 的握手包）时，**必须**使用 `0x11` 作为 Type 字段，将其构建的 `Version Information` 结构体合法地序列化到传输参数字节流中。
* **异常阻断状态抛出**：当发送端在比对服务端版本信息后，推演发现存在降级攻击或收到非法的 Version Negotiation 报文时，发送端的状态机模块**必须**向应用层或底层连接管理器抛出值为 `0x11` 的错误信号，并终止后续发包机制。
