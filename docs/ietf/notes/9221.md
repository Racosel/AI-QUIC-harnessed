### 1. 引言与规范要求

#### 双方均应遵守的设计
* **基础协议理解**：必须将 QUIC 传输协议（[RFC9000]）视作一个可提供安全、多路复用连接的底层协议。
* **可靠数据传输**：可靠的应用程序数据流必须使用 STREAM 帧进行传输。
* **不可靠数据传输**：对于不需要重传的不可靠应用程序数据（如实时数据），必须使用本规范定义的两种全新的 DATAGRAM QUIC 帧类型进行传输。
* **安全上下文共享**：DATAGRAM 帧的传输必须与可靠数据流共享相同的加密和身份验证上下文，替代传统的 UDP [RFC0768] + DTLS [RFC6347] 方案。
* **规范术语解释**：双方在解析协议规则时，对“MUST”、“MUST NOT”、“REQUIRED”、“SHALL”、“SHALL NOT”、“SHOULD”、“SHOULD NOT”、“RECOMMENDED”、“NOT RECOMMENDED”、“MAY”和“OPTIONAL”等全大写关键字的理解，必须严格遵守 BCP 14 [RFC2119] [RFC8174] 的定义。

#### 发送端应遵守的设计
* 发送端在发送不需要重传的应用程序数据时，应将其封装在 DATAGRAM 帧中进行发送。
* 发送端不得为 DATAGRAM 帧中的数据提供任何重传机制。

#### 接收端应遵守的设计
* 接收端在收到 DATAGRAM 帧时，必须按照不可靠数据的语义进行处理。
* 接收端不得因 DATAGRAM 帧的丢失而向发送端请求重传该帧内的数据。


### 2. 动机

#### 双方均应遵守的设计
* **统一握手与安全上下文**：系统应支持在同一对等端之间的可靠 QUIC 流和不可靠 QUIC 数据报流之间，共享单一的握手和身份验证上下文。这取代了过去需要同时建立 TLS 和 DTLS 连接的方案，从而有效降低握手延迟。
* **高效的丢包恢复机制**：系统应利用 QUIC 更细致的丢包恢复机制（相较于 DTLS 握手），以确保 QUIC 数据能够实现更快速的丢包恢复。
* **统一的拥塞控制**：可靠数据和不可靠数据（QUIC 数据报）均必须受制于单一的 QUIC 拥塞控制。提供统一的拥塞控制可以使网络传输更加高效。
* **应用场景适配**：整体设计应能为音视频流媒体、游戏以及其他实时网络应用的优化提供底层支持；同时支持实现基于 QUIC 的 IP 数据包隧道（例如 VPN）。

#### 发送端应遵守的设计
* 在需要同时发送可靠流和不可靠流时，发送端应在同一个 QUIC 连接中进行复用，以减少建立多重连接的开销。
* 当用于实现互联网层隧道协议（如 VPN）时，发送端应使用可靠的 QUIC 流来发送控制数据，并使用不可靠的 QUIC 数据报来进行 IP 数据包的安全传输。
* 发送端在发送不可靠数据报时，必须将其纳入 QUIC 发送端的拥塞控制计算中。

#### 接收端应遵守的设计
* 接收端必须能够基于单一的认证上下文，同时接收并处理来自发送端的可靠流数据和不可靠数据报。
* 接收端应配合 QUIC 的丢包恢复机制，更快速地处理数据报相关的确认与恢复逻辑。
* 在隧道（如 VPN）场景下，接收端需具备从可靠流中提取控制数据，并从不可靠数据报中提取 IP 数据包进行组合处理的能力。


### 3. 传输参数

#### 双方均应遵守的设计
* **参数定义与功能**：必须通过 QUIC 传输参数 `max_datagram_frame_size`（标识为 max_datagram_frame_size，值为 0x20）来声明对接收 DATAGRAM 帧的支持。该参数是一个可变长度的整数，表示端点愿意接收的 DATAGRAM 帧的最大大小（以字节为单位，包含帧类型、长度以及有效载荷）。
* **默认状态解析**：该参数的默认值为 0，明确表示端点不支持 DATAGRAM 帧。任何大于 0 的值均表示端点支持 DATAGRAM 帧类型，并愿意在当前连接中接收此类帧。
* **单向限制属性**：`max_datagram_frame_size` 传输参数具有单向性，仅代表单向的支持指示与限制。使用 DATAGRAM 帧的应用协议可以（MAY）选择仅在单方向上进行协商和使用。
* **缺失处理机制**：使用数据报的应用协议必须（MUST）明确规定在缺少 `max_datagram_frame_size` 传输参数时的响应机制。如果数据报支持对于该应用协议不可或缺，应用程序可以因该参数未呈现而宣告握手失败。

#### 发送端应遵守的设计
* **发送前置条件**：在握手期间（若使用 0-RTT，则为前一次握手期间）收到对端发送的非 0 值的 `max_datagram_frame_size` 传输参数之前，发送端绝对不得（MUST NOT）发送 DATAGRAM 帧。
* **最大载荷限制**：发送端发送的 DATAGRAM 帧的尺寸，绝对不得（MUST NOT）超过其从对端接收到的 `max_datagram_frame_size` 的值。
* **0-RTT 客户端逻辑**：当客户端使用 0-RTT 时，可以（MAY）存储服务器的 `max_datagram_frame_size` 传输参数值，以允许在 0-RTT 数据包中发送 DATAGRAM 帧。如果客户端将其与 0-RTT 状态一并存储，客户端必须（MUST）验证服务器在后续握手中发送的新 `max_datagram_frame_size` 值是否大于或等于该存储值；如果验证失败（新值小于存储值），客户端必须（MUST）以 `PROTOCOL_VIOLATION` 的错误终止连接。
* **0-RTT 服务端逻辑**：当服务器决定接收 0-RTT 数据时，必须（MUST）确保其发出的 `max_datagram_frame_size` 传输参数值，大于或等于其在向客户端发送 `NewSessionTicket` 消息的连接中所发送的值。

#### 接收端应遵守的设计
* **未授权接收违规处理**：如果接收端未曾通过传输参数指示支持 DATAGRAM 帧，却收到了 DATAGRAM 帧，接收端必须（MUST）以 `PROTOCOL_VIOLATION` 的错误类型终止该连接。
* **超限接收违规处理**：如果接收端收到了尺寸大于其在 `max_datagram_frame_size` 传输参数中声明的大小的 DATAGRAM 帧，接收端必须（MUST）以 `PROTOCOL_VIOLATION` 的错误类型终止该连接。
* **参数推荐值**：对于 DATAGRAM 帧的大多数使用场景，推荐（RECOMMENDED）接收端在 `max_datagram_frame_size` 传输参数中发送 65535，以此指示该端点愿意接受任何能够容纳在单个 QUIC 数据包内的 DATAGRAM 帧。


### 4. 数据报帧类型

#### 双方均应遵守的设计
* **基本用途**：DATAGRAM 帧必须用于以不可靠的方式传输应用程序数据。
* **帧类型标识**：DATAGRAM 帧的 Type（类型）字段格式必须为 `0b0011000X`（即十六进制值 `0x30` 和 `0x31`）。
* **LEN 标志位控制**：Type 字段的最低有效位是 LEN 位（`0x01`），用于指示是否存在 Length（长度）字段。
* **帧结构定义**：DATAGRAM 帧的标准结构依次包含：Type 字段（0x30 或 0x31）、可选的 Length 字段、以及 Datagram Data（数据报数据）字段。
* **长度字段规范**：Length 是一个可变长度的整数，用于指定 Datagram Data 字段的字节长度。
* **空数据报规则**：协议允许空数据报（即零长度的数据报）的存在。
* **数据载荷定义**：Datagram Data 字段包含实际需要交付的数据报字节。

#### 发送端应遵守的设计
* **显式长度声明**：如果发送端将 LEN 位设置为 1，则必须在 DATAGRAM 帧中包含 Length 字段，并准确指定数据的长度。
* **隐式长度声明（包尾对齐）**：如果发送端将 LEN 位设置为 0，则不得在帧中包含 Length 字段，且必须让 Datagram Data 字段的数据一直延伸到当前 QUIC 数据包的末尾。
* **发送空数据报**：发送端可以构造并发送不包含任何 Datagram Data 字节（长度为 0）的 DATAGRAM 帧。

#### 接收端应遵守的设计
* **类型与长度解析**：接收端在处理 DATAGRAM 帧时，必须首先检查 Type 字段的最低有效位（LEN 位）。
* **按字段截取数据**：如果接收端解析到 LEN 位为 1，必须读取紧随其后的 Length 字段，并严格按照该字段指定的字节数来提取 Datagram Data。
* **按包尾截取数据**：如果接收端解析到 LEN 位为 0，则必须跳过对 Length 字段的查找，直接将当前位置直到 QUIC 数据包末尾的所有剩余内容作为 Datagram Data 提取。
* **支持接收空数据报**：接收端必须能够正确处理提取到的零长度 Datagram Data，不得因此抛出格式错误。


### 5. 行为与用法

#### 双方均应遵守的设计
* **安全保护**：与 STREAM 帧一样，DATAGRAM 帧包含应用程序数据，双方必须（MUST）使用 0-RTT 或 1-RTT 密钥对其进行保护。
* **分片与尺寸限制**：DATAGRAM 帧不能被分片（cannot be fragmented）。DATAGRAM 帧的最大尺寸不仅受 `max_datagram_frame_size` 传输参数限制，还会被 `max_udp_payload_size` 传输参数和端点间路径的最大传输单元（MTU）进一步缩小。应用协议需要能够处理这种受其他因素限制最大数据报大小的情况。
* **连接级归属与多路复用**：DATAGRAM 帧作为一个整体属于 QUIC 连接，在 QUIC 层不与任何特定的流 ID 相关联。
* **标识符与解析定义**：运行在 QUIC 之上的应用协议负责定义标识符，用于多路复用不同类型或逻辑流的 DATAGRAM 帧。应用协议负责定义 Datagram Data 字段的语义及其解析方式。
* **推荐的复用模式**：如果应用需要支持多个数据报流共存，推荐（recommended）在 Datagram Data 字段开头使用一个可变长度的整数作为标识，以最小的空间开销支持大量流的编码。
* **优先级 API**：QUIC 的实现应该（SHOULD）向应用程序提供 API，以便其分配 DATAGRAM 帧之间以及 DATAGRAM 帧与 QUIC 流之间的相对优先级。
* **ACK 触发属性**：尽管 DATAGRAM 帧在检测到丢失时不会被重传，但它们属于触发确认的帧（ack-eliciting）。
* **无流控制**：DATAGRAM 帧不提供任何显式的流控制信号，且不计入任何单流或连接级别的流量限制。
* **受制于拥塞控制**：DATAGRAM 帧采用所在 QUIC 连接的拥塞控制器。

#### 发送端应遵守的设计
* **生成与发送时机**：当应用程序发送数据报时，QUIC 会生成一个新的 DATAGRAM 帧并在第一个可用的数据包中发送。该帧应该（SHOULD）尽快发送（受下文拥塞控制等因素影响），并且可以（MAY）与其他帧合并（coalesced）发送。
* **丢包探测**：同任何触发确认的帧一样，当发送端怀疑一个仅包含 DATAGRAM 帧的数据包丢失时，会发送探测数据包以触发更快的确认。
* **丢失通知（应用层）**：如果发送端检测到包含特定 DATAGRAM 帧的数据包可能已丢失，其实现可以（MAY）向应用程序发出该数据报被认为已丢失的通知。
* **确认通知（应用层）**：如果包含 DATAGRAM 帧的数据包被确认，其实现可以（MAY）通知应用程序该数据报已成功发送和接收（由于乱序，这可能包含曾被认为丢失但随后被接收确认的帧）。注意：此信号仅代表接收端的传输层处理了该帧，不能保证接收端应用层成功处理了数据，因此不能取代应用层自身的成功处理信号。
* **拥塞控制限制机制**：如果拥塞控制器不允许发送，发送端必须（MUST）延迟发送该 DATAGRAM 帧直至允许，或者在不发送的情况下直接丢弃该帧（此时可以 [MAY] 通知应用程序）。使用数据包起搏（packet pacing）的实现也可以延迟发送以维持一致的起搏节奏。
* **发送过期时间（可选）**：发送端实现可以选择性支持应用程序指定一个“发送过期时间”，超过该时间后，受拥塞控制限制的 DATAGRAM 帧应被丢弃而不发送。

#### 接收端应遵守的设计
* **即时交付**：当接收端收到有效的 DATAGRAM 帧时，只要能够处理该帧并且有内存存储其内容，就应该（SHOULD）立即将数据交付给应用程序。
* **延迟 ACK 机制**：对于仅包含 DATAGRAM 帧的数据包，接收端应该（SHOULD）支持延迟发送 ACK 帧（在 `max_ack_delay` 规定的限制内），因为暂时未确认这些数据包发送端也不会采取任何重传行动。
* **ACK 恢复发送条件**：当条件表明数据包可能丢失时（因为接收端不知道数据包的具体载荷），或者受 `max_ack_delay` 及其他协议组件的规则要求时，接收端仍将继续发送 ACK 帧。
* **因资源不足的合法丢弃**：由于没有流控制机制，接收端可能无法分配足够的资源（如内存）来处理 DATAGRAM 帧。由于 DATAGRAM 帧本身是不可靠的，在无法处理的情况下，接收端可以（MAY）直接丢弃它们。


### 6. 安全注意事项

#### 双方均应遵守的设计
* **共享安全属性**：DATAGRAM 帧与在 QUIC 连接中传输的其他数据共享相同的安全属性，必须相应地适用 [RFC9000] 中的安全注意事项。
* **强制密钥保护**：与 STREAM 帧类似，所有使用 DATAGRAM 帧传输的应用程序数据必须（MUST）受 0-RTT 或 1-RTT 密钥的加密保护。
* **0-RTT 使用规范**：允许在 0-RTT 数据包中发送 DATAGRAM 帧的应用协议，必须提供一个定义 0-RTT 可接受用途的配置文件或规范（参考 [RFC9001] 第 5.6 节）。
* **流量特征与隐私风险感知**：双方应当知晓，具备丢包能力的路径内网段攻击者（adversary on path）可能会探测到 DATAGRAM 帧的使用。由于 DATAGRAM 帧不提供传输层面的重传，使用 DATAGRAM 帧的连接会由于对丢包的响应不同，从而可能被攻击者从其他普通连接中区分出来。

#### 发送端应遵守的设计
* 发送端在构建并发送包含应用程序数据的 DATAGRAM 帧时，必须确保这些数据已被 0-RTT 或 1-RTT 密钥进行加密。
* 如果协议允许在 0-RTT 中发送，发送端必须严格按照应用协议关于 0-RTT 安全用途的配置文件执行，防止重放攻击等安全风险。

#### 接收端应遵守的设计
* 接收端必须确保接收到的 DATAGRAM 帧受 0-RTT 或 1-RTT 密钥的安全保护，对于未能正确解密或未受保护的数据报必须予以拒绝。


### 7. IANA 注意事项

#### 双方均应遵守的设计
* **官方注册值遵循**：双方的系统实现必须承认并遵守互联网号码分配局（IANA）正式注册的 QUIC 参数和帧类型（规范：RFC 9221）。
* **传输参数注册**：必须将值 `0x20` 识别为 `max_datagram_frame_size` 传输参数（状态：永久）。
* **帧类型注册**：必须将值 `0x30` 和 `0x31` 识别为 `DATAGRAM` 帧类型（状态：永久）。

#### 发送端应遵守的设计
* 发送端在握手阶段声明传输参数时，必须使用 IANA 注册的标准值 `0x20` 来标识 `max_datagram_frame_size`。
* 发送端在构建并发送 DATAGRAM 帧时，必须使用 IANA 注册的标准值 `0x30` 或 `0x31` 作为帧类型（Type）字段。

#### 接收端应遵守的设计
* 接收端在解析传入的 QUIC 传输参数时，必须能够将参数标识符 `0x20` 正确映射并解析为 `max_datagram_frame_size`。
* 接收端在解析传入的 QUIC 数据包时，必须将读取到的帧类型值 `0x30` 或 `0x31` 正确识别为 DATAGRAM 帧，并路由给相应的处理逻辑。
