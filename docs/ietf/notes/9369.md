### 1. 简介

#### 双方均应遵守的设计
* **防降级保护**：任何支持两个版本（如 QUIC v1 和 v2）的端点（Endpoint）必须实现版本协商机制，以防止降级攻击。
* **版本协商支持**：
    * **不兼容协商（Incompatible Version Negotiation）**：支持在任意 QUIC 版本间切换，具备完全通用性，但连接开始时需增加一个往返时间（RTT）。
    * **兼容协商（Compatible Version Negotiation）**：消除往返时间惩罚，但对版本间的语义差异程度有一定限制。
* **协议设计目标认知**：需明确 QUIC v2 的核心目的是缓解网络中间设备僵化（Ossification）并验证版本协商机制。

#### 发送端应遵守的设计
* **传输线版本号随机化**：在长报头（Long Header）中，第 2 至第 5 字节的传输线版本号应选择随机值而非固定数字“2”，以对抗中间设备对特定版本号（如 v1 的 0x00000001）的僵化。
* **报头类型比特位修改**：长报头中用于标识数据包类型的两个比特位必须设置为与 QUIC v1 不同的值。
* **初始包保护机制**：不得使用 v1 特定的盐值（Salt）和密钥派生方式对初始包（Initial packets）进行加密，防止中间设备对 TLS Client Hello 或 Server Hello 等明文信息的解析形成格式僵化。

#### 接收端应遵守的设计
* **动态版本号处理**：必须能够解析并处理传输线上随机出现的版本号，不得依赖固定的版本数值。
* **新型报头标识识别**：必须能够识别并解析与 QUIC v1 不同的长报头类型标识位。
* **解密逻辑适配**：不得依赖 v1 的密钥派生和数据包格式来处理初始包，需适配为了防僵化而修改后的新数据包格式。

### 3. 与 QUIC 版本 1 的差异

#### 双方均应遵守的设计
* **规范继承**：除本章列出的具体差异外，端点必须（MUST）完全实现 [QUIC]、[QUIC-TLS] 和 [QUIC-RECOVERY] 中描述的 QUIC 版本 1 规范。
* **版本字段（Version Field）**：长报头中的版本字段必须为 `0x6b3343cf`（该值取自字符串 "QUICv2 version number" 的 SHA-256 哈希值的前 4 个字节）。
* **初始盐值（Initial Salt）**：用于派生初始密钥（Initial keys）的盐值更改为 `0x0dede3def700a6db819381be6e269dcbf9bd2ed9`（该值取自字符串 "QUICv2 salt" 的 SHA-256 哈希值的前 20 个字节）。
* **HKDF 标签（HKDF Labels）**：用于派生数据包保护密钥、报头保护密钥、重试完整性标签（Retry Integrity Tag）密钥以及密钥更新的标签发生变更：
    * 从 "quic key" 更改为 "quicv2 key"
    * 从 "quic iv" 更改为 "quicv2 iv"
    * 从 "quic hp" 更改为 "quicv2 hp"
    * 从 "quic ku" 更改为 "quicv2 ku"
* **重试完整性标签（Retry Integrity Tag）参数**：用于重试完整性标签的密钥和随机数参数发生变更：
    * Secret = `0xc4dd2484d681aefa4ff4d69c2c20299984a765a5d3c31982f38fc74162155e9f`（字符串 "QUICv2 retry secret" 的 SHA-256 哈希值）
    * Key = `0x8fb4b01b56ac48e260fbcbcead7ccc92`（使用标签 "quicv2 key" 从上述 Secret 派生）
    * Nonce = `0xd86969bc2d7c6d9990efb04a`（使用标签 "quicv2 iv" 从上述 Secret 派生）

#### 发送端应遵守的设计
* **长报头数据包类型标识（Long Header Packet Types）**：在构建并发送版本 2 的长报头数据包时，必须使用全新的类型字段（Type field）比特值来标识数据包类型：
    * Initial 数据包：`0b01`
    * 0-RTT 数据包：`0b10`
    * Handshake 数据包：`0b11`
    * Retry 数据包：`0b00`
* **应用新的密码学参数**：发送端在生成 Initial 数据包、执行数据包/报头保护以及生成重试完整性标签时，必须使用本章规定的新版本字段、新初始盐值、新 HKDF 标签以及新重试密钥参数进行加密和封装。

#### 接收端应遵守的设计
* **解析长报头数据包类型**：在接收长报头数据包时，必须按照新的类型字段比特值映射关系（Initial=0b01, 0-RTT=0b10, Handshake=0b11, Retry=0b00）来正确分类和解析数据包，不可沿用 v1 版本的类型标识。
* **使用新的密码学参数解密**：接收端在派生 Initial 密钥以解密数据包、派生密钥以解除报头保护，以及验证重试完整性标签时，必须使用本章规定的新初始盐值、新 HKDF 标签以及新重试密钥参数进行解密与校验。


### 4. 版本协商注意事项 (Version Negotiation Considerations)

#### 双方均应遵守的设计
* **多版本支持与兼容**：QUIC v2 不旨在废弃 v1。支持 v2 的端点可以继续支持 v1 以最大化兼容性。v1 和 v2 互为兼容版本，支持这两种版本的端点应该（SHOULD）支持兼容版本协商（Compatible Version Negotiation），以避免增加额外的往返时间（RTT）开销。
* **防降级攻击**：任何支持 QUIC v2 的端点必须（MUST）发送、处理和验证 `version_information` 传输参数（详见 [QUIC-VN]），以防范版本降级攻击。
* **特定数据包的版本强制性**：两个端点都必须（MUST）使用**协商后的版本**发送 Handshake（握手）和 1-RTT 数据包。端点必须（MUST）丢弃使用任何其他版本的数据包，且无需生成用于解密或验证此类数据包的密钥材料。

#### 发送端（请求发起方/客户端）应遵守的设计
* **处理 Retry（重试）数据包**：必须忽略使用非原始版本（original version）的 Retry 数据包。
* **携带 Retry 令牌的 Initial 数据包**：在后续包含 Retry 令牌的 Initial（初始）数据包中，绝不能（MUST NOT）使用与原始版本不同的版本号。
* **确认协商版本**：通过观察接收到的第一个与原始版本不同的长报头 Version 字段来学习协商后的版本。如果收到服务端使用原始版本发送的 CRYPTO 帧，则表明最终协商版本等同于原始版本。
* **发送后续 Initial 数据包**：一旦获知协商后的版本，应该（SHOULD）使用该协商版本发送后续的 Initial 数据包。
* **发送 0-RTT 数据包**：绝不能（MUST NOT）使用协商后的版本发送 0-RTT 数据包（即使在处理了服务端该版本的数据包之后）。0-RTT 数据包必须仅使用原始版本发送。

#### 接收端（请求响应方/服务端）应遵守的设计
* **服务端多版本宣告**：HTTP 服务端应该（SHOULD）支持多个 QUIC 版本（例如，通过 Alt-Svc 宣告 "h3" 支持时，应支持原生的 QUIC v1，因为部分客户端仅支持 v1），以降低不兼容概率及版本协商或回退到 TCP 的成本。
* **发送 Retry 数据包**：如果决定发送 Retry 数据包，必须（MUST）使用原始版本。可以（MAY）在其 Retry 令牌中编码 QUIC 版本，以验证客户端是否违规切换了版本；若发现切换，则丢弃该数据包以强制客户端合规。
* **Retry 后的传输参数验证**：QUIC v2 使用与 v1 相同的传输参数来验证 Retry。在 Retry 后切换到协商版本时，服务端必须（MUST）包含相关的传输参数，以验证服务端发送的 Retry 和交互中使用的连接 ID。
* **发送 CRYPTO 帧的限制**：在处理完客户端的传输参数之前，不能发送 CRYPTO 帧。所有的 CRYPTO 帧必须（MUST）使用协商后的版本发送。
* **响应初始 Initial 数据包**：在能够处理客户端的传输参数之前，对于需要响应客户端的 Initial 数据包，必须使用原始版本。
* **保留初始接收密钥**：在成功处理使用协商版本发送的 Handshake 数据包之前，绝不能（MUST NOT）丢弃其原始版本的 Initial 接收密钥。
* **处理 0-RTT 数据包**：可以接受并处理使用原始版本发送的 0-RTT 数据包。


### 5. TLS 恢复与 NEW_TOKEN 令牌

#### 双方均应遵守的设计
* **凭证的版本绑定特性**：TLS 会话票据（Session Tickets）和 `NEW_TOKEN` 令牌均与提供它们的特定 QUIC 连接版本严格绑定。
* **兼容协商后的凭证归属**：当连接中包含兼容版本协商（Compatible Version Negotiation）时，服务端发出的任何令牌以及任何由此产生的 TLS 会话票据，均被视为源自**协商后的版本**，而不是发起连接的原始版本。

#### 发送端应遵守的设计
* **禁止跨版本使用凭证**：客户端绝不能（MUST NOT）使用来自 QUIC 版本 1 连接的会话票据或令牌来发起 QUIC 版本 2 的连接，反之亦然。

#### 接收端应遵守的设计
* **强制版本验证**：服务端必须（MUST）验证任何接收到的会话票据或令牌的来源 QUIC 版本，并且不得接受由其他不同版本签发的凭证。
* **票据拒绝的降级处理**：如果服务端拒绝了 TLS 会话票据，必须回退到完整的 TLS 握手流程，且不可使用 0-RTT（零往返时间）特性。
* **令牌拒绝的限制处理**：如果服务端拒绝了 `NEW_TOKEN` 令牌，将导致客户端地址保持“未验证”状态，服务端必须以此限制可向该客户端发送的数据量。


### 6. 僵化注意事项

#### 双方均应遵守的设计
* **对抗僵化机制与局限认知**：需明确 QUIC v2 提供了针对某些形式的网络僵化（Ossification）的保护机制。如果中间设备假设所有长报头都编码为版本 1，或假设版本 1 的 Initial 密钥派生公式永远不变，它们将无法正确处理版本 2 的数据包。
* **首包检查的局限性**：需认识到，由于前述章节中的兼容性考虑，许多中间设备（如防火墙）主要关注连接的第一个数据包，而该首包往往仍会保持版本 1 的格式。

#### 发送端应遵守的设计
* **主动发起 v2 连接的策略**：致力于对抗中间设备僵化的客户端，可以直接使用版本 2 发起连接，但必须满足两个前提：一是合理确信服务端支持版本 2；二是愿意承担因预判错误而产生的往返时间（RTT）增加的惩罚。
* **服务端支持意图的判断依据**：客户端可以将服务端签发的版本 2 会话票据（Session Ticket）作为判断依据。这表明服务端意图在该票据有效期内维持对版本 2 的支持（尽管无法提供绝对保证）。

#### 接收端应遵守的设计
* **票据签发与支持意图的传达**：服务端如果为版本 2 签发了会话票据，即代表向客户端传达了意图：在票据有效期内保持对版本 2 的支持。服务端应合理规划以履行这一支持意图。


### 7. 适用性

#### 双方均应遵守的设计
* **应用层能力一致性**：QUIC v2 提供给应用程序的能力与 QUIC v1 相比没有任何变化。
* **ALPN 兼容性**：所有被指定在 QUIC v1 上运行的应用层协议协商（ALPN）码点，均可直接在 QUIC v2 上运行。具体而言，包括但不限于 "h3"（HTTP/3）和 "doq"（DNS over QUIC）等 ALPN 均可在 QUIC v2 上运行。
* **扩展协议兼容性**：除非另有特别说明，所有定义为与 QUIC v1 配合使用的 QUIC 扩展，均自动适用于 QUIC v2。

#### 发送端应遵守的设计
* **复用上层协议标识**：在使用 QUIC v2 发起连接时，发送端可以并应当继续使用原本为 QUIC v1 设计的 ALPN 码点（如 "h3"）和 QUIC 扩展声明。

#### 接收端应遵守的设计
* **接受并处理兼容请求**：在接收并处理 QUIC v2 连接时，接收端必须能够接受和处理原本针对 QUIC v1 定义的 ALPN 码点和相关 QUIC 扩展请求，不能仅仅因为底层传输协议版本为 v2 而判定这些上层协议或扩展不适用。


### 8. 安全注意事项

#### 双方均应遵守的设计
* **安全与隐私属性一致性**：需明确 QUIC v2 与 QUIC v1 的版本属性完全相同，未对安全或隐私属性引入任何更改。
* **降级攻击的无害性认知**：虽然强制性的版本协商机制用于防范降级攻击，但由于 v1 和 v2 的安全属性完全一致，实际上降级本身不会带来额外的安全隐患。
* **指纹识别风险暴露**：需注意，支持 QUIC v2 这一特征本身，可能会被网络观察者利用，从而帮助其对客户端和服务端设备进行指纹识别（Fingerprinting）。

#### 发送端应遵守的设计
* **遵循通用安全基线**：本章节未引入针对发送端的新增安全机制，发送端在实现时直接继承并遵循 QUIC v1 的现有安全和隐私设计规范即可。

#### 接收端应遵守的设计
* **遵循通用安全基线**：本章节未引入针对接收端的新增安全机制，接收端在实现时直接继承并遵循 QUIC v1 的现有安全和隐私设计规范即可。


### 9. IANA 注意事项

#### 双方均应遵守的设计
* **IANA 注册表认知**：必须认知并遵循 IANA 在“QUIC 版本（QUIC Versions）”注册表（[https://www.iana.org/assignments/quic](https://www.iana.org/assignments/quic)）中为 QUIC v2（RFC 9369）添加的新条目。
* **版本号分配记录**：
    * **永久分配（Permanent）**：值 `0x6b3343cf` 被永久分配给 QUIC v2（变更控制方：IETF，联系方：QUIC WG）。
    * **临时分配（Provisional）**：值 `0x709a50c4` 被临时分配为 QUIC v2 的草案代码点（draft codepoint）。

#### 发送端应遵守的设计
* **标准版本号使用**：在发送正式的 QUIC v2 数据包或进行版本协商时，必须使用 IANA 官方永久分配的版本值 `0x6b3343cf`。
* **草案版本号使用**：在基于 QUIC v2 早期草案进行兼容性测试或通信时，可以使用临时分配的草案版本值 `0x709a50c4`。

#### 接收端应遵守的设计
* **标准版本号识别**：必须能够正确解析接收到的版本值 `0x6b3343cf`，并将其识别为官方正式发布的 QUIC v2 协议流量。
* **草案版本号识别**：应当能够解析接收到的临时版本值 `0x709a50c4`，并将其识别为 QUIC v2 的草案版本流量，以实现对早期实现的兼容与处理。


### 附录 A. 数据包保护示例

#### 双方均应遵守的设计
* **增量验证要求**：实现必须能够通过本附录提供的示例（包含客户端/服务端的 Initial 数据包及 Retry 数据包）进行增量逻辑验证。
* **连接 ID 规范**：所有示例数据包均使用由客户端选择的 8 字节目标连接 ID (DCID)：`0x8394c8f03e515708`。
* **HKDF 标签格式**：在执行 `HKDF-Expand-Label` 函数时，必须使用符合特定格式的标签（HkdfLabel.label），包括：
    * `client in` / `server in`
    * `quicv2 key` / `quicv2 iv` / `quicv2 hp` / `quicv2 ku`
* **初始密钥派生逻辑**：
    * 必须通过 `initial_secret = HKDF-Extract(initial_salt, cid)` 提取初始密钥。
    * 随后使用版本特定的标签派生出 `client_initial_secret`、`server_initial_secret` 及其对应的密钥（key）、初始向量（iv）和首部保护密钥（hp）。

#### 发送端应遵守的设计
* **Initial 数据包构建与填充**：
    * 负载（Payload）必须包含 `CRYPTO` 帧。
    * 必须添加足够的 `PADDING` 帧，确保负载达到 1162 字节，以满足路径 MTU 探测和防放大攻击的长度要求。
    * 未保护的报头必须正确包含 4 字节的数据包编号（如示例中的编号 2）。
* **应用首部保护（Header Protection）**：
    * 必须从加密后的负载中提取 16 字节的采样（sample）。
    * 使用 `AES-ECB` 算法（以 `hp` 为密钥）对采样进行处理生成掩码（mask）。
    * 将掩码的第一个字节与报头第 0 字节的低 4 位异或，掩码的其余部分与数据包编号字节异或。
* **Retry 数据包构造**：
    * 必须包含重试令牌（token）。
    * 必须计算重试完整性标签（Retry Integrity Tag），计算过程需包含客户端原始连接 ID，但该 ID 不得出现在发送的报文中。
* **短报头数据包（Short Header）构造**：
    * 使用 `AEAD_CHACHA20_POLY1305` 进行保护。
    * 随机数（nonce）必须由 IV 与 64 位宽的数据包编号（Packet Number）进行异或生成。

#### 接收端应遵守的设计
* **密钥同步与派生**：必须能够根据接收到的连接 ID 和版本特定的盐值，准确派生出用于解密对端流量的密钥对（key, iv, hp）。
* **报头去保护与解析**：
    * 在处理 Initial 包或短报头包时，必须首先根据加密负载中的采样恢复出原始报头字节和数据包编号。
    * 必须根据去保护后的报头信息识别数据包编号的长度（1~4 字节）。
* **解密验证流程**：
    * 使用派生的密钥和 IV 对负载进行 AEAD 解密。
    * 对于 Retry 数据包，必须使用专门的 `quicv2 retry secret` 派生的参数验证末尾的完整性标签。
* **短报头最小包处理**：必须能够处理极小长度的受保护数据包（如仅含 PING 帧的 21 字节数据包），并正确执行去保护和解密逻辑。
