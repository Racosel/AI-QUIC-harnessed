### 第一章 概述（Overview）


以下概念中，`>`表示左侧概念只包含一个右侧概念，`>>`表示可包含一个或多个右侧概念。


`QUIC连接（Conn） >> 流（Stream）`


`IP包（Packet） > UDP报文（Datagram） >> QUIC包（Packet） >> QUIC帧（Frame）`


### 第二章 流（Streams）

通用处理规则：

- 路由寻址：优先通过非零长度的目的连接 ID（DCID）匹配关联连接。若 DCID 长度为零，则退化为基于网络四元组（源/目的 IP 及端口）进行匹配。
- 状态一致性校验：丢弃与当前连接状态不符（如协议版本不同、解密失败）的数据包。若缺乏完整性保护的无效数据包在校验前已引发内存状态变更，必须执行状态回滚或触发连接错误。
- 孤立包处理：对于无法匹配现有连接的数据包，端点可发送无状态重置（Stateless Reset）报文，快速通知对端连接不可用。

服务端处理规则：

- 版本验证与协商：收到不支持版本的数据包时，若满足该版本最小包长限制，应返回版本协商（Version Negotiation）报文；若长度不足则直接丢弃。
- 新连接处理：当数据包无法匹配现有连接时：
  - 若为合法的 Initial 报文，分配资源并推进握手流程。
  - 若拒绝接入，返回携带 CONNECTION_REFUSED 错误码的 Initial 报文。
- 乱序包容错：允许缓存因乱序提前到达的无主 0-RTT 报文，等待 Initial 报文到达后合并处理。丢弃任何无主的 Handshake 报文及其他非法报文。

客户端处理规则：

- 强身份匹配：仅处理 DCID（或零长度下的本地 IP/端口）与本端选择完全匹配的数据包，匹配失败者直接丢弃。
- 版本零容忍：若接收到的数据包指示的协议版本与本端初始选择的版本不一致，必须无条件丢弃。
- 密钥就绪前缓存：允许缓存因网络乱序提前到达、但当前尚未获取对应解密密钥的加密数据包，待后续密钥就绪后再行解密。


### 第三章 流状态（Stream States）


#### 发送方状态机

```text
       o
       | 本端创建新流
       | 对端创建双向流
       v
   +-------+
   | Ready | 发送 RESET_STREAM
   |       |-----------------------.
   +-------+                       |
       |                           |
       | 发送  STREAM /             |
       |      STREAM_DATA_BLOCKED  |
       v                           |
   +-------+                       |
   | Send  | 发送  RESET_STREAM     |
   |       |---------------------->|
   +-------+                       |
       |                           |
       | 发送  STREAM + FIN         |
       v                           v
   +-------+                   +-------+
   | Data  | 发送 RESET_STREAM  | Reset |
   | Sent  |------------------>| Sent  |
   +-------+                   +-------+
       |                           |
       | 收齐全部 ACKs              | 收到 ACK
       v                           v
   +-------+                   +-------+
   | Data  |                   | Reset |
   | Recvd |                   | Recvd |
   +-------+                   +-------+

```
发送方：

- 正常发送带有数据的`STREAM`帧，被流控限制时，发送`STREAM_DATA_BLOCKED`帧通知接收方；
- 发送完成，并转换到`Data Sent`状态时，只重传必要的丢包，不再遵循流控；
- 主动放弃发送，或接收到对端的`STOP_SENDING`帧时，必须发送`RESET_STREAM`，进入终止流程。

#### 接收方状态机

```text
       o
       | Recv STREAM / STREAM_DATA_BLOCKED / RESET_STREAM
       | Create Bidirectional Stream (Sending)
       | Recv MAX_STREAM_DATA / STOP_SENDING (Bidirectional)
       | Create Higher-Numbered Stream
       v
   +-------+
   | Recv  | 接收 RESET_STREAM
   |       |-----------------------.
   +-------+                       |
       |                           |
       | 接收 STREAM + FIN          |
       v                           |
   +-------+                       |
   | Size  | 接收 RESET_STREAM      |
   | Known |---------------------->|
   +-------+                       |
       |                           |
       | 收齐全部数据                |
       v                           v
   +-------+ 接收 RESET_STREAM  +-------+
   | Data  |------(可选项)----->| Reset |
   | Recvd |    收齐全部数据     | Recvd |
   +-------+<-----(可选项)------+-------+
       |                           |
       | 应用完成数据读取             | 应用读取 RESET
       v                           v
   +-------+                   +-------+
   | Data  |                   | Reset |
   | Read  |                   | Read  |
   +-------+                   +-------+
```

接收方：

>（注意乱序，流控配额、取消等帧可能先于`STREAM`帧到达）

- 收到某流的第一个`STREAM`、`STREAM_DATA_BLOCKED`或`RESET_STREAM`帧时，创建接收侧；
- 对端发起的双向流，接收到`MAX_STREAM_DATA`或`STOP_SENDING`帧，也创建接收侧；

  > 即使乱序，收到“给予配额”或“停止发送”，代表对端一定建立了双向流。而第一条中的帧，无法判断该流的类型。

- 使用`MAX_STREAM_DATA`通知本端可用资源，在`Recv`状态下告知，转移到`Size Known`后不需要再告知（但仍要收齐缺失数据）；

  > 转移到`Size Known`，证明`FIN`包之前尚未收齐的数据量，在最后一次`MAX_STREAM_DATA`的配额范围内，之后无需再通告配额。

- 同一个数据包可能造成两次状态切换：`Recv`->`Size Known`->`Data Recvd`（带FIN的包刚好最后到达）；
- 在`Data Recvd`状态下，`RESET_STREAM`意味着无法保证应用获取完整数据，但接收方可扣押`RESET_STREAM`并完成向应用的数据递交（保持在`Data Recvd`）；或转移到`Reset Recvd`，并通知应用该流被重置（在`Reset Read`处结束）。


#### 工程实现

在遵循状态机示意图进行设计实现的同时，还需注意以下要点：

通用规则：
- 标识与隔离：流通过单次生命周期内绝对不可复用的 Stream ID 唯一标识。流关闭后，其 ID 永久作废。
- 双向关闭协同：彻底关闭双向流需双方配合——端点发送 RESET_STREAM 终止“本端到对端”的发送路径，发送 STOP_SENDING 触发“对端到本端”的终止流程。双方应在上述控制帧中镜像复制相同的应用级错误码，以保证上下层认知一致。
- 乱序与安全底线：协议栈必须能处理在任意状态下乱序到达的帧。若在重组数据时发现相同偏移量处的重复数据内容不一致（被篡改或严重错乱），必须立刻触发连接致命错误。

发送方规则：
- 发送与流控豁免：在活跃发送期（Ready / Send 状态），正常发送数据，遇流控阻塞时主动发送 STREAM_DATA_BLOCKED。一旦发送带有 FIN 标志的帧（切入 Data Sent 状态），该流宣告结束，此后仅重传丢失报文，不再受任何流控配额限制。
- 强制关闭执行：无论处于何种活跃状态，一旦决定主动放弃或收到对端的 STOP_SENDING 帧，必须发送 RESET_STREAM 切入终止流程。若当前已处于 Data Sent 状态，可推迟发送重置帧等待最后 ACK，但绝对禁止再重传任何业务数据。

接收方规则：
- 隐式创建与乱序接纳：必须容忍网络乱序。若收到配额内合法流的高序号首帧（包括数据帧或控制帧），接收方必须自动在内存中隐式创建该流及所有缺失的前置同类型流，并初始化为 Recv 状态。
- 配额通告与终值锁定：仅在 Recv 状态下使用 MAX_STREAM_DATA 帧向对端通告缓冲区额度。一旦收到携带 FIN 的帧（切入 Size Known 状态），必须立刻停止配额通告，锁定该流的“最终大小（Final Size）”，但需继续等待收齐此前遗漏的乱序包。
- 单向终止请求：接收方可通过发送 STOP_SENDING 帧单方面请求对端闭嘴。发出此帧后，接收方可直接丢弃该流新到达的业务数据（节省内存），但必须将这些被丢弃的数据字节数计入连接级别的全局流控消耗中，防止恶意带宽攻击。


### 第四章 流量控制（Flow Control）


双方都应该实现的通用架构：

- 流控的核心模块应当是可插拔的，可自由在多套流控算法上选择，可以先选择其一进行实现。

- 双方的状态机都必须维护三个维度的独立限额变量：
  - 单流字节偏移量（Stream-level byte offset）
  - 连接总字节偏移量（Connection-level byte offset）
  - 单向/双向流的并发总数（Stream count limit）

- 必须为 TLS 握手数据（CRYPTO 帧）实现一套完全独立的流控系统。它的计算配额和数据收发绝对不能与普通的 STREAM 数据帧混用，以防握手被死锁。
- 双方必须对一个流的最终大小保持绝对一致的认知。它代表该流对整个连接级配额的“永久性扣除”。正常结束时，通过带有`FIN`标志的`STREAM`帧传达，最终大小 = `Offset + Length`；异常结束时，通过`RESET_STREAM`帧传达，该帧内部专门有一个`Final Size`字段来携带这个值。


发送端的工程实现：

- 发送配额的单调递增逻辑：在接收对端发来的三种 `MAX_*` 限额帧时，必须实现防乱序机制。由于网络乱序，后收到的限额通告可能比当前的还要小。代码必须强制忽略较小的限额通告，永远只取历史最大值。

- 当发送端发现自己的数据或流数量被配额卡住时，必须启动一个机制，向对方发送相应的阻塞帧（`STREAM_DATA_BLOCKED`、`DATA_BLOCKED` 或 `STREAMS_BLOCKED`）。如果迟迟未解封，必须定期重传这些阻塞帧，以防止底层连接因为长时间没有数据包交互而触发空闲超时被强行关闭。

- 精确宣告最终大小（`Final Size`）：
  - 正常发完： 当业务发完数据时，打上`FIN`标志，底层协议栈必须能准确计算 `Offset + Length`，作为该流的最终大小。
  - 中途重置： 如果主动放弃发送（发送`RESET_STREAM`），必须在代码里准确抓取当前已发送的最大字节偏移量，填入该帧的`Final Size`字段。

接收端的工程实现:

- 主动且智能的窗口更新：
  - 绝不被动等待：不得将发送窗口更新（`MAX_*`）绑定在收到对方的阻塞帧这一事件上。
  - 动态步长与捎带：需要实现一套类似TCP的自适应算法，根据读取速度和RTT，提前算出需要给对方增加多少限额，并尽量寻找顺风车（与其他数据帧/ACK帧一起捎带发送），减少网络开销。
  
- 严格的越界拦截与错误分发引擎：接收端的解析器必须像防火墙一样严密，并抛出极度精确的错误码：
  - 发现对方发来的数据量超过了自己给的配额：触发`FLOW_CONTROL_ERROR`。
  - 发现对方开启的流总数超过了自己给的配额：触发`STREAM_LIMIT_ERROR`。
  - 数值溢出：在解析 `MAX_STREAMS`（无论是在握手参数还是在控制帧中）时，必须校验其值是否超过了 $2^{60}$。若超过，根据上下文分别触发 `TRANSPORT_PARAMETER_ERROR`（握手期）或 `FRAME_ENCODING_ERROR`（运行时）。
  
- 安全且不可变的流收尾逻辑：
  - 状态冻结：一旦收到对方的`RESET_STREAM`或带有`FIN`的帧，立刻停止对该流的配额更新，并锁定其`Final Size`。此后该流到达的任何超出该大小的数据包，全部直接丢弃。
  - 全局扣除：立刻用这个确定的`Final Size`去更新连接级别的总消耗量。
  - 防篡改校验：如果后续收到乱序的数据包，且这些包暗示的流大小与已经锁定的 `Final Size`不一致，应当立刻以`FINAL_SIZE_ERROR`终止整个连接，防止状态机错乱和内存越界。


### 第五章 连接（Connections）


#### 工程实现
作为客户端角色，应用层协议可以：

- 打开一个连接，与服务端开始加密和传输握手的交互；
- 在可用时启用Early Data（0-RTT数据）；
- 当Early Data被服务端接受或拒绝时被通知。

作为服务端角色，应用层协议可以：

- 侦听传入连接，为第7章中描述的交互做准备；
- 如果支持Early Data，则在发送给客户端的TLS会话恢复消息中嵌入应用控制数据；
- 如果支持Early Data，则从客户端的TLS会话恢复消息中提取应用控制数据，并根据该信息选择接受或拒绝Early Data。
  
作为任一角色，应用层协议都可以：

- 为每种类型允许的流个数初始值设置最小值；
- 通过为流和连接设置流控限额来控制接收缓冲区的资源分配；
- 确认握手是成功完成还是仍在进行中；
- 通过生成PING帧或在空闲超时到期之前请求传输层发送其他帧，保持连接不静默关闭；
- 立即关闭连接。


### 第六章 版本协商（Version Negotiation）

1. 触发机制与首包长度约束 (Trigger & Sizing)
协商触发：当服务端收到无法识别或不支持的 QUIC 版本数据包时，通过发送 Version Negotiation 报文来开启协商。

客户端的填充义务：为了确保服务端能正确响应，支持多版本的客户端应当（SHOULD）使用 PADDING 帧，将其发出的首个 UDP 报文长度填充至：其支持的所有版本中“最小容许长度”的最大值。

服务端的丢弃下限：如果服务端收到的首个报文长度，比它自己支持的所有版本的“最小容许长度”还要小，服务端可以直接忽略该报文，不发送任何协商响应。

2. 服务端发送策略与无状态容错 (Server Sending Strategy)
无状态响应：服务端在回复 Version Negotiation 报文（内含服务端支持的版本列表）时，允许在完全丢失连接状态（无状态）的情况下进行。

频率控制：服务端**可以（MAY）**自行限制发送版本协商包的数量。例如，当识别出这是一个超前的 0-RTT 数据包时，服务端可以选择不回复协商包，静默等待客户端后续发送正常的 Initial 握手包。

3. 客户端处理逻辑与异常豁免 (Client Handling & Exceptions)
禁止套娃：客户端**严禁（MUST NOT）**发送 Version Negotiation 报文去响应接收到的 Version Negotiation 报文。

默认断连：仅支持当前版本的客户端在收到版本协商包时，默认**必须（MUST）**放弃当前的连接尝试。

强制忽略的例外情况（防欺骗/防乱序）：在以下两种情况下，客户端**必须（MUST）**无视并丢弃收到的版本协商包：

客户端之前已经成功接收并处理了该连接的其他数据包（说明连接已建立，此协商包可能是旧包乱序到达，或是中间人伪造）。

收到的协商包中，服务端声明支持的版本列表里包含了客户端当前正在使用的版本（说明这是一个非法的协商包）。

4. 保留版本测试机制 (Reserved Versions)
测试专用版本号：QUIC 协议特意保留了格式为 0x?a?a?a?a 的版本号，专门用于未来的兼容性测试。

主动注入测试：端点**可以（MAY）**在自己发送的协商版本列表中混入这些保留版本号，或者直接发送携带保留版本号的数据包，以此来测试对端协议栈是否能够健壮地、正确地忽略或丢弃不支持的版本，而不至于发生崩溃。


### 第7章：加密与传输握手 (Cryptographic and Transport Handshake)

#### 1. 双方均应遵守的设计

* **握手机制整合**：QUIC 结合了密码学握手和传输层握手以将连接建立的延迟降至最低。必须使用 `CRYPTO` 帧来传输加密的握手数据。
* **版本与协议**：本文档定义的 QUIC 版本为 `0x00000001`，并使用 TLS 进行握手；不同的 QUIC 版本可能指示使用不同的密码学握手协议。
* **数据交付与偏移**：QUIC 必须为加密握手数据提供可靠且有序的交付。`CRYPTO` 帧使用的偏移量在每个数据包号空间（Packet Number Space）中必须从 0 开始。
* **强制的密码学属性**：加密握手必须提供以下属性：
    * **认证的密钥交换**：服务器必须始终被认证；客户端可以选择性被认证；每个连接产生独立且不相关的密钥；密钥材料可用于 0-RTT 和 1-RTT 包保护。
    * **传输参数的认证交换**：认证双方的传输参数值，并为服务器的传输参数提供机密性保护。
    * **应用协议的认证协商**：端点必须显式协商应用层协议（如使用 TLS 的 ALPN），避免协议使用分歧。
* **ECN 测试**：端点可以利用握手期间发送的数据包来测试显式拥塞通知 (ECN) 支持（通过观察 ACK 帧中是否携带 ECN 计数）。
* **连接ID (Connection ID) 协商与使用**：
    * 握手期间的长报头包含两个连接ID：目标连接ID (DCID) 和源连接ID (SCID)。
    * 处理完第一个 Initial 数据包后，端点必须将后续发送的数据包中的 DCID 字段设置为其接收到的 SCID 字段的值。
* **连接ID 认证 (7.3)**：
    * 握手期间对连接ID的选择必须通过在“传输参数”中包含所有相关值来进行认证。
    * 端点必须验证接收到的传输参数是否与数据包中接收到的连接 ID 值相匹配。
    * 缺少必要的连接ID传输参数（如 `initial_source_connection_id`，或服务器侧的 `original_destination_connection_id`），端点必须将其视为 `TRANSPORT_PARAMETER_ERROR` 类型的连接错误。
    * 传输参数与包首部字段的值不匹配，或者 Retry 包相关参数异常，必须视为 `TRANSPORT_PARAMETER_ERROR` 或 `PROTOCOL_VIOLATION`。
* **传输参数通用规则 (7.4)**：
    * 端点必须将接收到的无效传输参数值视为 `TRANSPORT_PARAMETER_ERROR` 错误。
    * 端点严禁在同一个传输参数扩展中多次发送同一个参数；收到重复参数应当视为 `TRANSPORT_PARAMETER_ERROR`。
    * 端点必须忽略它不支持的传输参数。

#### 2. 发送端应遵守的设计

* **数据包合并发送**：发送端可以将多个不同类型的 QUIC 数据包合并到单个 UDP 数据报中发送。
* **客户端发送 Initial 包**：
    * 在未收到服务器 Initial 或 Retry 包之前，客户端必须为 DCID 填充一个不可预测的值，且长度至少为 8 字节。
    * 在收到服务器数据包之前，客户端必须在该连接的所有数据包上使用这个相同的 DCID。
    * 客户端需自行选择 SCID 的值并设置相应长度字段。
* **客户端发送 0-RTT 包**：
    * 第一批 0-RTT 数据包必须使用与客户端第一个 Initial 包相同的 DCID 和 SCID。
    * 客户端严禁使用记住的特定参数（如 `ack_delay_exponent`, `max_ack_delay`, `preferred_address` 等），必须改用服务器提供的新值或默认值。
    * 发送 0-RTT 包中的帧时，客户端必须仅使用“记住的”传输参数，严禁使用从服务器新学到的更新值（新值仅适用于 1-RTT 包）。
* **更改 DCID 的限制**：
    * 客户端必须仅响应收到的第一个 Initial 或 Retry 包来更改其用于发送数据包的 DCID。
    * 服务器必须仅根据收到的第一个 Initial 包来设置其用于发送数据包的 DCID。后续的更改只能通过 `NEW_CONNECTION_ID` 帧获取。

#### 3. 接收端应遵守的设计

* **非法数据包丢弃机制**：
    * 一旦客户端收到了来自服务器的有效 Initial 包，必须丢弃随后在该连接上收到的任何具有不同 SCID 的数据包。
    * 如果后续的 Initial 包包含不同的 SCID，必须将它们丢弃，以避免无状态处理带来的不可预测后果。
* **服务器接收并处理 0-RTT 数据**：
    * 服务器使用记住的传输参数来决定是否接受 0-RTT 数据。
    * 如果服务器接受了 0-RTT 数据，它严禁降低任何限制或更改任何可能被 0-RTT 数据违反的值（例如各种 stream 限制不能小于记住的值）。
    * 如果恢复的传输参数值无法被当前连接支持，服务器必须拒绝 0-RTT 数据。
    * 服务器可以将客户端在 0-RTT 中使用更新后的传输参数视为 `PROTOCOL_VIOLATION`。
* **加密消息缓冲机制 (7.5)**：
    * 接收端必须支持缓冲至少 4096 字节的乱序 `CRYPTO` 帧数据。
    * 如果握手期间缓冲区被超出，且未临时扩容，接收端必须以 `CRYPTO_BUFFER_EXCEEDED` 错误码关闭连接。
    * 握手完成后，如果接收端无法缓冲 `CRYPTO` 帧中的所有数据，可以丢弃该帧以及未来收到的所有 `CRYPTO` 帧，或者以 `CRYPTO_BUFFER_EXCEEDED` 关闭连接。
    * 包含被丢弃 `CRYPTO` 帧的数据包也必须被确认 (Acknowledged)，因为该数据包已被传输层接收和处理。


### 第8章：地址验证 (Address Validation)

#### 1. 双方均应遵守的设计

* **防放大攻击限制 (Anti-amplification limit)**：为了防止流量放大攻击，在收到来自尚未验证地址的数据包后，端点**必须 (MUST)** 将发送到该未验证地址的数据量限制为不超过从该地址接收到的数据量的三倍。
* **地址验证场景**：地址验证在连接建立期间和连接迁移期间都必须执行。
* **隐式地址验证**：成功处理来自对端的 Handshake 数据包后，即可认为对端地址已成功验证。此外，如果对端使用的连接 ID 是由本端点选择的，且该连接 ID 包含至少 64 位的熵，端点**可以 (MAY)** 认为对端地址已验证。
* **路径验证机制 (Path Validation)**：用于连接迁移等场景以测试 2 元组（IP地址和端口）的可达性。
    * 验证不意味着可以确认对端能在返回方向上发送数据。
    * 收到 `PATH_CHALLENGE` 数据包的确认 (ACK) 不等同于路径验证成功，因为 ACK 可能被伪造。必须依赖 `PATH_RESPONSE` 帧。
* **路径验证的异常处理**：
    * 端点**应当 (SHOULD)** 基于计时器放弃路径验证（推荐超时时间为：当前 PTO 与新路径 PTO 较大者的 3 倍）。
    * 与其他要求扩展数据报大小的情况不同，端点**严禁 (MUST NOT)** 丢弃包含 `PATH_CHALLENGE` 或 `PATH_RESPONSE` 帧但看起来太小的数据报。

#### 2. 发送端应遵守的设计

* **服务器端的初始发送限制**：在验证客户端地址之前，服务器**必须 (MUST)** 遵守 3 倍防放大限制。在计算时，服务器**必须 (MUST)** 将唯一归属于该单个连接的所有 UDP 数据报中的有效载荷字节数都计算在内（包括被成功处理的包和被全盘丢弃的包）。
* **客户端的 Initial 数据包填充**：客户端**必须 (MUST)** 确保包含 Initial 数据包的 UDP 数据报具有至少 1200 字节的 UDP 有效载荷，必要时需添加 `PADDING` 帧。
* **死锁预防机制**：为防止服务器触发防放大限制导致连接卡死，客户端**必须 (MUST)** 在探测超时 (PTO) 时发送数据包。如果没有 Handshake 密钥，客户端**必须 (MUST)** 发送包含至少 1200 字节的 Initial 数据报；否则发送 Handshake 数据包。
* **Token 的发送与使用 (客户端行为)**：
    * **Retry Token**：收到 Retry 数据包后，客户端**必须 (MUST)** 在该连接后续发送的所有 Initial 数据包中重复该 Token。该 Token **严禁 (MUST NOT)** 用于未来的其他连接。
    * **NEW_TOKEN**：客户端在收到 `NEW_TOKEN` 帧后，**应当 (SHOULD)** 在未来连接该服务器时将此 Token 包含在 Initial 数据包中。
    * **隐私保护**：为了防止被网络路径上的实体关联，客户端**应当避免 (SHOULD NOT)** 在不同的连接尝试中重复使用来自 `NEW_TOKEN` 的同一个 Token。
* **发起路径验证 (Initiating Path Validation)**：
    * 端点**必须 (MUST)** 在每个 `PATH_CHALLENGE` 帧中使用不可预测的数据，以便与对端的回显关联。
    * 端点**应当避免 (SHOULD NOT)** 在单个数据包中发送多个 `PATH_CHALLENGE` 帧。
    * 端点**必须 (MUST)** 将包含 `PATH_CHALLENGE` 帧的数据报扩展至至少 1200 字节（最小允许的最大数据报大小），除非防放大限制不允许发送此大小的数据报。
    * 如果因防放大限制未能扩展到 1200 字节，端点**必须 (MUST)** 在后续条件允许时执行第二次携带 1200 字节数据报的路径验证，以确保路径 MTU 达标。

#### 3. 接收端应遵守的设计

* **Token 的构造与完整性 (服务器行为)**：
    * 服务器构造的 Token **必须 (MUST)** 允许其区分 Token 是来源于 Retry 还是 `NEW_TOKEN`。
    * Token **必须 (MUST)** 包含完整性保护，防止被客户端修改或伪造。
    * `NEW_TOKEN` 帧发出的 Token **严禁 (MUST NOT)** 包含允许观察者将值与发出它的连接相关联的信息（除非已加密）。服务器**必须 (MUST)** 确保发送的每个 `NEW_TOKEN` 跨所有客户端是唯一的。
* **Token 的接收与校验 (服务器行为)**：
    * 收到携带 Token 的 Initial 数据包时，除非已完成验证，否则服务器**必须 (MUST)** 尝试验证该 Token。
    * 如果收到包含无效 Retry Token 的 Initial 包，服务器**应当 (SHOULD)** 立即以 `INVALID_TOKEN` 错误码关闭连接。
    * 如果 `NEW_TOKEN` 或其他 Token 验证失败，服务器不应直接丢弃数据包，而**应当 (SHOULD)** 继续处理，将其视为未验证地址的客户端（可能会进而发送 Retry 包）。
* **响应路径验证 (Path Validation Responses)**：
    * 收到 `PATH_CHALLENGE` 帧后，端点**必须 (MUST)** 在 `PATH_RESPONSE` 帧中回显完全相同的数据。
    * 除非受拥塞控制限制，端点**严禁 (MUST NOT)** 延迟发送包含 `PATH_RESPONSE` 的数据包。
    * `PATH_RESPONSE` 帧**必须 (MUST)** 在接收到对应 `PATH_CHALLENGE` 的同一网络路径上发送。
    * 端点**必须 (MUST)** 将包含 `PATH_RESPONSE` 帧的数据报扩展至至少 1200 字节，但如果扩展后会超出防放大限制，则**严禁 (MUST NOT)** 扩展。
    * 对于一个 `PATH_CHALLENGE` 帧，端点**严禁 (MUST NOT)** 发送超过一个 `PATH_RESPONSE` 帧作为响应。


### 第9章：连接迁移 (Connection Migration)

#### 1. 双方均应遵守的设计

* **迁移前提限制**：端点**严禁 (MUST NOT)** 在握手被确认 (confirmed) 之前发起连接迁移。
* **禁用主动迁移 (disable_active_migration)**：如果对端发送了此传输参数，端点**严禁 (MUST NOT)** 从不同的本地地址向对端在握手期间使用的地址发送数据包（除非是为了响应 `preferred_address`）。如果对端违反此要求，端点**必须 (MUST)** 直接丢弃该路径上的入站数据包，或者继续进行路径验证并允许迁移，但**严禁**生成无状态重置 (Stateless Reset) 或关闭连接，以防被第三方网络攻击者利用。
* **被动迁移与 NAT 重新绑定**：并非所有地址更改都是主动迁移（如 NAT 导致的端口/IP变化）。如果端点检测到对端地址发生任何变化，除非之前已验证过该地址，否则**必须 (MUST)** 执行路径验证。
* **连接状态清理**：当端点没有经过验证的路径可以发送数据包时，它**可以 (MAY)** 丢弃连接状态（支持迁移的端点也可以选择等待新路径可用）。
* **连接 ID (Connection ID) 的隐私与重用限制 (9.5)**：
    * 端点**严禁 (MUST NOT)** 在从多个本地地址发送数据包时重用同一个连接 ID（例如发起迁移或探测新路径时）。
    * 端点**严禁 (MUST NOT)** 在发送到多个目标地址时重用同一个连接 ID。
    * 如果对端请求了零长度 (zero-length) 的连接 ID，端点**应当避免 (SHOULD NOT)** 发起迁移，因为这会导致流量极易被链接分析。
    * 为了确保迁移可行且流量无法被关联，端点**应当 (SHOULD)** 在对端迁移之前提供新的连接 ID。
* **IPv6 流标签 (9.7)**：发送 IPv6 数据的端点**应当 (SHOULD)** 应用 IPv6 流标签，且生成算法**必须 (MUST)** 尽量减少与以前使用的流标签的关联性。

#### 2. 发送端应遵守的设计

*(注：本文档限制只能迁移到新的客户端地址，因此客户端通常作为迁移的发起端/发送端)*

* **发起迁移的责任**：客户端负责发起所有迁移。
* **发起迁移的方式 (9.2)**：端点可以通过从新的本地地址发送包含**非探测帧 (non-probing frames)** 的数据包来将连接迁移到该新地址。（注：探测帧仅包含 `PATH_CHALLENGE`, `PATH_RESPONSE`, `NEW_CONNECTION_ID`, `PADDING`，其余均为非探测帧）。
* **免首包验证**：由于握手时已验证过对端地址，迁移端点向对端发送数据时，可以直接迁移到新的本地地址，而无需事先验证对端地址。
* **建立新路径可达性**：端点**必须**在新路径上发起路径验证，但**可以 (MAY)** 推迟到对端向其新地址发送下一个非探测帧之后进行。
* **拥塞控制与丢包恢复 (9.4)**：
    * 确认对端拥有新地址后，端点**必须 (MUST)** 立即将新路径的拥塞控制器和往返时间 (RTT) 估计器重置为初始值（除非仅端口发生改变，此时**可以 (MAY)** 保留原有状态）。
    * 在旧路径上发送的数据包**严禁 (MUST NOT)** 参与新路径的拥塞控制或 RTT 估算。
* **迁移到服务器首选地址 (Preferred Address, 9.6)**：
    * 如果服务器提供了 `preferred_address`，握手确认后，客户端**应当 (SHOULD)** 选择其中一个地址并使用未使用的连接 ID 发起路径验证。验证成功后，**应当**将所有未来数据包发往新地址；如果失败，**必须 (MUST)** 继续发送到服务器的原始 IP。
    * 客户端在迁移到首选地址之前**必须 (MUST)** 对其进行验证。

#### 3. 接收端应遵守的设计

*(注：服务器通常作为迁移的接收端/响应端)*

* **处理未知地址的数据包**：
    * 服务器在看到来自新客户端地址的非探测数据包之前，**严禁**向该地址发送非探测数据包。
    * 如果客户端收到来自未知服务器地址的数据包（且自身未发起向该地址的迁移），客户端**必须 (MUST)** 丢弃这些数据包。
* **响应连接迁移 (9.3)**：
    * 收到来自新地址的非探测帧后，如果接收端允许迁移，它**必须 (MUST)** 将后续数据包发送到该新地址，并且如果尚未验证，**必须 (MUST)** 启动路径验证。
    * 接收端**必须**仅响应具有**最高数据包号 (highest-numbered)** 的非探测数据包来更改发送地址，以防止乱序数据包导致发送地址回滚。
* **防范地址欺骗与攻击**：
    * **验证前限制**：在验证对端新地址之前，必须限制向该地址发送的数据量（遵守防放大限制）。
    * **路径欺骗 (On-Path Spoofing, 9.3.2)**：如果新对端地址的验证失败，接收端**必须 (MUST)** 恢复使用最后一次验证过的对端地址。如果没有任何关于上次验证地址的状态，它**必须 (MUST)** 丢弃所有状态并静默关闭连接。
    * **离线包转发 (Off-Path Packet Forwarding, 9.3.3)**：为了防范攻击者转发旧数据包伪造 NAT 重新绑定，端点在响应明显的迁移时，**必须 (MUST)** 使用 `PATH_CHALLENGE` 帧验证先前处于活动状态的路径。
* **服务器处理首选地址迁移 (9.6.2)**：
    * 服务器接受连接后，如果在其首选地址收到 `PATH_CHALLENGE`，必须响应 `PATH_RESPONSE`。
    * 在从首选地址收到客户端的非探测数据包并验证新路径之前，服务器**必须 (MUST)** 继续从原始地址发送非探测数据包。
    * 服务器**必须 (MUST)** 从其首选地址向客户端进行路径探测，以防范攻击者发起虚假迁移。
    * 一旦验证完成并在首选地址收到具有新最大数据包号的非探测数据包，服务器**应当 (SHOULD)** 丢弃在旧 IP 地址上收到的该连接的较新数据包。


### 第10章：连接终止 (Connection Termination)

#### 1. 双方均应遵守的设计

* **连接终止方式**：已建立的 QUIC 连接可以通过三种方式终止：空闲超时 (Idle Timeout)、立即关闭 (Immediate Close) 或 无状态重置 (Stateless Reset)。
* **空闲超时机制 (10.1)**：
    * 有效的空闲超时时间是两个端点声明的 `max_idle_timeout` 参数的**最小值**。连接空闲时间超过此值将被静默关闭并丢弃状态。
    * 收到并成功处理对端的数据包，或者在未发送过其他需要确认的数据包时发送了 ack-eliciting（需要确认）的数据包，均会重启空闲计时器。
    * 为了避免超时时间过短，端点**必须 (MUST)** 将空闲超时期间增加到至少当前探测超时 (PTO) 的 3 倍。
    * 端点可以使用 PING 或其他需要确认的帧来进行活性测试 (Liveness Testing) 或推迟空闲超时。
* **立即关闭的连接状态持续时间 (10.2)**：
    * closing（关闭）和 draining（排空）状态**应当 (SHOULD)** 至少持续当前 PTO 间隔的 3 倍。
    * 状态结束后，端点**应当 (SHOULD)** 丢弃所有连接状态。对于后续属于该连接的入站包，端点**可以 (MAY)** 发送无状态重置作为响应。
* **无状态重置 Token 规则 (10.3)**：
    * 令牌 (Token) 是一个难以猜测的 16 字节值，与特定的连接 ID (CID) 绑定。当 CID 被退休 (RETIRE_CONNECTION_ID) 时，对应的令牌失效。
    * 计算令牌的方式必须难以猜测（如使用静态密钥和 CID 通过 HKDF 生成）。
    * 相同的无状态重置令牌**严禁 (MUST NOT)** 用于多个 CID。重复的值**可以 (MAY)** 被视为 `PROTOCOL_VIOLATION` 连接错误。

#### 2. 发送端应遵守的设计

*(注：此处发送端指主动发起连接关闭或无状态重置的一方)*

* **发起立即关闭 (Immediate Close, 10.2 & 10.2.1)**：
    * 端点发送 `CONNECTION_CLOSE` 帧以立即终止连接（这会隐式重置所有流），并立即进入 **closing（关闭）状态**。
    * 在 closing 状态下，端点仅保留生成 `CONNECTION_CLOSE` 帧及识别该连接所需的信息。收到归属该连接的包时，端点**必须**响应 `CONNECTION_CLOSE`。
    * 端点**应当 (SHOULD)** 限制在 closing 状态下生成响应数据包的速率。
    * 如果端点丢弃了包保护密钥，**必须 (MUST)** 将其发送数据包的累积大小限制为收到数据包大小的 3 倍，以防止被用于放大攻击。
    * 在 closing 状态下，如果收到来自未验证地址的数据包，端点**必须 (MUST)** 要么丢弃该包，要么将发送的累积大小限制为收到大小的 3 倍。
* **握手期间的立即关闭 (10.2.3)**：
    * 握手确认后，**必须 (MUST)** 在 1-RTT 数据包中发送 `CONNECTION_CLOSE` 帧。
    * 在握手确认之前，端点**应当 (SHOULD)** 在 Initial 或 Handshake 等多个可用级别的包中冗余发送，以确保对端能够处理。
    * 在 Initial 或 Handshake 包中发送 `CONNECTION_CLOSE` 时，**必须 (MUST)** 将应用层错误 (类型 0x1d) 替换为传输层错误 (类型 0x1c) 并清空 Reason Phrase 字段，以防泄漏应用状态。
* **发送无状态重置 (Stateless Reset, 10.3)**：
    * 仅作为端点失去连接状态时的最后手段。如果端点可以通信，**严禁 (MUST NOT)** 使用它来指示活动连接的致命错误，而必须使用 `CONNECTION_CLOSE`。
    * 无状态重置包**必须 (MUST)** 格式化为短报头 (short header) 数据包，最后 16 字节包含重置令牌。
    * 端点**应当 (SHOULD)** 确保其发送的所有正常数据包至少比要求对端使用的最小 CID 长度长 22 字节（必要时添加 `PADDING`），以使无状态重置包在外观上与普通包无法区分。
    * 严禁发送大小达到或超过收到数据包 3 倍的无状态重置包。
    * **防死循环 (Looping, 10.3.3)**：端点**必须 (MUST)** 确保其发送的每个无状态重置包都小于触发它的数据包，除非它维护了足以防止循环的状态。

#### 3. 接收端应遵守的设计

*(注：此处接收端指接收到关闭信号或无状态重置的一方)*

* **接收立即关闭与 Draining 状态 (10.2.2)**：
    * 收到 `CONNECTION_CLOSE` 帧后，端点进入 **draining（排空/引流）状态**（这表明对端正在关闭或排空）。
    * 在 draining 状态下，端点**严禁 (MUST NOT)** 发送任何数据包。
    * 端点在进入 draining 状态前，**可以 (MAY)** 发送唯一一个包含 `CONNECTION_CLOSE` 帧的数据包，之后必须保持静默，以防止双方无限交换关闭帧。
* **检测与处理无状态重置 (10.3.1)**：
    * 端点**必须 (MUST)** 丢弃太小而不可能是有效 QUIC 数据包的数据包（例如在现有 AEAD 算法下，短于 21 字节的短报头数据包均无效）。
    * 尽管无状态重置必须格式化为短报头，但端点**必须 (MUST)** 将任何以有效无状态重置令牌结尾的数据包视为无状态重置（兼容未来可能使用长报头的版本）。
    * 当传入数据报的第一个包无法关联到现有连接或无法解密时，端点**必须 (MUST)** 使用数据报的最后 16 字节与已知的令牌进行比较。
    * 比较令牌的操作**必须 (MUST)** 在不泄漏令牌值信息的情况下进行（例如必须使用恒定时间比较 Constant-time comparison，防止时序侧信道攻击）。
    * 端点**严禁 (MUST NOT)** 检查与其未使用过的或已退休的 CID 相关联的重置令牌。
    * 如果数据报的最后 16 字节与无状态重置令牌完全相同，端点**必须 (MUST)** 进入 draining（排空）状态，并停止在该连接上发送任何数据包。


### 第11章：错误处理 (Error Handling)

#### 1. 双方均应遵守的设计

* **错误信号传递**：检测到错误的端点**应当 (SHOULD)** 向对端发出存在该错误的信号。应当在帧中包含最合适的错误码（见第20章）。
* **错误码的降级使用**：在检测到错误情况时，端点**可以 (MAY)** 使用任何适用的错误码；在无法确定具体错误时，始终可以使用通用错误码（例如 `PROTOCOL_VIOLATION` 或 `INTERNAL_ERROR`）来代替特定的错误码。
* **无状态重置 (Stateless Reset) 的限制**：无状态重置不适用于任何可以使用 `CONNECTION_CLOSE` 或 `RESET_STREAM` 帧发出信号的错误。如果端点拥有在连接上发送帧所需的连接状态，则**严禁 (MUST NOT)** 使用无状态重置。

#### 2. 发送端应遵守的设计

*(注：此处发送端指发出错误信号的一方)*

* **连接级错误 (Connection Errors, 11.1)**：
    * 导致连接不可用的错误（如明显的协议语义违规或影响整个连接的状态损坏）**必须 (MUST)** 使用 `CONNECTION_CLOSE` 帧发出信号。
    * 传输层特定的错误（包括本文档中描述的所有错误）**必须**使用帧类型为 `0x1c` 的 `CONNECTION_CLOSE` 帧。
    * 应用程序特定的协议错误**必须**使用帧类型为 `0x1d` 的 `CONNECTION_CLOSE` 帧。
* **重传关闭帧**：如果端点在已终止的连接上继续收到更多数据包，它**应当 (SHOULD)** 准备好重传包含 `CONNECTION_CLOSE` 帧的数据包（同时需限制重传次数和时间以节省资源）。如果不选择重传，一旦关闭帧丢失，端点只能依赖无状态重置机制 (10.3)。
* **处理无效的 Initial 包**：由于 Initial 包的 AEAD 不提供强认证，端点**可以 (MAY)** 直接丢弃无效的 Initial 包（即使规范在其他地方强制要求视为连接错误），以减少被拒绝服务攻击 (DoS) 的风险。前提是端点不处理该包中的帧，或者能回滚任何处理的影响。
* **流级错误 (Stream Errors, 11.2)**：
    * 如果应用层错误仅影响单个流且连接可恢复，端点可以使用带有适当错误码的 `RESET_STREAM` 帧仅终止受影响的流。

#### 3. 接收端应遵守的设计

*(注：在错误处理机制中，端点关于流错误的行为主要由应用协议控制)*

* **流重置的触发限制 (11.2)**：
    * 没有应用协议参与的情况下重置流可能会导致应用协议进入不可恢复的状态。因此，`RESET_STREAM` **必须 (MUST)** 仅由使用 QUIC 的应用协议发起（通过直接 API 调用），或由远程实例发送 `STOP_SENDING` 帧触发。
    * 应用协议（而非 QUIC 传输层）**应当 (SHOULD)** 定义处理被任何一端提前取消的流的规则。


### 第12章：数据包和帧 (Packets and Frames)

#### 1. 双方均应遵守的设计

* **数据包加密与保护级别 (12.1)**：
    * **1-RTT & 0-RTT 包**：使用加密握手派生的密钥，具有强机密性和完整性保护。
    * **Initial 包**：使用网络路径上可见的值派生密钥，因此**没有**有效的机密性保护，仅用于验证路径和防止意外修改。
    * **Retry 包**：使用 AEAD 防篡改，无机密性。
    * **版本协商包 (Version Negotiation)**：无任何密码学保护。
* **数据包号空间 (Packet Number Spaces)**：
    * 划分为三个空间：**Initial 空间**、**Handshake 空间** 和 **Application Data 空间**（包含 0-RTT 和 1-RTT 包）。
    * 各个空间的数据包号均从 0 开始。
    * 后续在同一空间发送的数据包，其数据包号**必须 (MUST)** 至少增加 1。
    * 端点**严禁 (MUST NOT)** 在同一连接的同一包号空间内重用数据包号。
* **帧的通用约束 (12.4)**：
    * 包含帧的数据包的有效载荷**必须 (MUST)** 包含至少一个帧。收到不含任何帧的数据包**必须 (MUST)** 视为 `PROTOCOL_VIOLATION`。
    * 帧**必须**完全适合单个 QUIC 数据包内部，**严禁**跨越多个数据包。
    * 帧类型**必须 (MUST)** 使用可能的最短变量整数编码（例如 `PING` 必须编码为单字节 `0x01`）。过度编码**可以 (MAY)** 视为 `PROTOCOL_VIOLATION`。
    * 收到未知类型的帧**必须 (MUST)** 视为 `FRAME_ENCODING_ERROR`。
* **帧与数据包类型的映射规则 (12.4 & 12.5)**：
    * `PADDING`, `PING`, 和 `CRYPTO` 帧可以出现在任何包号空间中。
    * 传输层错误帧 `CONNECTION_CLOSE` (类型 0x1c) 可以出现在任何包号空间中；应用层错误帧 `CONNECTION_CLOSE` (类型 0x1d) **必须 (MUST)** 仅出现在 Application Data 空间中。
    * `ACK` 帧可以出现在任何空间，但**必须**只确认同一空间中的数据包。
    * 其他所有帧类型**必须 (MUST)** 仅在 Application Data 空间（1-RTT，部分 0-RTT）中发送。
    * 端点将帧包含在不允许出现的包类型中，接收端**必须 (MUST)** 视为 `PROTOCOL_VIOLATION`。

#### 2. 发送端应遵守的设计

* **数据包合并 (Coalescing Packets, 12.2)**：
    * 发送端可以利用 `Length` 字段将多个不同加密级别的 QUIC 数据包合并到一个 UDP 数据报中发送。
    * 发送端**严禁 (MUST NOT)** 将具有不同连接 ID (Connection ID) 的 QUIC 数据包合并到同一个 UDP 数据报中。
    * 如果要在同一个加密级别发送多个帧，端点**应当 (SHOULD)** 将它们放入单个数据包中，而不是合并同加密级别的多个数据包。
    * 短报头数据包 (1-RTT)、Retry 包和版本协商包不包含 `Length` 字段，因此它们后面无法跟随其他数据包，只能是 UDP 数据报中的最后一个包。
* **数据包号上限限制 (12.3)**：
    * 包号范围是 0 到 $2^{62}-1$。
    * 如果用于发送的数据包号达到 $2^{62}-1$，发送端**必须 (MUST)** 关闭连接，不得发送 `CONNECTION_CLOSE` 帧或任何进一步的数据包；但对于后续收到的对端数据包，**可以 (MAY)** 发送无状态重置 (Stateless Reset)。
* **0-RTT 数据包中的限制 (12.5)**：
    * 发送端**严禁**在 0-RTT 数据包中发送以下帧：`ACK`, `CRYPTO`, `HANDSHAKE_DONE`, `NEW_TOKEN`, `PATH_RESPONSE`, 以及 `RETIRE_CONNECTION_ID`。

#### 3. 接收端应遵守的设计

* **处理合并的数据包 (12.2)**：
    * 接收端**必须 (MUST)** 具备处理合并数据包的能力。
    * 合并的 UDP 数据报中的每个 QUIC 数据包都是独立且完整的。接收端**必须 (MUST)** 单独处理每个 QUIC 数据包并单独确认 (ACK) 它们。
    * 如果合并包中的某个数据包解密失败，接收端**可以 (MAY)** 丢弃或缓冲它以备后用，并且**必须 (MUST)** 尝试处理该数据报中的剩余数据包。
    * 接收端**应当 (SHOULD)** 忽略数据报中目标连接 ID (DCID) 与第一个数据包 DCID 不同的任何后续数据包。
* **数据包去重处理 (12.3)**：
    * 由于不同加密级别使用不同密钥，重复数据包的丢弃 (Duplicate Suppression) **必须 (MUST)** 在移除包保护（即解密）之后进行。
    * 接收端**必须 (MUST)** 丢弃刚解除保护的数据包，除非它确信尚未处理过来自同一数据包号空间内具有相同数据包号的另一个数据包。
    * 为了防止为检测重复而维护过多的状态，接收端可以设置一个最低数据包号水位线，低于该水位线的所有数据包将被立即丢弃。
* **处理非法的 0-RTT 帧 (12.5)**：
    * 服务器如果在 0-RTT 数据包中收到了不允许包含的帧（如 `ACK` 或 `CRYPTO`），**可以 (MAY)** 将其视为 `PROTOCOL_VIOLATION` 错误。


### 第13章：打包与可靠性 (Packetization and Reliability)

#### 1. 双方均应遵守的设计

* **包处理与确认基础 (13.1)**：
    * 数据包**严禁 (MUST NOT)** 在成功移除包保护（解密）且包含的所有帧都已被处理之前被确认 (ACK)。
    * 对于虚假确认（即收到自己未曾发送过的数据包的 ACK），端点**应当 (SHOULD)** 视其为 `PROTOCOL_VIOLATION` 类型的连接错误。
* **重传机制原则 (13.3)**：
    * QUIC **不原样重传**被判定为丢失的整个数据包或帧。相反，需要重传的信息会被重新打包进新的帧和新的数据包中发送。
    * 端点**应当 (SHOULD)** 优先考虑重传丢失的数据，而不是发送新数据（除非应用层指定的优先级另有规定）。
    * 接收端**必须 (MUST)** 接受包含过时帧的数据包（例如，包含比旧包中更小的最大数据值 `MAX_DATA` 帧）。
* **ECN 的使用与反馈 (13.4)**：
    * 即使端点在发送时未设置 ECT (ECN-Capable Transport) 字段，如果它能访问接收到的 IP 头的 ECN 字段，它也**必须 (MUST)** 提供有关其收到的 ECN 标记的反馈，否则对端将禁用 ECN。
    * 每个数据包号空间维护独立的确认状态和 ECN 计数。合并的 QUIC 数据包 (Coalesced packets) 共享同一个 IP 头，因此根据该单个 IP 头的 ECN 字段为每个包号空间递增一次计数。
    * 重复的 QUIC 数据包不被处理，也**严禁**增加 ECN 计数。

#### 2. 发送端应遵守的设计

* **数据包组装 (Packetization)**：
    * 发送端可以等待很短的时间以收集多个帧放入单个最大化填充的包中，但延迟应保守以避免增加延迟。
    * 包含 `PADDING` 帧的数据包被计入拥塞控制的在途字节数。发送端**应当 (SHOULD)** 确保除了 `PADDING` 帧外，还定期发送其他帧以触发接收端的确认，避免拥塞窗口死锁。
* **特定信息的重传规则 (13.3)**：
    * `CRYPTO` 帧中的数据和 `STREAM` 帧中的应用数据必须重传直至被确认（除非流发送了 `RESET_STREAM`）。
    * `RESET_STREAM` 帧再次发送时，其内容**严禁 (MUST NOT)** 更改。
    * 包含 `CONNECTION_CLOSE` 帧的连接关闭信号在检测到丢包时**不进行重传**（参考第10章的特定重传机制）。
    * 各类流量控制帧（`MAX_DATA`, `MAX_STREAM_DATA`, `MAX_STREAMS`）和阻塞信号帧（`DATA_BLOCKED` 等）在最新包丢失或限制更新时发送更新值，但需注意避免发送过于频繁。
    * `HANDSHAKE_DONE` 帧**必须 (MUST)** 重传直至被确认。
* **ECN 验证与降级 (13.4.2)**：
    * 发送端收到包含 ECN 计数的 ACK 帧时，必须对其进行验证（例如检查确认了 ECT(0)/ECT(1) 包的 ACK 是否确实包含了 ECN 计数，或计数增长是否合理）。
    * 处理没有增加最大已确认包号的乱序 ACK 帧时，发送端**严禁 (MUST NOT)** 因此而判定 ECN 验证失败。
    * 如果 ECN 验证失败，发送端**必须 (MUST)** 禁用 ECN（停止在发出的 IP 包中设置 ECT 代码点）。

#### 3. 接收端应遵守的设计

* **生成 ACK 帧的规则 (13.2.1)**：
    * 每个包都**应当 (SHOULD)** 被确认至少一次。触发确认的包 (ack-eliciting packets) **必须 (MUST)** 在端点声明的 `max_ack_delay` 期限内至少被确认一次。
    * 接收端**必须 (MUST)** 立即确认所有触发确认的 Initial 和 Handshake 包。
    * 接收端**严禁 (MUST NOT)** 在响应收到触发确认包时发送超过一个纯 ACK 包（因其不受拥塞控制）。
    * 接收端**严禁 (MUST NOT)** 为了响应非触发确认包（如纯 ACK 包）而发送非触发确认包，以避免确认死循环。
* **需要无延迟立即发送 ACK 的情况 (13.2.1)**：
    * 收到的包的包号**小于**另一个已收到的触发确认包号。
    * 收到的包的包号**大于**已收到的最大触发确认包号，且两者之间存在包号空缺（即检测到乱序或丢包）。
    * 收到了 IP 头带有 ECN-CE (拥塞经历) 标记的数据包。
* **ACK 频率与区间管理 (13.2.2 & 13.2.3)**：
    * 接收端**应当 (SHOULD)** 在收到至少两个触发确认包后发送一个 ACK 帧。
    * 接收端**必须 (MUST)** 保留 ACK 区间 (ACK Range)，除非它能确保后续不会接受该区间内的数据包。即使丢弃所有区间，也**必须 (MUST)** 保留已成功处理的最大数据包号。
    * 接收端**应当 (SHOULD)** 在每个 ACK 帧中包含一个含有最大接收包号的 ACK 区间。
* **测算并报告主机延迟 (ACK Delay, 13.2.5)**：
    * 接收端在填充 ACK 帧的 ACK Delay 字段时，**严禁 (MUST NOT)** 包含它无法控制的延迟（例如包被挂起在 OS 内核中的时间）。
    * 接收端**应当 (SHOULD)** 包含由于解密密钥不可用而导致的缓冲延迟。
    * 当测算出的确认延迟大于 `max_ack_delay` 时，接收端**应当 (SHOULD)** 报告该测算延迟。
* **包保护空间的隔离 (13.2.6)**：
    * ACK 帧**必须 (MUST)** 仅携带在与其确认的数据包相同的包号空间的数据包中。（注意：客户端用 0-RTT 保护发送的包，服务器**必须 (MUST)** 用 1-RTT 密钥保护的数据包来确认）。


### 第14章：数据报大小 (Datagram Size)

#### 1. 双方均应遵守的设计

* **数据报的定义**：数据报大小 (Datagram size) 指的是承载 QUIC 数据包的单个 UDP 数据报的总 UDP 有效载荷大小（包含一个或多个 QUIC 报头和受保护的载荷，但不包括 UDP 或 IP 首部）。
* **绝对最小尺寸要求**：如果网络路径无法支持至少 1200 字节的最大数据报大小，则**严禁 (MUST NOT)** 使用 QUIC。
* **分片禁令**：UDP 数据报**严禁 (MUST NOT)** 在 IP 层进行分片。在 IPv4 中，如果可能，**必须 (MUST)** 设置不分片 (DF) 位以防止路径上的分片。
* **数据报大小验证的松散性**：QUIC 规范有时要求数据报不小于特定大小（例如地址验证时的 1200 字节要求）。但是数据报的大小本身是不被认证的（接收端无法确定发送端是以这个大小发送的，还是在路径上被更改了）。因此，当端点收到不符合大小约束的数据报时，**严禁 (MUST NOT)** 关闭连接；但**可以 (MAY)** 直接丢弃此类数据报。

#### 2. 发送端应遵守的设计

* **Initial 数据报的强制扩展 (14.1)**：
    * 客户端**必须 (MUST)** 将承载 Initial 数据包的所有 UDP 数据报的有效载荷扩展至至少 1200 字节（最小允许的最大数据报大小）。可通过向 Initial 包中添加 `PADDING` 帧或合并包来实现。
    * 服务器**必须 (MUST)** 将承载*触发确认 (ack-eliciting)* 的 Initial 数据包的所有 UDP 数据报的有效载荷扩展至至少 1200 字节。
    * 发送方如果确信网络路径和对端支持，发送的 Initial 数据报**可以 (MAY)** 超过 1200 字节。
* **路径最大传输单元探测 (PMTU, 14.2)**：
    * 端点**应当 (SHOULD)** 使用 DPLPMTUD 或 PMTUD 机制来探测支持的最大数据报大小。
    * 在缺乏这些探测机制的情况下，发送端**应当避免 (SHOULD NOT)** 发送大于 1200 字节（最小允许的最大数据报大小）的数据报。
    * 所有的非 PMTU 探测数据包**应当 (SHOULD)** 调整其大小以适应最大数据报大小，以防碎片化或被丢弃。
    * 如果确定某路径的 PMTU 无法支持至少 1200 字节，端点**必须 (MUST)** 立即停止在该受影响的路径上发送 QUIC 数据包（除了 PMTU 探测包或关闭帧）。
    * 由于每对本地和远程地址组合可能具有不同的 PMTU，实现**应当 (SHOULD)** 为每对组合维护一个最大数据报大小。
* **发送 PMTU 探测包 (14.4)**：
    * PMTU 探测包是触发确认的数据包 (ack-eliciting packets)。
    * PMTU 探测包可能包含长报头（为了让 Source Connection ID 放入前部），发送端可以组合长报头包（仅用于 ICMP 路由截断截取）与短报头包（用于实际确认）进行探测。
    * PMTU 探测包丢失不代表拥塞，**应当避免 (SHOULD NOT)** 触发拥塞控制降速反应，但由于其占用拥塞窗口，需谨慎发送以防阻塞应用层数据。

#### 3. 接收端应遵守的设计

* **Initial 包的小于限制处理 (14.1)**：
    * 服务器**必须 (MUST)** 丢弃承载在有效载荷小于 1200 字节的 UDP 数据报中的 Initial 数据包。
    * 对于此类非法包，服务器**可以 (MAY)** 立即以 `PROTOCOL_VIOLATION` 错误码发送 `CONNECTION_CLOSE` 帧以关闭连接。
* **通过 ICMP 消息进行 PMTUD 处理 (14.2.1)**：
    * 如果 ICMP 消息声称 PMTU 已经降至低于 QUIC 允许的最小数据报大小（1200 字节）以下，接收端**必须 (MUST)** 忽略该 ICMP 消息。
    * 接收端**应当 (SHOULD)** 验证收到的 ICMP 消息，以防范包注入攻击（验证应包括匹配 IP 地址、UDP 端口，并在可能的情况下匹配活动 QUIC 会话的 Connection ID）。
    * 接收端**应当 (SHOULD)** 忽略所有未通过验证的 ICMP 消息。
    * 接收端**严禁 (MUST NOT)** 仅基于 ICMP 消息去*增加* PMTU 的值。
    * 在响应 ICMP 消息降低最大数据报大小时，这种降低**可以 (MAY)** 是临时性的，直到 QUIC 的丢包检测算法真正确认该数据包已丢失。


### 第15章：版本 (Versions)

#### 1. 双方均应遵守的设计

* **版本号格式**：QUIC 版本使用 32 位无符号整数进行标识。
* **特定版本分配**：
    * `0x00000000`：保留用于表示版本协商 (Version Negotiation)。
    * `0x00000001`：标识本文档规范的 QUIC 版本，并且该版本使用 TLS 作为密码学握手协议。
* **版本不变性**：其他版本的 QUIC 可能具有与此版本不同的属性，但所有版本保证一致的 QUIC 属性在 `[QUIC-INVARIANTS]` 中有统一描述。
* **保留版本段**：最高有效 16 位被清零 (cleared) 的版本号，保留供未来的 IETF 共识文档使用。
* **强制版本协商的保留值**：模式为 `0x?a?a?a?a`（即所有字节的低四位在二进制下均为 `1010`）的版本号被保留，专门用于强制执行版本协商测试。这些保留版本号永远不会代表真实的协议。
* **声明支持**：端点（客户端或服务器）**可以 (MAY)** 声明支持任何上述用于强制版本协商的保留版本。

#### 2. 发送端应遵守的设计

*(注：在此上下文中，主动发送特定版本号以触发协商的通常为客户端)*

* **触发协商**：客户端**可以 (MAY)** 在发送数据包时使用保留的伪版本号（如 `0x?a?a?a?a` 模式），以此期望服务器启动版本协商过程。

#### 3. 接收端应遵守的设计

*(注：在此上下文中，广播支持版本或响应协商的通常为服务器)*

* **处理保留版本**：服务器**可以 (MAY)** 声明自己支持这些用于强制协商的保留版本，同时可以合理期望客户端会忽略该声明值。


### 第16章：可变长度整数编码 (Variable-Length Integer Encoding)

#### 1. 双方均应遵守的设计

* **编码机制**：QUIC 数据包和帧通常对非负整数值使用可变长度编码。该编码保留第一个字节的最高两位 (2MSB) ，用于编码以字节为单位的整数编码长度的以 2 为底的对数。整数值在剩余的位上以网络字节序 (network byte order) 进行编码。
* **编码规格与范围**：
    * **2MSB `00`**：长度 1 字节，可用位数 6，范围 0 - 63。
    * **2MSB `01`**：长度 2 字节，可用位数 14，范围 0 - 16383。
    * **2MSB `10`**：长度 4 字节，可用位数 30，范围 0 - 1073741823。
    * **2MSB `11`**：长度 8 字节，可用位数 62，范围 0 - 4611686018427387903。
* **不使用此编码的例外字段**：以下字段虽然被描述为整数，但**不使用**此可变长度编码：
    * 版本号 (Versions，第15章)
    * 报头中发送的数据包号 (Packet numbers，第17.1节)
    * 长报头数据包中的连接 ID 长度 (Length of connection IDs in long header packets，第17.2节)

#### 2. 发送端应遵守的设计

* **非强制的最小编码**：在绝大多数情况下，发送端不需要将值编码为必需的最小字节数（例如，值 1 也可以合法地编码为 2 字节、4 字节或 8 字节格式）。
* **Frame Type 字段的强制例外**：作为唯一的例外，发送端在编码帧类型 (Frame Type) 字段时，**必须 (MUST)** 使用可能的最短编码（如第 12.4 节所述）。

#### 3. 接收端应遵守的设计

* **解码兼容**：接收端必须根据第一个字节的最高两位 (2MSB) 准确识别并解码 1、2、4 或 8 字节的整数，并且应当能够正确处理和接受那些未使用最小必要字节数进行编码的合法整数值（除了违反了最短编码规则的 Frame Type 字段）。


### 第17章：数据包格式 (Packet Formats)

#### 1. 双方均应遵守的设计

* **编码基础**：所有数值均以网络字节序（即大端序，big-endian）编码。
* **Fixed Bit (固定位) 规则**：在长报头（版本协商包除外）和短报头（1-RTT包）中，字节 0 的 `0x40` 位**必须**设置为 1。包含 0 值的包在此版本中无效且**必须 (MUST)** 被丢弃。
* **Reserved Bits (保留位) 规则**：长短报头中预留的位在应用报头保护之前**必须 (MUST)** 设置为 0。接收端在移除包保护和报头保护后，如果发现这些位为非零值，**必须 (MUST)** 将其视为 `PROTOCOL_VIOLATION` 类型的连接错误。
* **连接 ID 长度限制 (17.2)**：在 QUIC v1 版本的长报头包中，目标连接 ID (DCID) 和源连接 ID (SCID) 的长度**严禁 (MUST NOT)** 超过 20 字节。接收端收到超过 20 字节的 v1 长报头包**必须 (MUST)** 直接丢弃。
* **废弃 Initial 包与状态 (17.2.2.1)**：
    * 客户端在发送其第一个 Handshake 包时停止发送和处理 Initial 包。
    * 服务器在收到其第一个 Handshake 包时停止发送和处理 Initial 包。
    * 到达此阶段后，双方必须丢弃 Initial 包的保护密钥以及相关的 `CRYPTO` 数据。
* **处理非法帧 (17.2.4)**：Initial 和 Handshake 数据包只能包含特定类型的帧（如 `CRYPTO`, `ACK`, `PING`, `PADDING`, 或类型 0x1c 的 `CONNECTION_CLOSE`）。端点如果收到包含其他类型帧的包，**必须 (MUST)** 将其视为 `PROTOCOL_VIOLATION`。
* **延迟自旋位 (Latency Spin Bit, 17.4)**：
    * 这是一项可选功能，仅在 1-RTT 包中存在。端点管理员必须能够全局或逐连接禁用它。
    * 即使未被管理员禁用，端点也**必须 (MUST)** 在至少 1/16 的网络路径或连接 ID 上随机禁用自旋位，以确保网络上能常见到禁用的情况。
    * 禁用自旋位时，端点可以设置任意值，并**必须 (MUST)** 忽略入站值（建议发送随机值）。
    * 在同一网络路径上更改连接 ID (CID) 时，端点必须将该路径的自旋值重置为 0。

#### 2. 发送端应遵守的设计

* **数据包号编码长度 (17.1)**：
    * 在收到某个包号空间的 ACK 之前，发送端**必须 (MUST)** 包含完整的包号（不进行截断）。
    * 收到 ACK 后，发送端**必须 (MUST)** 使用能表示“所发送的包号与最大已确认包号之间差值两倍以上”范围的编码长度（至少比连续未确认包数量的二进制对数多 1 位）。
* **版本协商包发送 (Version Negotiation, 17.2.1)**：
    * **必须 (MUST)** 将 `Version` 字段设置为 `0x00000000`。
    * 服务器**必须 (MUST)** 将收到的 SCID 复制到 DCID 中，将收到的 DCID 复制到 SCID 中，原样回显给客户端。
    * 服务器针对单个 UDP 数据报**严禁 (MUST NOT)** 发送超过一个版本协商包。
* **Initial 包发送 (17.2.2)**：服务器发送的 Initial 包**必须 (MUST)** 将 Token Length (令牌长度) 字段设为 0。
* **0-RTT 包发送 (17.2.3)**：
    * 客户端一旦开始处理来自服务器的 1-RTT 包，**严禁 (MUST NOT)** 再发送 0-RTT 包。
    * 0-RTT **严禁**包含对 1-RTT 帧的响应（例如 0-RTT 中不能包含 ACK）。
* **Retry 包发送 (17.2.5.1)**：
    * 服务器选择的源连接 ID (SCID) **严禁 (MUST NOT)** 与客户端发送数据包的 DCID 相同。
    * 服务器针对单个 UDP 数据报**严禁 (MUST NOT)** 发送超过一个 Retry 包。
* **发送 Retry 后的重新握手 (17.2.5.3)**：客户端收到 Retry 包后重发 0-RTT 数据包时，**必须 (MUST)** 使用全新的数据包号（防止相同的包号与密钥产生 nonce 重用从而破坏加密）。客户端**严禁 (MUST NOT)** 在处理 Retry 后重置任何包号空间的当前数据包号。
* **自旋位的设置 (17.4)**：启用自旋位时：
    * 服务器收到增加当前路径最高包号的包时，将其发送的自旋值设为与收到包中的自旋位**相同**。
    * 客户端收到增加当前路径最高包号的包时，将其发送的自旋值设为与收到包中的自旋位**相反 (inverse)**。

#### 3. 接收端应遵守的设计

* **数据包号解码 (17.1)**：接收端**必须**在移除包保护（解密）之前先移除报头保护。恢复完整包号的逻辑是寻找与“下一个预期包”（最高已接收包号 + 1）最接近的匹配值。
* **处理 Version Negotiation 包 (17.2.1)**：客户端**必须 (MUST)** 忽略该包中的 `Unused` 字段的值。
* **处理 Initial 包 (17.2.2)**：如果客户端收到一个 Token Length 字段非零的 Initial 包，**必须 (MUST)** 丢弃该包或生成 `PROTOCOL_VIOLATION` 错误。
* **处理 0-RTT 限制 (17.2.3)**：服务器**应当 (SHOULD)** 将 0-RTT 数据中违反“记住的传输参数限制”（如超出流数据限制）的行为视为相应类型的连接错误（例如 `FLOW_CONTROL_ERROR`）。
* **处理 Retry 包 (17.2.5.2 & 17.2.5.3)**：
    * 客户端对于每次连接尝试**必须 (MUST)** 最多接受并处理一个 Retry 包；处理过 Initial 或 Retry 包后，**必须 (MUST)** 丢弃后续收到的任何 Retry 包。
    * 客户端**必须 (MUST)** 丢弃 Retry Integrity Tag (重试完整性标签) 无法验证的 Retry 包，也必须丢弃 Token 长度为 0 的 Retry 包。
    * 客户端**必须 (MUST)** 丢弃 SCID 与其首个 Initial 包 DCID 完全相同的 Retry 包。
    * 客户端在响应 Retry 包发送新 Initial 包时，**必须 (MUST)** 将新 DCID 更改为 Retry 包中提供的 SCID，并携带 Token，但**严禁 (MUST NOT)** 更改自己的 SCID（防止破坏服务器的 Token 验证逻辑）。
    * 如果服务器检测到客户端在处理 Retry 包后非法重置了数据包号，服务器**可以 (MAY)** 中止连接。


### 第18章：传输参数编码 (Transport Parameter Encoding)

#### 1. 双方均应遵守的设计

* **编码结构**：传输参数包含在 `quic_transport_parameters` 扩展（定义于 `[QUIC-TLS]`）的 `extension_data` 字段中。它们被编码为一个参数序列，每个参数被编码为 `(Identifier, Length, Value)` 元组。
    * Length 字段以字节为单位表示 Value 字段的长度。
    * QUIC 将其编码为字节序列并包含在密码学握手中。
* **保留的标识符 (18.1)**：形式为 `31 * N + 27`（N 为整数）的标识符被保留，用于强制执行“忽略未知传输参数”的要求。这些参数没有语义，可以携带任意值。
* **默认值与整数编码**：
    * 如果传输参数缺失，除非另有说明，否则其默认值为 0。
    * 标识为整数的传输参数**必须**使用可变长度整数编码（第16章）。
* **连接 ID 限制 (active_connection_id_limit)**：
    * 限制端点愿意存储的来自对端的最大 CID 数量。
    * 该参数的值**必须 (MUST)** 至少为 2（如果不指定则默认为 2）。如果收到小于 2 的值，**必须 (MUST)** 以 `TRANSPORT_PARAMETER_ERROR` 错误码关闭连接。
    * 如果端点发出零长度的 CID，它将忽略从对端收到的此限制。
* **关闭主动迁移 (disable_active_migration)**：如果存在此参数（零长度值），接收到此参数的端点**严禁 (MUST NOT)** 在向对端握手时使用的地址发送数据时使用新的本地地址。
* **确认延迟参数**：
    * `ack_delay_exponent`：用于解码 ACK 帧中的延迟字段，默认值为 3（乘数 8）。大于 20 的值无效。
    * `max_ack_delay`：最大确认延迟，默认 25ms。大于或等于 $2^{14}$ 的值无效。
* **初始流控限制**：参数 `initial_max_data`、`initial_max_stream_data_*` 等等效于握手后立即发送相应的 `MAX_DATA` 或 `MAX_STREAM_DATA` 帧。如果参数缺失，该类型的流控制限制从 0 开始。
* **验证握手连接 ID**：`initial_source_connection_id` 包含端点在它发送的第一个 Initial 包中设置的 Source Connection ID，用于验证（详见第7.3节）。

#### 2. 发送端应遵守的设计

*(注：涉及服务端专用的传输参数)*

* **服务器专用的参数发送**：
    * `original_destination_connection_id`：**仅限服务器发送**。包含客户端首个 Initial 包的 DCID。
    * `stateless_reset_token`：**仅限服务器发送**。包含 16 字节令牌。如果服务器不发送此参数，则无法对握手期间协商的 CID 使用无状态重置。
    * `preferred_address`：**仅限服务器发送**。用于在握手结束时将客户端迁移到服务器的首选地址。包含 IPv4/IPv6 地址、端口、一个新的 CID 和对应的无状态重置令牌。
        * 服务器如果选择使用零长度的 CID，**严禁 (MUST NOT)** 提供首选地址。
        * 服务器**严禁 (MUST NOT)** 在此参数中包含零长度的 CID。
    * `retry_source_connection_id`：**仅限服务器发送**。包含服务器在 Retry 包中使用的 SCID。

#### 3. 接收端应遵守的设计

* **客户端对服务器专用参数的处理**：
    * 客户端**严禁 (MUST NOT)** 在其传输参数中包含任何仅限服务器的传输参数（即 `original_destination_connection_id`, `preferred_address`, `retry_source_connection_id`, 或 `stateless_reset_token`）。
* **服务器对非法参数的处理**：
    * 服务器如果收到客户端发送的上述任何“仅限服务器”的传输参数，**必须 (MUST)** 将其视为 `TRANSPORT_PARAMETER_ERROR` 类型的连接错误。
* **客户端对非法 preferred_address 的处理**：
    * 客户端如果收到包含零长度 CID 的 `preferred_address` 参数，或者当服务器协商使用零长度 CID 时提供了 `preferred_address`，客户端**必须 (MUST)** 将其视为 `TRANSPORT_PARAMETER_ERROR` 类型的连接错误。
* **处理最大 UDP 有效载荷 (max_udp_payload_size)**：
    * 限制接收端愿意接收的最大 UDP 负载大小（默认 65527）。接收端收到小于 1200 的值视为无效。

### 第19章：帧类型与格式 (Frame Types and Formats)

#### 0. 通用与扩展帧规则 (General & Extension Frames)
* **包含限制**：数据包的有效载荷中必须包含一个或多个帧。
* **扩展帧 (Type=未知)**：
  * QUIC 帧没有自描述编码。发送未知类型的扩展帧前，必须通过传输参数确保对端支持。
  * 扩展帧**必须 (MUST)** 受拥塞控制，并**必须 (MUST)** 触发 ACK 帧（除非该帧用于替换/补充 ACK）。

---

#### 1. PADDING 帧 (Type=0x00)
* **作用**：无语义值。用于增加数据包大小（例如将 Initial 包填充到最小要求大小，或防范流量分析）。
* **格式**：仅包含 1 字节的帧类型标识（无额外载荷）。

#### 2. PING 帧 (Type=0x01)
* **作用**：用于验证对端存活或检查可达性，也可用于保持连接活跃以防止超时。
* **格式**：仅包含 1 字节的帧类型标识（无额外载荷）。
* **规则**：接收端只需确认包含此帧的数据包即可。

#### 3. ACK 帧 (Type=0x02 或 0x03)
* **作用**：通知发送端哪些数据包已被接收和处理。QUIC 的确认是**不可撤销的**。
* **类型区分**：`0x03` 类型额外包含 ECN（显式拥塞通知）计数。
* **字段**：
  * `Largest Acknowledged` (最大确认包号)：正在确认的最大包号（完整值，未截断）。
  * `ACK Delay` (ACK 延迟)：以微秒为单位，解码时需乘以 2 的 `ack_delay_exponent` 次方。
  * `ACK Range Count` (ACK 区间计数)：后续 ACK Range 的数量。
  * `First ACK Range` (首个 ACK 区间)：`Largest Acknowledged` 之前的连续已确认包的数量。
  * `ACK Ranges`：交替出现的 `Gap`（未确认的间隙大小）和 `ACK Range Length`（连续确认的长度）。
  * `[ECN Counts]` (仅 0x03 包含)：包含 `ECT0 Count`, `ECT1 Count`, `ECN-CE Count`。
* **规则/错误**：
  * ACK 帧只能确认与其处于**相同数据包号空间**的包。
  * 如果根据 Gap 和 Range 计算出的包号为负数，**必须 (MUST)** 生成 `FRAME_ENCODING_ERROR`。

#### 4. RESET_STREAM 帧 (Type=0x04)
* **作用**：发送端用于突然终止某个流的发送部分。
* **字段**：`Stream ID` (流ID), `Application Protocol Error Code` (应用层错误码), `Final Size` (流的最终字节大小)。
* **规则/错误**：如果针对**仅发送 (send-only)** 的流收到此帧，**必须 (MUST)** 以 `STREAM_STATE_ERROR` 终止连接。

#### 5. STOP_SENDING 帧 (Type=0x05)
* **作用**：请求对端停止在某个流上发送数据（接收端直接丢弃后续入站数据）。
* **字段**：`Stream ID` (流ID), `Application Protocol Error Code` (应用层错误码)。
* **规则/错误**：如果针对**尚未创建的本地发起流**或**仅接收 (receive-only)** 的流收到此帧，**必须 (MUST)** 终止连接并返回 `STREAM_STATE_ERROR`。

#### 6. CRYPTO 帧 (Type=0x06)
* **作用**：传输密码学握手消息。提供有序的字节流，但没有流ID、没有流量控制、没有 FIN 标志。
* **字段**：`Offset` (字节偏移量), `Length` (数据长度), `Crypto Data` (密码学数据)。
* **规则/错误**：
  * 可在除 0-RTT 外的所有包类型中发送。每个加密级别是一个独立的流，Offset 从 0 开始。
  * Offset 与 Length 之和不能超过 `2^62-1`，否则**必须 (MUST)** 视为 `FRAME_ENCODING_ERROR` 或 `CRYPTO_BUFFER_EXCEEDED`。

#### 7. NEW_TOKEN 帧 (Type=0x07)
* **作用**：服务器向客户端提供 Token，供未来的 Initial 包使用。
* **字段**：`Token Length` (Token长度), `Token` (不透明的 Token 数据)。
* **规则/错误**：
  * Token **严禁 (MUST NOT)** 为空。收到空 Token **必须 (MUST)** 视为 `FRAME_ENCODING_ERROR`。
  * 客户端**严禁 (MUST NOT)** 发送此帧。服务器收到此帧**必须 (MUST)** 视为 `PROTOCOL_VIOLATION`。

#### 8. STREAM 帧 (Type=0x08..0x0f)
* **作用**：隐式创建流并承载流数据。低三位控制帧格式：`OFF`位(0x04)决定是否包含偏移量，`LEN`位(0x02)决定是否包含长度，`FIN`位(0x01)标记流结束。
* **字段**：`Stream ID`, `[Offset]` (缺省为0), `[Length]` (缺省则延伸至包尾), `Stream Data`。
* **规则/错误**：
  * 针对**尚未创建的本地发起流**或**仅发送流**收到此帧，**必须 (MUST)** 返回 `STREAM_STATE_ERROR`。
  * 最大偏移量超过 `2^62-1` **必须 (MUST)** 视为 `FRAME_ENCODING_ERROR` 或 `FLOW_CONTROL_ERROR`。

#### 9. MAX_DATA 帧 (Type=0x10)
* **作用**：连接级流控，通知对端整个连接上可发送的最大数据量。
* **字段**：`Maximum Data` (最大数据量，字节)。
* **规则/错误**：收到数据总量超过发送过的最大值，**必须 (MUST)** 返回 `FLOW_CONTROL_ERROR`。

#### 10. MAX_STREAM_DATA 帧 (Type=0x11)
* **作用**：流级流控，通知对端特定流上可发送的最大数据量。
* **字段**：`Stream ID`, `Maximum Stream Data` (最大流数据量，字节)。
* **规则/错误**：
  * 针对**未创建的本地发起流**或**仅接收流**收到此帧，**必须 (MUST)** 返回 `STREAM_STATE_ERROR`。
  * 某流收到数据超过上限，**必须 (MUST)** 返回 `FLOW_CONTROL_ERROR`。

#### 11. MAX_STREAMS 帧 (Type=0x12..0x13)
* **作用**：通知对端允许打开的给定类型流的累积数量。0x12 用于双向流，0x13 用于单向流。
* **字段**：`Maximum Streams` (最大流数量)。
* **规则/错误**：
  * 数量不能超过 `2^60`，超限**必须 (MUST)** 视为 `FRAME_ENCODING_ERROR`。
  * 如果对端打开流超过限制，**必须 (MUST)** 返回 `STREAM_LIMIT_ERROR`。必须忽略未增加限制的旧帧。

#### 12. DATA_BLOCKED 帧 (Type=0x14)
* **作用**：发送端因达到连接级流控限制而无法发送数据时发送，用于调整流控。
* **字段**：`Maximum Data` (发生阻塞时的连接级数据上限)。

#### 13. STREAM_DATA_BLOCKED 帧 (Type=0x15)
* **作用**：因达到流级流控限制而无法发送数据时发送。
* **字段**：`Stream ID`, `Maximum Stream Data` (发生阻塞时的流偏移量上限)。
* **规则/错误**：针对**仅发送流**收到此帧，**必须 (MUST)** 返回 `STREAM_STATE_ERROR`。

#### 14. STREAMS_BLOCKED 帧 (Type=0x16..0x17)
* **作用**：需要打开新流但受限于最大流数量时发送。0x16 用于双向，0x17 用于单向。
* **字段**：`Maximum Streams` (发送该帧时的最大流数量限制)。
* **规则/错误**：值超过 `2^60` **必须 (MUST)** 视为 `STREAM_LIMIT_ERROR` 或 `FRAME_ENCODING_ERROR`。

#### 15. NEW_CONNECTION_ID 帧 (Type=0x18)
* **作用**：向对端提供备用的连接 ID，用于连接迁移时打破链路关联。
* **字段**：`Sequence Number` (序列号), `Retire Prior To` (应在此之前退休的编号), `Length` (CID长度), `Connection ID`, `Stateless Reset Token` (对应的无状态重置令牌)。
* **规则/错误**：
  * 长度小于 1 或大于 20，**必须 (MUST)** 视为 `FRAME_ENCODING_ERROR`。
  * 发送端当前要求零长度 DCID，**严禁 (MUST NOT)** 发送此帧；接收端要求零长度且收到此帧，**必须 (MUST)** 返回 `PROTOCOL_VIOLATION`。
  * `Retire Prior To` > `Sequence Number` **必须 (MUST)** 视为 `FRAME_ENCODING_ERROR`。

#### 16. RETIRE_CONNECTION_ID 帧 (Type=0x19)
* **作用**：指示将不再使用对端发放的某个连接 ID，其关联的无状态重置令牌失效。
* **字段**：`Sequence Number` (正在退休的 CID 序列号)。
* **规则/错误**：
  * 序列号大于发送过的任何序列号，**必须 (MUST)** 返回 `PROTOCOL_VIOLATION`。
  * 序列号**严禁 (MUST NOT)** 指向承载该帧的数据包本身的 DCID，否则**可以 (MAY)** 返回 `PROTOCOL_VIOLATION`。
  * 零长度 CID 场景下**严禁 (MUST NOT)** 发送或接收此帧，否则视为 `PROTOCOL_VIOLATION`。

#### 17. PATH_CHALLENGE 帧 (Type=0x1a)
* **作用**：验证路径可达性（用于连接迁移等）。
* **字段**：`Data` (64位的任意数据/熵)。
* **规则**：接收端**必须 (MUST)** 生成包含相同 Data 值的 PATH_RESPONSE 帧。

#### 18. PATH_RESPONSE 帧 (Type=0x1b)
* **作用**：响应 PATH_CHALLENGE 帧。
* **字段**：`Data` (64位，拷贝自 Challenge)。
* **规则/错误**：如果 Data 与发送过的 Challenge 不匹配，**可以 (MAY)** 返回 `PROTOCOL_VIOLATION`。

#### 19. CONNECTION_CLOSE 帧 (Type=0x1c..0x1d)
* **作用**：通知对端连接正在关闭。0x1c 表示 QUIC 传输层错误（或 NO_ERROR），0x1d 表示应用层错误。
* **字段**：`Error Code` (错误码), `[Frame Type]` (触发错误的帧类型，仅 0x1c 包含), `Reason Phrase Length` (原因描述长度), `Reason Phrase` (原因描述，应为 UTF-8)。
* **规则**：0x1d 只能在 0-RTT 或 1-RTT 包中发送。握手期间的应用层错误应在 Initial/Handshake 包中使用 0x1c 和 `APPLICATION_ERROR` 码。

#### 20. HANDSHAKE_DONE 帧 (Type=0x1e)
* **作用**：服务器向客户端发送，确认握手完成。
* **格式**：空（无内容）。
* **规则/错误**：
  * 只能由服务器发送。服务器**严禁 (MUST NOT)** 在握手完成前发送。
  * 服务器收到此帧**必须 (MUST)** 视为 `PROTOCOL_VIOLATION`。


### 第20章：错误码 (Error Codes)

#### 1. 双方均应遵守的设计

* **数据类型与范围**：QUIC 的传输错误码 (Transport Error Codes) 和应用错误码 (Application Error Codes) 均为 **62 位无符号整数**。
* **应用协议错误码 (20.2)**：
  * 应用错误码的管理由具体的应用协议负责。
  * 它们专门用于 `RESET_STREAM` 帧（第19.4节）、`STOP_SENDING` 帧（第19.5节）以及类型为 `0x1d` 的 `CONNECTION_CLOSE` 帧（第19.19节）。
* **传输错误码字典 (20.1)**：这些错误码用于类型为 `0x1c` 的 `CONNECTION_CLOSE` 帧，并且其影响适用于整个连接。包含以下定义：
  * `NO_ERROR (0x00)`：在没有发生任何错误的情况下，端点突然关闭连接。
  * `INTERNAL_ERROR (0x01)`：端点遇到内部错误，无法继续连接。
  * `CONNECTION_REFUSED (0x02)`：服务器拒绝接受新连接。
  * `FLOW_CONTROL_ERROR (0x03)`：端点接收到的数据超过了其通告的数据流控限制。
  * `STREAM_LIMIT_ERROR (0x04)`：端点收到的帧的流标识符超出了其为相应流类型通告的流数量限制。
  * `STREAM_STATE_ERROR (0x05)`：端点收到一个帧，但该流当前的状态不允许接收该帧。
  * `FINAL_SIZE_ERROR (0x06)`：流的最终大小出现错误，包括：(1) 收到的 `STREAM` 帧包含的数据超出了之前确立的最终大小；(2) 收到的 `STREAM` 或 `RESET_STREAM` 帧中声明的最终大小低于已收到的流数据大小；(3) 声明的最终大小与之前确立的最终大小不一致。
  * `FRAME_ENCODING_ERROR (0x07)`：收到格式错误的帧（例如，未知类型的帧，或包含超出数据包剩余容量的 ACK 区间数量的 ACK 帧）。
  * `TRANSPORT_PARAMETER_ERROR (0x08)`：收到的传输参数格式错误、包含无效值、遗漏了强制参数、包含了被禁止的参数或存在其他错误。
  * `CONNECTION_ID_LIMIT_ERROR (0x09)`：对端提供的连接 ID (CID) 数量超过了本端通告的 `active_connection_id_limit`。
  * `PROTOCOL_VIOLATION (0x0a)`：检测到违反协议规范的错误，且该错误没有被更具体的错误码所涵盖。
  * `INVALID_TOKEN (0x0b)`：服务器收到包含无效 Token 字段的客户端 Initial 数据包。
  * `APPLICATION_ERROR (0x0c)`：应用或应用协议导致连接被关闭。
  * `CRYPTO_BUFFER_EXCEEDED (0x0d)`：端点收到的 `CRYPTO` 帧数据量超过了它的缓冲能力。
  * `KEY_UPDATE_ERROR (0x0e)`：端点在执行密钥更新时检测到错误。
  * `AEAD_LIMIT_REACHED (0x0f)`：端点达到了给定连接所使用的 AEAD 算法的机密性或完整性限制。
  * `NO_VIABLE_PATH (0x10)`：端点确定网络路径无法支持 QUIC（除非路径不支持足够大的 MTU，否则端点不太可能收到携带此代码的 `CONNECTION_CLOSE` 帧）。
  * `CRYPTO_ERROR (0x0100-0x01ff)`：密码学握手失败。此范围保留了 256 个值用于携带特定于密码学握手的错误。

#### 2. 发送端应遵守的设计

* **错误帧的正确选用**：
  * 当发送端需要发出传输层错误信号时，**必须**使用类型为 `0x1c` 的 `CONNECTION_CLOSE` 帧。
  * 当发送端需要发出应用层错误信号时，**必须**使用 `RESET_STREAM`、`STOP_SENDING` 或类型为 `0x1d` 的 `CONNECTION_CLOSE` 帧。
* **错误码的选择原则 (20.1)**：
  * 可能需要接收端采取特定操作的错误情况应使用唯一的特定代码。
  * 表示常见状态的错误应使用特定代码。
  * 如果不满足上述条件，应使用错误码来标识协议栈的一般功能模块（如流控制、传输参数处理等）。
  * 如果实现无法或不愿意使用更具体的代码，则**应当**提供通用的错误码（如 `PROTOCOL_VIOLATION` 或 `INTERNAL_ERROR`）。

#### 3. 接收端应遵守的设计

* **错误类型的区分与解析**：接收端**必须**能够解析长达 62 位的错误码，并能够根据 `CONNECTION_CLOSE` 的帧类型（`0x1c` 与 `0x1d`）严格区分并将其分别递交给传输层或应用层协议进行处理。
* **通用错误处理**：由于发送端有权在特定错误难以界定时使用通用错误码（如 `PROTOCOL_VIOLATION`），接收端在收到这些通用错误码时**必须**能够优雅地终止连接，即使缺乏详细的诊断信息。


### 第21章：安全注意事项 (Security Considerations)

#### 1. 双方均应遵守的设计

* **防放大攻击 (Anti-Amplification, 21.1.1.1)**：在地址验证完成之前，端点**严禁 (MUST NOT)** 向未验证的地址发送超过从该地址接收到的数据量三倍的数据。（注意：此限制不适用于客户端建立新连接或发起连接迁移时）。
* **防范乐观 ACK 攻击 (Optimistic ACK Attack, 21.4)**：为了检测对端是否确认了尚未收到的数据包，端点**可以 (MAY)** 在发送数据包时跳过某些数据包号。如果检测到此行为，端点可以立即以 `PROTOCOL_VIOLATION` 错误码关闭连接。
* **防范请求伪造 (Request Forgery, 21.5.6)**：
    * 端点**可以 (MAY)** 阻止向环回地址 (loopback address) 发起连接或迁移。如果相同服务之前在非环回接口上可用，端点**应当避免 (SHOULD NOT)** 允许迁移到环回地址。
    * 端点**可以 (MAY)** 将从公网地址向链路本地地址或私有地址的迁移视为潜在的请求伪造尝试，但除非有明确的安全指示，否则**应当避免 (SHOULD NOT)** 直接拒绝使用这些地址。
    * 端点**可以 (MAY)** 通过不在完成地址验证之前发送非探测帧，来降低请求伪造的风险。
* **防范对等端拒绝服务与资源耗尽 (Peer DoS & Resource Exhaustion)**：
    * **Slowloris 攻击 (21.6)**：部署**应当 (SHOULD)** 提供缓解措施，如限制单 IP 连接数、限制最低传输速度、限制连接最大存活时间等。
    * **流碎片化攻击 (21.7)**：部署**应当 (SHOULD)** 提供缓解措施，避免过度承诺内存，限制跟踪数据结构的大小，或延迟 `STREAM` 帧的重组。
    * **处理成本耗尽 (21.9)**：实现**应当 (SHOULD)** 跟踪处理成本与实际进展的比例，并将大量非生产性数据包视为攻击。端点**可以 (MAY)** 以连接错误或丢弃数据包作为响应。
* **防范 ECN 攻击 (21.10)**：为了防止攻击者伪造 IP 头部的 ECN 字段以影响发送速率，端点**必须**忽略 IP 数据包中的 ECN 字段，除非该 IP 数据包中至少有一个 QUIC 数据包被成功处理（解密并验证）。
* **防范流量分析 (Traffic Analysis, 21.14)**：端点可以使用 `PADDING` 帧来模糊数据包内容的实际长度，以防范基于包长度的流量分析。

#### 2. 发送端应遵守的设计

* **Initial 包的 DCID 随机性 (21.2)**：客户端在 Initial 数据包中选择的目标连接 ID (DCID) 必须是不可预测的。这可以确保握手数据包的密钥具有随机性，从而有力地证明发送方确实收到了响应的 Initial 包，防范脱机 (off-path) 攻击。
* **Token 的安全与使用**：
    * 客户端**可以**选择在服务器地址发生变化时不使用 `NEW_TOKEN` 帧提供的值（发送空 Token 字段），以避免利用 Token 进行的请求伪造攻击 (21.5.2)。
    * 服务器**应当 (SHOULD)** 通过限制地址验证 Token 的使用次数和生命周期，来缓解可能的放大攻击 (21.3)。
* **首选地址的验证限制 (21.5.3)**：在对首选地址 (Preferred Address) 进行验证之前，客户端**严禁 (MUST NOT)** 向该地址发送非探测帧 (non-probing frames)。
* **无状态重置预言机防范 (Stateless Reset Oracle, 21.11)**：
    * 共享静态密钥用于无状态重置的服务器集群，**必须 (MUST)** 确保具有给定 CID 的数据包始终路由到持有该连接状态的实例。
    * 如果具有相应 CID 的连接可能在任何使用相同静态密钥的端点上处于活动状态，服务器**严禁 (MUST NOT)** 生成无状态重置。

#### 3. 接收端应遵守的设计

* **未认证数据包的丢弃 (21.2)**：一旦密码学握手完成，QUIC 接收端会丢弃绝大多数未通过身份验证（解密失败）的数据包。这极大地限制了攻击者干扰现有连接的能力。
* **防范服务器端 DoS 计算耗尽 (21.1.1.2)**：由于计算首次完整的 TLS 握手响应成本高昂，服务器可以使用 `Retry` 数据包进行廉价的 Token 交换，在进行昂贵计算前先通过一个往返 (RTT) 验证客户端的 IP 地址。
* **请求伪造的底层防范 (21.5)**：由于伪造迁移攻击非常强大且缺乏绝对的反制措施，QUIC 服务器实现应当假定攻击者可以促使它们向任意目标生成 UDP 负载。因此，QUIC 服务器**应当避免 (SHOULD NOT)** 部署在未配置入口过滤 (ingress filtering) 且 UDP 端点未受充分保护的网络中。


### 第22章：IANA 注意事项 (IANA Considerations)

#### 1. 双方均应遵守的设计

*(注：本章主要针对互联网编号分配机构 (IANA) 以及协议扩展的开发者，定义了代码点 (codepoints) 的注册与管理策略。)*

* **注册表通用策略 (22.1)**：
  * QUIC 建立了多个注册表（版本、传输参数、帧类型、错误码），均允许临时 (Provisional) 和永久 (Permanent) 注册。
  * **代码点选择**：对 QUIC 注册表新代码点的请求**应当 (SHOULD)** 随机选择，避开已分配和首个未分配的值，以降低不同实现将不同语义赋予同一代码点的风险。
  * **可变长度整数编码的代码点**：对于使用可变长度整数编码的代码点（如帧类型），除非其用法对编码长度特别敏感，否则**应当 (SHOULD)** 使用编码为 4 字节或 8 字节的值（即值 $\ge 2^{14}$）。
* **QUIC 版本注册表 (Versions Registry, 22.2)**：
  * 使用 32 位空间。
  * `0x00000001`：永久分配给本文档定义的 QUIC 协议。
  * `0x00000000`：永久保留，专用于版本协商。
  * `0x?a?a?a?a`：所有符合该模式的代码点被保留，IANA **严禁 (MUST NOT)** 分配这些值，且它们**严禁 (MUST NOT)** 出现在已分配的值列表中。
* **QUIC 传输参数注册表 (Transport Parameters Registry, 22.3)**：
  * 使用 62 位空间。
  * `31 * N + 27` (N 为整数，即 27, 58, 89...)：这些形式的值被专门保留，用于测试对未知参数的忽略能力。IANA **严禁 (MUST NOT)** 分配这些值。
* **QUIC 帧类型注册表 (Frame Types Registry, 22.4)**：
  * 使用 62 位空间。
  * 新永久注册的规范**应当 (SHOULD)** 描述端点确定其“能够发送该识别类型的帧”的协商机制（通常需要伴随一个传输参数的注册）。
* **QUIC 传输错误码注册表 (Transport Error Codes Registry, 22.5)**：
  * 使用 62 位空间。
  * 对 `0x00` 到 `0x3f`（十六进制）之间的值，必须通过“标准操作”或“IESG 批准”进行分配，其他值需要“规范要求”。

#### 2. 发送端应遵守的设计

### 附录 A：伪代码示例 (Appendix A. Pseudocode)

#### 1. 双方均应遵守的设计

* **伪代码定位**：本节中的伪代码描述了示例算法，旨在保证逻辑的正确性和清晰度，而非追求极致的运行性能。
* **可变长度整数解码机制 (A.1)**：读取字节流时，根据首字节的最高两位（2MSB）确定该整数的总长度（1、2、4 或 8 字节）。剥离这最高两位后，将其余位与后续字节拼接，即可恢复出完整的整数值。
* **ECN 验证状态机概念 (A.4)**：为了测试新路径是否支持 ECN（显式拥塞通知），端点为路径分配一个 ECN 状态，包括：`"testing"`（测试中）、`"unknown"`（未知）、`"failed"`（失败）或 `"capable"`（可用）。端点也可以定期对判定为不支持 ECN 的路径进行重新评估。

#### 2. 发送端应遵守的设计

* **数据包号编码算法 (EncodePacketNumber, A.2)**：
    * 发送端在编码数据包号时，必须基于当前准备发送的完整包号 (`full_pn`) 和对端在当前包号空间已确认的最大包号 (`largest_acked`) 来决定编码长度。
    * 所需的位数**必须**至少比“连续未确认的数据包号数量（包含新包本身）”的以 2 为底的对数多 1 位。
    * 计算出所需的字节数（1、2、3 或 4 字节）后，将完整包号截断，仅保留并发送所需的最低有效字节。
* **ECN 探测发送与状态转换 (A.4)**：
    * 在 `"testing"` 或 `"capable"` 状态下，发送端发送带有 ECT 标记（默认为 ECT(0)）的数据包；在其他状态下发送无标记包。
    * 开始测试时，状态设为 `"testing"`，并记录现有的 ECN 计数作为基线。
    * 测试期（例如 10 个数据包或 3 倍 PTO）结束后，路径的 ECN 状态变为 `"unknown"`。
    * 从 `"unknown"` 状态开始，如果 ACK 帧中的 ECN 计数验证成功，路径状态变为 `"capable"`（除非没有任何带标记的包被确认）。
    * 如果带标记的包全部被声明为丢失，或者全部被标记为 ECN-CE，发送端**可以**直接将路径的 ECN 状态标记为 `"failed"`。如果在任何时候 ECN 计数验证失败，状态也变为 `"failed"`。

#### 3. 接收端应遵守的设计

* **数据包号解码算法 (DecodePacketNumber, A.3)**：
    * 接收端在移除报头保护后，需要基于“当前包号空间中已成功处理的最大包号 (`largest_pn`)”、“接收到的截断包号 (`truncated_pn`)”以及“包号字段的位数 (`pn_nbits`)”来恢复出完整的包号。
    * **解码核心逻辑**：
        1. 计算下一个预期包号 (`expected_pn = largest_pn + 1`)。
        2. 构建一个大小为 $2^{pn\_nbits}$ 的窗口 (`pn_win`) 和半窗口 (`pn_hwin`)。
        3. 在 `expected_pn` 减去半窗口到加上半窗口的范围内，寻找与截断值匹配的最接近的候选包号 (`candidate_pn`)。
    * 算法**必须**包含额外的边界检查，以防止包号在计算候选值时发生溢出或下溢（即确保候选包号在合理的有效范围内）。恢复出完整的包号是成功移除数据包加密保护的必要前提。
