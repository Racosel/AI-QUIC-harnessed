### 1. 简介 (Introduction)

#### 双方均应遵守的设计

* **背景与核心目标**：
    * QUIC传输协议旨在支持HTTP语义。虽然HTTP/2使用HPACK进行头部压缩，但在HTTP/3中如果直接使用HPACK，由于HPACK依赖所有流上的帧完全有序的内置假设，会导致严重的队头阻塞（Head-of-Line blocking）。
    * QPACK复用了HPACK的核心概念，但进行了重新设计，以确保在数据包乱序交付（out-of-order delivery）时的正确性。
    * 设计允许具体实现具有灵活性，以便在“抵抗队头阻塞”和“优化压缩率”之间取得平衡。
    * 总体目标：在相同的丢包条件下，压缩率应极度接近HPACK，同时大幅减少队头阻塞。
* **术语与通用定义**：
    * **关键字规范**：必须遵守 BCP 14 ([RFC2119] [RFC8174]) 中关于 "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHOULD" 等大写关键字的约束。
    * **HTTP字段 (HTTP fields)**：作为HTTP消息一部分发送的元数据。该术语同时包含头部字段（header）和尾部字段（trailer）。为保证通用性，统一使用“字段（fields）”。
    * **HTTP字段行 (HTTP field line)**：作为HTTP字段部分发送的键值对（name-value pair）。
    * **HTTP字段值 (HTTP field value)**：与字段名称关联的数据。由该部分中具有该字段名称的所有字段行值，通过逗号分隔符拼接而成。
    * **字段部分 (Field section)**：与HTTP消息关联的有序HTTP字段行集合。它可以包含多个同名字段行，也可包含重复的字段行。HTTP消息可以同时包含头部部分和尾部部分。
    * **表示 (Representation)**：用于表示一个字段行的指令，通常通过引用动态表（dynamic table）和静态表（static table）来实现。
    * **绝对索引 (Absolute Index)**：动态表中每个条目拥有的唯一索引。
    * **基准 (Base)**：相对索引（relative indices）和基准后索引（post-Base indices）的参考点。引用动态表条目的“表示”是相对于Base计算的。
    * **插入计数 (Insert Count)**：插入动态表中的条目总数。
    * **名称说明**：QPACK是一个专有名词，而不是缩写。
* **符号与图表约定**：
    * `x (A)`：表示 x 的长度为 A 个比特（bits）。
    * `x (A+)`：表示 x 使用前缀整数编码（后续4.1.1节定义），并以一个 A 比特的前缀开始。
    * `x ...`：表示 x 是可变长度的，并且一直延伸到该数据区域的末尾。

#### 发送端应遵守的设计

* **编码器角色 (Encoder)**：发送端作为编码器的具体实现，负责将HTTP字段部分（field sections）进行压缩和编码。

#### 接收端应遵守的设计

* **解码器角色 (Decoder)**：接收端作为解码器的具体实现，负责对接收到的已编码HTTP字段部分（encoded field sections）进行还原和解码。


### 2. 压缩过程概述 (Compression Process Overview)

#### 双方均应遵守的设计

* **双表机制**：与 HPACK 类似，QPACK 使用两个表将字段行（“头部”）与索引建立关联。
  * **静态表 (Static Table)**：预先定义，包含常见的头部字段行（部分带有空值）。
  * **动态表 (Dynamic Table)**：在连接过程中逐步建立，编码器可利用它来索引已编码字段部分中的头部和尾部字段行。
* **单向流通信**：QPACK 定义了专用的单向流，用于在编码器到解码器、以及解码器到编码器之间发送控制指令。
* **指令与数据的分离**：已编码的字段部分（encoded field sections）与修改动态表的编码器流指令（encoder stream instructions）在独立的流上到达（这与 HPACK 中指令混合在头部块中不同）。

#### 发送端应遵守的设计

* **基本编码与顺序**：编码器负责将头部或尾部部分转换为一系列“表示（representations）”（包括索引表示和字面量表示）。**必须（MUST）**严格按照输入字段部分中字段行的出现顺序发出其对应的表示。
* **状态跟踪的转移**：QPACK 的设计将可选的状态跟踪负担放在了发送端（编码器）上，以此来保持接收端（解码器）的相对简单。
* **动态表插入与驱逐限制**：
  * 编码器**可以（MAY）**在动态表中插入任何它选择的条目，不局限于当前正在压缩的字段行。
  * 动态表条目插入后不能立即被驱逐。只有当插入操作被解码器确认，且在未确认的表示中不再存在对该条目的未决引用时，该条目才变为可驱逐状态（evictable）。
  * 如果动态表空间不足，且需要被驱逐的旧条目尚未处于可驱逐状态，编码器**不得（MUST NOT）**将新条目（包括现有条目的副本）插入动态表。为避免此情况，编码器必须跟踪每个尚未被解码器确认的动态表引用。
* **避免受限插入的启发式策略**：
  * 为了防止无法添加新条目，编码器应避免引用即将被驱逐的旧条目。作为替代，编码器可以发出“重复（Duplicate）”指令并引用该条目的新副本。
  * 编码器可维护一个“排空索引（draining index）”，即它愿意引用的动态表中的最小绝对索引。通过不引用小于该索引的条目，确保旧条目的引用计数最终归零并可被驱逐。
* **阻塞流（Blocked Streams）控制**：
  * 每个已编码的字段部分都包含一个“所需插入计数（Required Insert Count, RIC）”。引用未确认的动态表条目会导致 RIC 增加，若发生丢包或乱序，可能导致解码器由于尚未收到对应的动态表条目而发生流阻塞。
  * 编码器**必须（MUST）**在任何时候都将可能被阻塞的流数量限制在解码器通过 `SETTINGS_QPACK_BLOCKED_STREAMS` 设置所允许的上限值内。
  * 编码器需自行权衡：引用传输中的动态表条目可以提高压缩率但有阻塞解码器的风险；而仅引用已确认的条目（降级使用字面量）虽无阻塞风险，但会增加数据体积，可能导致编码器自身受限于拥塞或流量控制（flow-control）。
* **避免流量控制死锁**：为避免指令不完整导致的死锁，除非有足够的流和连接级流量控制信用（flow-control credit）来发送完整的指令，否则编码器**不应（SHOULD NOT）**向流写入指令。
* **已知接收计数（Known Received Count）追踪**：编码器通过追踪解码器确认的动态表插入和重复总数（KRC），来识别哪些动态表条目可以在没有阻塞风险的情况下被引用。该计数根据解码器发回的“部分确认（Section Acknowledgment）”或“插入计数增量（Insert Count Increment）”指令进行更新。

#### 接收端应遵守的设计

* **基本解码与顺序**：解码器处理一系列表示并输出对应的字段部分，同时处理编码器流上用于修改动态表的指令。解码器**必须（MUST）**严格按照表示在已编码字段部分中出现的顺序发出字段行。
* **阻塞解码机制 (Blocked Decoding)**：
  * 收到已编码字段部分时，解码器需检查其“所需插入计数 (RIC)”。如果 RIC 小于或等于解码器当前的插入计数，立即处理；否则，接收该字段部分的流进入阻塞状态。
  * 阻塞期间，已编码数据**应该（SHOULD）**保留在阻塞流的流量控制窗口内。过早释放流量控制额度会使解码器面临内存耗尽攻击的风险。
* **流阻塞数量校验**：如果解码器遇到的阻塞流数量超过了它通过配置承诺支持的上限，**必须（MUST）**将其视为类型为 `QPACK_DECOMPRESSION_FAILED` 的连接错误。
* **RIC 异常处理**：
  * 如果解码器发现实际的 RIC 小于理论预期（基于编码规则的最优值），**必须（MUST）**将其视为 `QPACK_DECOMPRESSION_FAILED` 连接错误。
  * 如果解码器发现实际的 RIC 大于理论预期，**可以（MAY）**将其视为 `QPACK_DECOMPRESSION_FAILED` 连接错误。
* **状态同步与反馈机制**：
  * **完成字段部分处理**：完成包含动态表引用的字段部分的解码后，解码器**必须（MUST）**在解码器流上发出“部分确认（Section Acknowledgment）”指令。
  * **流废弃或重置**：如果提前收到流重置或放弃读取流，解码器生成“流取消（Stream Cancellation）”指令，通知编码器该流上的所有动态表引用均已失效。如果解码器声明的动态表最大容量为0，则**可以（MAY）**省略此步骤。
  * **确认新表条目**：收到新动态表条目后，解码器需发送“插入计数增量（Insert Count Increment）”指令。解码器可自行决定发送时机，可以合并多条指令或用“部分确认”替代以减少网络开销，但延迟过长会导致编码器端压缩效率下降。
* **无效引用处理 (Invalid References)**：
  * 如果在字段行表示中遇到对已驱逐动态表条目的引用，或引用的绝对索引大于等于声明的 RIC，解码器**必须（MUST）**将其视为 `QPACK_DECOMPRESSION_FAILED` 连接错误。
  * 如果在编码器发出的控制指令中遇到对已驱逐动态表条目的引用，解码器**必须（MUST）**将其视为 `QPACK_ENCODER_STREAM_ERROR` 连接错误。


### 3. 参考表 (Reference Tables)

#### 双方均应遵守的设计

* **独立寻址机制**：与 HPACK 不同，QPACK 中静态表和动态表的条目是分别独立寻址的。
* **静态表 (Static Table) 规范**：
    * 包含一个预定义的字段行列表，其条目索引在整个生命周期内固定不变（具体条目定义见附录A）。
    * 所有条目均包含名称和值，其中值可以为空（即长度为0）。
    * 静态表的索引从 `0` 开始（注意：HPACK的静态表索引是从 `1` 开始的）。
* **动态表 (Dynamic Table) 规范**：
    * 包含一个按照先进先出（FIFO）顺序维护的字段行列表。
    * 编码器和解码器共享同一个动态表，初始状态为空。
    * 允许存在重复条目（即名称和值完全相同的条目）。
    * 动态表条目允许包含空值。
* **动态表大小计算 (Dynamic Table Size)**：
    * 动态表的大小是其所有条目大小的总和。
    * 单个条目的大小 = 名称的字节长度 + 值的字节长度 + 32字节附加开销。（长度计算时不应用 Huffman 编码）。
* **绝对索引 (Absolute Indexing)**：
    * 每个条目拥有一个固定的绝对索引。第一个插入的条目绝对索引为 `0`，后续每次插入递增 1。
* **相对索引 (Relative Indexing)**：
    * 从 `0` 开始，增长方向与绝对索引相反。
    * 在**字段行表示 (field line representations)**中，相对索引的 `0` 指代绝对索引等于 `Base - 1` 的条目。这种设计确保了即使已编码字段部分和动态表更新乱序到达，引用依然保持稳定。
* **基准后索引 (Post-Base Indexing)**：
    * 用于字段行表示中，专门指代绝对索引大于或等于 `Base` 的条目。
    * 从 `0` 开始（指代绝对索引等于 `Base` 的条目），增长方向与绝对索引相同。

#### 发送端应遵守的设计

* **动态表写入与管理**：
    * 编码器负责向动态表添加条目，并通过编码器流 (encoder stream) 上的指令将它们发送给解码器。
* **动态表容量与限制 (Capacity)**：
    * 编码器设定动态表的容量（作为其大小的上限）。动态表初始容量为 0，编码器需发送非零的“设置动态表容量 (Set Dynamic Table Capacity)”指令后才能开始使用。
    * 编码器**不得（MUST NOT）**设置超过解码器声明的最大限制（Maximum Dynamic Table Capacity）的容量值，但可以选择使用更低的容量。
    * 当最大表容量限制为 0 时，编码器**不得（MUST NOT）**向动态表插入条目，且**不得（MUST NOT）**在编码器流上发送任何编码器指令。
* **表驱逐机制 (Eviction)**：
    * 在添加新条目之前，如果空间不足，必须从动态表末尾（最早的条目）驱逐条目，直到表大小小于或等于 `(表容量 - 新条目大小)`。
    * 编码器**不得（MUST NOT）**导致尚不可驱逐（unevictable）的条目被驱逐。
    * 当编码器主动缩小动态表容量时，也需从表末尾驱逐条目，直到表大小符合新容量。将容量设置为 0 可以清空动态表。
    * 新插入的条目可以引用即将因本次插入而被驱逐的旧条目（实现时需注意防止因内存提前释放导致数据丢失）。
* **编码器指令中的相对索引**：
    * 在编码器控制指令中，相对索引 `0` 总是指代动态表中最新插入的条目（即参考点随插入操作动态变化）。
* **单遍处理优化**：
    * 编码器可利用“基准后索引 (Post-Base Indexing)”在单次遍历中处理字段部分，并把处理该字段部分期间新增的动态表条目引用包含进去。
* **0-RTT 场景下的异常处理**：
    * 对于使用 0-RTT 数据的客户端，如果服务器之前记住的最大表容量非零，但服务器在当前的 SETTINGS 帧中指定了其他值或省略了该设置，编码器**必须（MUST）**将其视为 `QPACK_DECODER_STREAM_ERROR` 类型的连接错误。

#### 接收端应遵守的设计

* **静态表错误处理**：
    * 当解码器在字段行表示 (field line representation) 中遇到无效的静态表索引时，**必须（MUST）**将其视为类型为 `QPACK_DECOMPRESSION_FAILED` 的连接错误。
    * 如果在编码器流 (encoder stream) 上接收到无效的静态表索引指令，**必须（MUST）**将其视为类型为 `QPACK_ENCODER_STREAM_ERROR` 的连接错误。
* **动态表错误处理**：
    * 由于动态表允许重复条目，解码器**不得（MUST NOT）**将重复条目视为错误。
    * 如果编码器试图添加一个大小超过当前动态表容量的条目，解码器**必须（MUST）**将其视为类型为 `QPACK_ENCODER_STREAM_ERROR` 的连接错误。
* **最大动态表容量限制设定**：
    * 为了限制内存需求，解码器通过发送 `SETTINGS_QPACK_MAX_TABLE_CAPACITY` 来设定编码器被允许使用的最大动态表容量。
    * 对于不使用 0-RTT 数据的情况及所有 HTTP/3 服务器，在处理包含非零值容量的 SETTINGS 帧之前，最大表容量默认为 0。
* **0-RTT 场景下的容量设定**：
    * 对于使用 0-RTT 数据的客户端，服务器的最大表容量是记忆的设置值（若之前未发送过则为 0）。
    * 当客户端 0-RTT 的设定值为 0 时，服务器**可以（MAY）**在其 SETTINGS 帧中将其设置为非零值。
    * 如果记忆的值非零，服务器**必须（MUST）**在其 SETTINGS 帧中发送相同的非零值。


### 4. 传输格式 (Wire Format)

#### 双方均应遵守的设计

* **原语 (Primitives)**：
    * **前缀整数 (Prefixed Integers)**：未经修改地使用 HPACK (RFC7541) 中定义的格式。QPACK 实现**必须（MUST）**能够解码最大达 62 位的整数。
    * **字符串字面量 (String Literals)**：基于 HPACK 中定义的格式，包含可选的 Huffman 编码。在原有的 8 位前缀字符串字面量基础上，扩展支持了不在字节边界上开始的 "N 位前缀字符串字面量" (N 可取 2 到 8)。
* **流管理 (Streams)**：
    * **编码器流 (Encoder Stream)**：类型为 `0x02` 的单向流。从编码器发往解码器，携带无帧边界的编码器指令序列。
    * **解码器流 (Decoder Stream)**：类型为 `0x03` 的单向流。从解码器发往编码器，携带无帧边界的解码器指令序列。
    * 每个 HTTP/3 端点**必须（MUST）**最多启动一个编码器流和最多一个解码器流。如果收到相同类型流的第二个实例，**必须（MUST）**将其视为 `H3_STREAM_CREATION_ERROR` 类型的连接错误。
    * 发送方**不得（MUST NOT）**关闭这两种流，接收方也**不得（MUST NOT）**请求发送方关闭。关闭任何一种流**必须（MUST）**被视为 `H3_CLOSED_CRITICAL_STREAM` 类型的连接错误。
    * 即使连接设置不允许使用这些流，端点也**必须（MUST）**允许对端创建编码器流和解码器流。

#### 发送端应遵守的设计

* **编码器流创建**：如果编码器不打算使用动态表（或者对端允许的最大动态表大小为 0），端点**可以（MAY）**不创建编码器流。
* **编码器指令 (Encoder Instructions)**：编码器在编码器流上发送指令以设置表容量、添加条目或复制条目。
    * **设置动态表容量 (Set Dynamic Table Capacity)**：以 `001` 开始。新容量**必须（MUST）**小于或等于解码器设置的限制。此操作不可导致不可驱逐的条目被驱逐。不产生确认。
    * **按名称引用插入 (Insert with Name Reference)**：以 `1` 开始。T 位指示引用静态表 (1) 还是动态表 (0)。名称通过索引提供，值通过字符串字面量提供。
    * **字面量名称插入 (Insert with Literal Name)**：以 `01` 开始。名称和值均以字符串字面量形式提供。
    * **重复 (Duplicate)**：以 `000` 开始。复制动态表中已存在的条目，用于防止常用旧条目被驱逐。
* **字段部分编码 (Field Line Representations)**：
    * **前缀 (Prefix)**：
        * **Required Insert Count**：编码前需进行转换 (`EncInsertCount = (ReqInsertCount mod (2 * MaxEntries)) + 1`)，以限制长连接上的前缀长度。若 ReqInsertCount 为 0，则直接编码为 0。
        * **Base**：通过一个符号位 ('S') 和 'Delta Base' 相对于 Required Insert Count 进行编码。Base 的值**不得（MUST NOT）**为负。
    * **字段行表示形式**：
        * **带索引的字段行 (Indexed Field Line)**：引用静态表或 Base 之前的动态表。
        * **带基准后索引的字段行 (Indexed Field Line with Post-Base Index)**：引用 Base 之后的动态表。
        * **字面量字段行带名称引用 (Literal Field Line with Name Reference)**：名称来自静态表或 Base 之前的动态表，值是字面量。包含 'N' 位（若置位，强制后续中介不压缩该字段）。
        * **字面量字段行带基准后名称引用 (Literal Field Line with Post-Base Name Reference)**：名称来自 Base 之后的动态表，值是字面量。包含 'N' 位。
        * **纯字面量字段行 (Literal Field Line with Literal Name)**：名称和值均为字面量。包含 'N' 位。
* **'N' 位规则**：当编码器收到对端带有 'N' 位（表示保护、不压缩）的字段行时，如果要转发，**必须（MUST）**继续使用包含字面量的表示形式转发，确保它不被放入动态表。

#### 接收端应遵守的设计

* **解码器流创建**：如果解码器将动态表最大容量设置为 0，端点**可以（MAY）**不创建解码器流。
* **解码器指令 (Decoder Instructions)**：解码器在解码器流上发送指令以维持动态表一致性。
    * **部分确认 (Section Acknowledgment)**：处理完 `Required Insert Count` 不为 0 的字段部分后发出。流 ID 以 7 位前缀整数编码。
    * **流取消 (Stream Cancellation)**：当流被重置或放弃读取时发出，通知编码器停止追踪该流的阻塞状态。
    * **插入计数增量 (Insert Count Increment)**：指示已收到的动态表条目数（包括插入和重复），通过加上增量参数来更新“已知接收计数”。
* **错误处理**：
    * **动态表容量超限**：如果收到超过允许上限的 `Set Dynamic Table Capacity` 指令，解码器**必须（MUST）**视为 `QPACK_ENCODER_STREAM_ERROR`。
    * **无效的 Required Insert Count 编码**：如果解码器计算出的 Required Insert Count 是一个非法的编码值（例如解算出负值或溢出），**必须（MUST）**视为 `QPACK_DECOMPRESSION_FAILED`。
    * **无效的 Base 解码**：如果解析字段部分前缀时发现 Required Insert Count 小于或等于 Delta Base 且符号位为 1（这将导致 Base 为负数），端点**必须（MUST）**将该字段块视为无效。
    * **超前确认 (Section Acknowledgment)**：如果编码器收到的 `Section Acknowledgment` 引用了一个所有块都已被确认过的流，编码器（作为接收该指令的一方）**必须（MUST）**将其视为 `QPACK_DECODER_STREAM_ERROR`。
    * **异常增量 (Insert Count Increment)**：如果编码器收到增量值为 0，或者导致已知接收计数超过了它实际发送的条目数的指令，编码器**必须（MUST）**视为 `QPACK_DECODER_STREAM_ERROR`。

### 6. 错误处理 (Error Handling)

### 7. 安全注意事项 (Security Considerations)

#### 双方均应遵守的设计

* **防范动态表探测攻击 (Probing Dynamic Table State)**：
  * **威胁模型**：攻击者可能利用压缩作为基于长度的预言机（length-based oracle），通过观察编码后的数据长度来确认对动态表状态的猜测（类似于 CRIME 攻击）。由于 TLS 和 QUIC 仅提供机密性保护，无法隐藏数据长度，因此这种攻击依然可能。
  * **QPACK 的缓解**：QPACK 通过强制匹配整个字段行而不是单个字符来缓解此类攻击，使攻击者只能针对给定的字段名暴力猜测整个字段值。因此，高熵值的字段很难被成功恢复，但低熵值的字段仍然脆弱。
* **永不索引字面量 (Never-Indexed Literals) 的透传规则**：
  * 为保护敏感字段，可以将其编码为字面量并设置 'N' 位（表示绝不索引）。
  * 任何中间节点（Intermediary）在转发带有 'N' 位设置的字面量表示的值时，**不得（MUST NOT）**使用会将其编入索引的表示形式进行重新编码。
  * 如果中间节点使用 QPACK 重新编码该字段，**必须（MUST）**继续使用带有 'N' 位的字面量表示形式。
  * 如果中间节点使用 HPACK 重新编码该字段，**必须（MUST）**使用 HPACK 规范中定义的“永不索引的字面量”表示形式。
* **静态哈夫曼编码 (Static Huffman Encoding) 安全性**：
  * 目前没有已知的针对静态哈夫曼编码的有效攻击。虽然研究表明静态表会产生一定的信息泄露，但攻击者无法利用此泄露恢复任何有意义的信息量。
* **内存消耗限制与管理 (Memory Consumption)**：
  * **临时内存**：双方均可以通过顺序处理字段行来限制临时内存的消耗。不论是正在解码的解码器，还是使用单遍算法编码的编码器，都不需要在 QPACK 层面上保留完整的字段行列表。
  * **未发送数据的堆积**：由于流量控制（flow control）导致无法立即发送的数据不受动态表大小限制的约束。双方的实现都应该限制未发送数据的大小（尤其是在解码器流上）。应对策略可包括限制对端打开新流、仅从编码器流中读取数据，或直接关闭连接。
* **实现层面的极限值处理 (Implementation Limits)**：
  * 任何实现都必须为其接受的整数值、编码长度以及字符串字面量长度设置安全上限，以防止超长数值带来的安全漏洞。这些限制**应该（SHOULD）**足够大，以处理该 HTTP 实现被配置为能够接受的最大单个字段。
  * 如果实现遇到大于其能够解码的值：
    * 如果发生在请求/响应流上，**必须（MUST）**将其视为类型为 `QPACK_DECOMPRESSION_FAILED` 的流错误 (stream error)。
    * 如果发生在编码器流或解码器流上，**必须（MUST）**将其视为适当类型的连接错误 (connection error)（即 `QPACK_ENCODER_STREAM_ERROR` 或 `QPACK_DECODER_STREAM_ERROR`）。

#### 发送端应遵守的设计

* **敏感数据的编码决策**：
  * 对于需要保密的字段（如 `Cookie` 或 `Authorization`），特别是那些较短或低熵值的字段，编码器应当考虑不将其编入索引，而是使用带有 'N' 位的字面量。
  * 相反，对于即便暴露也没有价值的字段（如通常不变的 `User-Agent`），编码器可以优先选择索引它们以提升压缩率。
* **隔离压缩上下文与惩罚机制**：
  * 如果互不信任的实体（例如同一连接上来自不同 web origin 的请求）共享一个压缩上下文，编码器应尽可能基于构造消息的实体来隔离动态表的访问权限。
  * 如果编码器无法区分字段值的来源，可以引入惩罚机制：如果同一个字段名出现了大量不同的字段值（疑似暴力破解行为），编码器可以停止将该字段与动态表条目进行比较，以阻止进一步的猜测。这种限制措施的生效速度或概率可以与字段值的长度成反比（值越短，越快被封禁）。
* **控制自身追踪状态的内存**：
  * 编码器需要分配内存来跟踪未确认字段部分中的所有动态表引用。编码器可以通过仅引用它有能力追踪的条目数量来直接限制这种状态内存的消耗（无需通知解码器），尽管这会降低压缩效率。
  * 编码器还可以通过设置一个比解码器允许值更小的动态表容量来限制自己使用的状态内存（通过控制指令通知解码器）。

#### 接收端应遵守的设计

* **防御内存耗尽攻击**：
  * 攻击者可能试图利用动态表或阻塞流来耗尽解码器的内存。解码器通过发送配置参数来确立内存使用的硬性上限。
  * 解码器通过 `SETTINGS_QPACK_MAX_TABLE_CAPACITY` 来限制动态表使用的最大状态内存。
  * 解码器通过 `SETTINGS_QPACK_BLOCKED_STREAMS` 来限制它为了追踪阻塞流而使用的状态内存。没有引起阻塞风险的流不会在解码器端消耗额外的追踪内存。

Appendix C.  Sample Single-Pass Encoding Algorithm

   Pseudocode for single-pass encoding, excluding handling of
   duplicates, non-blocking mode, available encoder stream flow control
   and reference tracking.

   # Helper functions:
   # ====
   # Encode an integer with the specified prefix and length
   encodeInteger(buffer, prefix, value, prefixLength)

   # Encode a dynamic table insert instruction with optional static
   # or dynamic name index (but not both)
   encodeInsert(buffer, staticNameIndex, dynamicNameIndex, fieldLine)

   # Encode a static index reference
   encodeStaticIndexReference(buffer, staticIndex)

   # Encode a dynamic index reference relative to Base
   encodeDynamicIndexReference(buffer, dynamicIndex, base)

   # Encode a literal with an optional static name index
   encodeLiteral(buffer, staticNameIndex, fieldLine)

   # Encode a literal with a dynamic name index relative to Base
   encodeDynamicLiteral(buffer, dynamicNameIndex, base, fieldLine)

   # Encoding Algorithm
   # ====
   base = dynamicTable.getInsertCount()
   requiredInsertCount = 0
   for line in fieldLines:
     staticIndex = staticTable.findIndex(line)
     if staticIndex is not None:
       encodeStaticIndexReference(streamBuffer, staticIndex)
       continue

     dynamicIndex = dynamicTable.findIndex(line)
     if dynamicIndex is None:
       # No matching entry.  Either insert+index or encode literal
       staticNameIndex = staticTable.findName(line.name)
       if staticNameIndex is None:
          dynamicNameIndex = dynamicTable.findName(line.name)

       if shouldIndex(line) and dynamicTable.canIndex(line):
         encodeInsert(encoderBuffer, staticNameIndex,
                      dynamicNameIndex, line)
         dynamicIndex = dynamicTable.add(line)

     if dynamicIndex is None:
       # Could not index it, literal
       if dynamicNameIndex is not None:
         # Encode literal with dynamic name, possibly above Base
         encodeDynamicLiteral(streamBuffer, dynamicNameIndex,
                              base, line)
         requiredInsertCount = max(requiredInsertCount,
                                   dynamicNameIndex)
       else:
         # Encodes a literal with a static name or literal name
         encodeLiteral(streamBuffer, staticNameIndex, line)
     else:
       # Dynamic index reference
       assert(dynamicIndex is not None)
       requiredInsertCount = max(requiredInsertCount, dynamicIndex)
       # Encode dynamicIndex, possibly above Base
       encodeDynamicIndexReference(streamBuffer, dynamicIndex, base)

   # encode the prefix
   if requiredInsertCount == 0:
     encodeInteger(prefixBuffer, 0x00, 0, 8)
     encodeInteger(prefixBuffer, 0x00, 0, 7)
   else:
     wireRIC = (
       requiredInsertCount
       % (2 * getMaxEntries(maxTableCapacity))
     ) + 1;
     encodeInteger(prefixBuffer, 0x00, wireRIC, 8)
     if base >= requiredInsertCount:
       encodeInteger(prefixBuffer, 0x00,
                     base - requiredInsertCount, 7)
     else:
       encodeInteger(prefixBuffer, 0x80,
                     requiredInsertCount - base - 1, 7)

   return encoderBuffer, prefixBuffer + streamBuffer
